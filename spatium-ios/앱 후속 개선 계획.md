# Spatium iOS 후속 개선 계획

> 작성 기준: 2026-07-18 · iOS `develop-ios` 브랜치 · 커밋 `c8fed67`
>
> 범위: 현재 iOS 앱과 연동 백엔드의 성능·메모리·배터리·유지보수 위험 검토

## 1. 결론

현재 앱에서 즉시 수정하지 않으면 크래시나 데이터 손실로 이어지는 신규 문제는 발견되지 않았다. 최근 작업으로 프로젝트 캐시, 사용자 가구 저장소, 에디터 복구본, 서버 룸 USDZ 복원, 로컬 방 스캔 자산 처리의 메인 스레드 파일 작업이 대부분 제거됐고 SceneKit 증분 렌더링과 모델 템플릿 LRU도 적용돼 있다.

따라서 남은 항목을 코드만 보고 연속으로 수정하기보다는 다음 순서로 진행하는 것이 안전하다.

1. 실기기 Release 빌드에서 기준 성능 측정
2. 서버 룸 USDZ의 Base64 전송 제거
3. 에디터 최초 SceneKit·GLB 로드 지연 계측 및 단계적 로드
4. 에디터 상태 관찰 범위 축소
5. 숨겨진 애니메이션과 유휴 렌더링 정지
6. SceneKit 화면 종료 후 메모리 회수 검증
7. 서버 룸 요약 계약과 디스크 캐시 정책 개선

코드 검토만으로 원인이 확인된 항목은 이 문서에서 **코드 근거 확인**으로 표시한다. 실제 CPU·프레임·메모리 수치가 있어야 구현 여부를 결정할 항목은 **프로파일링 필요**로 표시한다.

## 2. 현재 완료된 최적화 기준선

다음 작업은 이미 적용돼 있으므로 같은 방식의 중복 수정을 피한다.

- SceneKit 선택 하이라이트와 측정 노드의 증분 갱신
- 가구 드래그 중 치수 갱신 최대 60Hz 제한
- 벽 투명도 평가 최대 30Hz 제한 및 상태 미변경 프레임 생략
- 가구 GLB 템플릿 최대 8개·원본 파일 합계 64MiB LRU 제한
- 메모리 경고 시 가구 템플릿 캐시 해제
- ImgTo3D 화면 비활성화 시 SceneKit 씬·정점 샘플·표시 이미지 해제
- 대용량 GLB의 파일 기반 다운로드·multipart 업로드·변환 저장
- 프로필과 ImgTo3D 이미지의 백그라운드 다운샘플링
- 프로젝트 캐시, 사용자 가구 카탈로그, 에디터 복구본, RoomScenes, RoomScans 파일 작업의 메인 스레드 분리
- 연속 저장의 revision 순서 보장과 실패 복구
- 유닛 테스트 75개, UI 테스트 23개, 런치 테스트 1개

## 4. P0 — 실기기 성능 기준선 구축

### 현재 근거

- `SpatiumUITests/SpatiumUITests.swift:39`에 앱 실행 시간 측정만 존재한다.
- SceneKit 방 생성, USDZ 파싱, GLB 파싱, 가구 노드 생성, 에디터 첫 상호작용 가능 시점을 구분하는 signpost가 없다.
- 현재 최적화는 빌드·기능 테스트로 검증됐지만 CPU, hitch, 메모리 피크의 전후 수치는 없다.

- 시뮬레이터가 아닌 실제 iPhone 사용

- 최초 실행과 두 번째 실행을 분리해 콜드 캐시·웜 캐시 결과를 각각 기록

## 5. P1 — 서버 룸 USDZ Base64 전송 제거

### 현재 근거

- 백엔드 `RoomService.getRoomScene`은 USDZ 입력 스트림을 `readAllBytes()`로 메모리에 올린다.
- 백엔드는 전체 바이트를 다시 Base64 문자열로 변환해 `/api/rooms/{roomId}/scene` JSON에 포함한다.
- `RoomDTO.RoomSceneModelResponse`가 `dataBase64`를 응답 필드로 가진다.
- iOS `ProjectService.fetchRoomScene`은 Base64 문자열을 받은 뒤 `RoomSceneModelDiskStore`에서 다시 `Data`로 디코딩하고 USDZ 파일로 기록한다.
- iOS의 디코딩은 백그라운드로 이동했지만 Base64의 약 33% 전송량 증가와 서버·앱의 일시적 대형 메모리 할당은 그대로 남아 있다.

### 권장 계약

우선안은 metadata JSON과 USDZ 바이너리를 분리하는 것이다.

1. `GET /api/rooms/{roomId}/scene`
   - 룸 이름, 편집 metadata, 모델 버전 또는 `updatedAt` 반환
   - `dataBase64` 제거
2. `GET /api/rooms/{roomId}/model`
   - `Content-Type: model/vnd.usdz+zip`
   - 파일 저장소 입력 스트림을 HTTP 응답으로 직접 스트리밍
   - `ETag`, `Last-Modified`, `Content-Length` 제공
3. 대안
   - 인증이 유지되는 짧은 만료의 서명 URL 반환
   - 앱이 URLSession download task로 직접 다운로드

### iOS 변경 범위

- `ProjectService.fetchRoomScene`에서 metadata와 모델 다운로드를 구조화된 동시 작업으로 수행
- `SpatiumAPIClient.downloadFile` 재사용
- 다운로드 임시 파일을 `RoomScenes/scene-<roomId>-<version>.usdz`로 atomic 이동
- ETag 또는 버전이 같으면 기존 파일 재사용
- 취소·401 토큰 갱신·재시도 시 임시 파일 정리
- 모델 다운로드 실패 시 기존처럼 metadata 기반 편집 화면으로 폴백할지 제품 정책 확인

### 백엔드 변경 범위

- `RoomDTO.RoomSceneModelResponse.dataBase64` 제거 또는 하위 호환 기간 동안 선택 필드화
- `RoomService.getRoomScene`의 `readAllBytes()` 제거
- 파일 스트림 응답용 서비스·컨트롤러 추가
- 소유권 검증과 Content-Type·길이·캐시 헤더 유지
- 구버전 앱 지원 기간이 필요하면 API 버전 또는 capability 필드 제공

### 검증

- 동일 USDZ의 SHA-256이 기존 Base64 방식과 바이너리 방식에서 일치
- 401 후 토큰 재발급과 한 번 재시도 검증
- 다운로드 취소·네트워크 단절·디스크 부족 시 임시 파일 잔류 없음
- 50MiB 이상 테스트 파일에서 서버 heap, 응답 크기, 앱 메모리 피크 비교
- 기존 metadata-only 폴백 테스트 유지

### 게스트 모드

인증된 서버 룸에만 적용된다. 게스트의 로컬 RoomPlan 스캔, 파일 공유, 로컬 3D 편집에는 동작 변화나 신규 제한이 없다.

## 6. P1 — 에디터 최초 SceneKit·GLB 로드 지연 분리

### 현재 근거

- `RoomEditorSceneView.makeUIView`는 메인 액터에서 즉시 `buildScene(for:)`를 호출한다 (`RoomEditorSceneView.swift:43`).
- 스캔 방은 `SCNScene(url:)`로 USDZ를 동기 파싱한다 (`RoomEditorSceneView.swift:476-489`).
- 가구 템플릿 최초 로드는 `GLTFAsset`과 `SCNScene(gltfAsset:)`을 동기 생성한다 (`FurnitureModelLoader.swift:221-234`).
- 방 셸 bounds, 바닥 높이, 벽 collider, 가구 hierarchy bounds 계산도 첫 씬 생성 경로에 포함된다.
- 캐시 적중 이후 비용은 줄었지만 콜드 캐시 첫 진입은 여전히 메인 스레드 지연 후보다.

### 안전한 개선 순서

1. P0 signpost로 `SCNScene(url:)`, `GLTFAsset`, bounds, collider별 시간을 분리
2. 100ms 이상 구간이 확인될 때만 로드 구조 변경
3. 에디터 컨테이너와 로딩 UI를 먼저 표시
4. 파일 읽기·모델 메타데이터 준비처럼 thread-safe한 값 작업만 백그라운드 수행
5. SceneKit 객체 생성·씬 부착은 MainActor에서 짧은 단계로 나눠 수행
6. 여러 가구는 화면에 가까운 항목 또는 선택 항목부터 점진적으로 부착
7. 화면을 닫거나 다른 방으로 바꾸면 이전 준비 작업 취소

### 주의 사항

- SceneKit과 GLTFKit2 객체가 thread-safe하다고 가정하고 무조건 `Task.detached`로 옮기면 안 된다.
- 하나의 `SCNNode`를 백그라운드와 메인 스레드에서 동시에 변경하면 안 된다.
- 모든 모델을 앱 시작 때 미리 파싱하면 런치 시간과 메모리 피크가 증가하므로 금지한다.
- 로딩 중 placeholder를 사용하더라도 저장 데이터의 위치·회전·크기와 최종 GLB 치수가 달라지지 않아야 한다.

### 검증

- 콜드 캐시와 웜 캐시 각각 첫 화면·첫 조작 가능 시간 측정
- 로딩 중 방 닫기, 빠른 방 전환, 앱 백그라운드 이동 취소 검증
- 같은 가구 10개 배치 시 GLB 템플릿 파싱 1회 유지
- placeholder에서 실제 모델로 교체된 뒤 선택 ID, 측정값, 충돌 bounds 유지
- 게스트 내장 스캔과 서버 방 양쪽 테스트

### 게스트 모드

게스트가 가장 많이 사용하는 로컬 스캔 편집의 첫 진입 속도도 개선 대상이다. 기능 제한은 추가하지 않는다.

## 7. P1 — 에디터 상태 관찰 범위 축소

### 현재 근거

- `RoomEditorViewModel`은 `layout`, 저장 상태, 선택 상태, 측정 상태, 꾸미기 상태, 씬 revision 등 20개 이상의 `@Published` 값을 한 객체에 가진다 (`RoomEditorViewModel.swift:30-79`).
- `RoomEditorView`, `RoomEditorSceneView`, `EditorDecorControlsView`, `EditorFurnitureSelectionControlsView`가 같은 `ObservableObject` 전체를 관찰한다.
- `ObservableObject`는 어떤 `@Published` 값이 바뀌어도 모든 관찰 뷰에 `objectWillChange`를 전달한다.
- `RoomEditorSceneView.updateUIView`는 내부 guard가 있어 노드 재생성은 줄었지만 선택·저장 문구·draft 상태 변경만으로도 Representable 갱신 자체는 호출될 수 있다.
- `RoomEditorView.visibleItems`는 카탈로그 배열을 필터링하고 body 안에서 개수·빈 상태·ForEach에 반복 참조된다 (`RoomEditorView.swift:104-118`, `543-546`, `658`).

### 권장 구조

대규모 일괄 변경보다 다음 순서로 좁힌다.

1. 에디터 캔버스를 별도 하위 View로 분리
2. 씬에 필요한 입력만 값으로 전달
   - `sceneRevision`
   - `viewMode`
   - `selectedItemID`
   - `isMeasuring`
   - 이동·꾸미기 활성 상태
3. 저장·오류·draft 상태를 내비게이션과 상태 배지 전용 모델로 분리
4. 선택·가구 편집 상태를 선택 컨트롤 전용 모델로 분리
5. iOS 17 최소 버전을 활용할 경우 Observation의 `@Observable` 전환을 작은 단위에서 검토
6. 카탈로그 검색 결과와 그룹 목록은 카탈로그 패널 입력이 바뀔 때 한 번만 계산

### 피해야 할 방식

- 전체 에디터에 무조건 `.equatable()` 적용
- 큰 `RoomLayout` 전체 equality를 매 프레임 계산
- `@State`를 임의 계산 캐시로 사용하고 갱신 조건을 정의하지 않는 방식
- 모든 상태를 하나의 environment 값으로 옮기는 방식

### 검증

- SwiftUI Instruments의 Cause & Effect Graph에서 저장 상태 변경이 SceneKit Representable 갱신으로 전파되지 않는지 확인
- 드래그 10초 동안 `RoomEditorView.body`와 `RoomEditorSceneView.updateUIView` 호출 수 전후 비교
- 선택·측정·undo·redo·draft 저장·꾸미기 기능 회귀 테스트 유지
- equality나 상태 복제 비용이 기존 body 계산보다 커지지 않는지 Time Profiler 확인

### 게스트 모드

상태 전달 구조만 변경한다. 게스트의 로컬 편집·임시 저장·복구 기능을 그대로 유지해야 한다.

## 8. P1 — 숨겨진 애니메이션 정지와 접근성 반영

- 4x MSAA 변경은 실제 GPU Frame Capture 또는 Power Profiler 비교 후 결정한다.

### 검증

- ImgTo3D 탭에서 30초 유휴 시 홈 애니메이션 관련 CPU 샘플이 없는지 확인
- 홈 복귀 시 애니메이션이 중복 시작되거나 속도가 누적되지 않는지 확인

## 9. P2 — SceneKit 종료 정리와 반복 진입 메모리 검증

### 현재 근거

- `ImgTo3DModelViewer.dismantleUIView`는 scene, pointOfView, gesture, geometry, material texture, 정점 샘플을 명시적으로 정리한다.
- `RoomEditorSceneView.dismantleUIView`는 현재 사람 뷰의 `CADisplayLink`만 중지한다 (`RoomEditorSceneView.swift:144-147`).
- 에디터 Coordinator는 `shellCache`, furniture node 상태, collider 배열, measurement node, 카메라와 SceneKit 노드를 보관한다.
- Coordinator가 정상 해제되면 대부분 같이 해제될 수 있으므로 실제 누수라고 단정할 수는 없다. Memgraph 확인이 필요하다.

### Memgraph 확인 절차

1. 동일 방을 열고 닫는 작업 10회 수행
2. 각 반복 뒤 2~3초 유휴 상태 유지
3. 1회차와 10회차의 `SCNNode`, `SCNGeometry`, `SCNMaterial`, `MTLTexture`, `RoomEditorSceneView.Coordinator` 개수 비교
4. Memory Graph에서 `CADisplayLink → Coordinator`, gesture target, SCNView delegate 참조 확인
5. 메모리 경고를 발생시켜 정적 모델 템플릿 캐시가 비워지는지 확인

### 잔류가 확인될 때의 변경

- `dismantleUIView`에서 `uiView.isPlaying = false`, delegate와 pointOfView 제거
- 등록한 gesture 비활성화·제거
- `uiView.scene = nil`, coordinator의 `sceneView = nil`
- measurement·furniture·shell 노드 계층 제거
- `shellCache`, collider, 투명 벽 목록, 렌더 상태 딕셔너리 비우기
- 재질 texture 정리는 에디터가 소유한 clone에만 적용하고 공유 LRU 템플릿은 손상시키지 않도록 분리

### 검증

- weak reference 기반 coordinator·SCNView 해제 테스트
- 반복 진입 Allocations 그래프의 지속 우상향 제거
- 화면 종료 후 `CADisplayLink` 콜백 0회
- 재진입 시 빈 씬 또는 재질 손실 없이 정상 표시

### 게스트 모드

로컬 에디터 반복 사용의 메모리 안정성이 개선된다. 기능 제한은 없다.

## 10. P2 — 룸 요약 계약과 항목 수 조회 개선

### 현재 근거

- 백엔드 `ResponseRoomSummaryDTO`에는 `roomId`, `roomName`, `area`, `thumbnailUrl`, `updatedAt`만 있다.
- iOS `ProjectService.fetchRooms`는 모든 서버 룸의 `itemCount`와 `photoCount`를 0으로 만든다 (`ProjectService.swift:166-180`).
- `ProjectStore.refreshItemCounts`는 스캔 파일 참조가 있는 방만 JSON을 추가 로드하도록 되어 있지만 현재 서버 요약 `RoomRecord`에는 해당 참조가 없어 서버 룸 항목 수를 채우지 못한다.
- RoomScans 경로가 사용되는 경우에도 `loadItemCount`는 `export.items()`를 호출해 개수 표시만을 위해 전체 `EditableScanItem`을 생성한다 (`RoomScanAssetService.swift:224-235`).
- 프로젝트·룸 개수 갱신은 task group에 모든 요청을 한꺼번에 넣으므로 데이터 규모가 커지면 순간 요청 수가 커질 수 있다.

### 권장 백엔드 계약

- `ResponseRoomSummaryDTO`에 `itemCount`, `photoCount`, `sceneVersion` 추가
- `itemCount`는 목록 요청 때 metadata 파일을 매번 파싱하지 않고 룸 생성·편집 저장 시 계산해 DB에 저장
- `sceneVersion`은 metadata 또는 USDZ가 바뀔 때 증가
- `updatedAt`은 timezone이 포함된 ISO 8601 문자열 또는 epoch로 반환
  - 현재 백엔드는 `LocalDateTime.toString()`을 반환해 timezone 정보가 없다.
  - iOS `ISO8601DateFormatter`가 실패하면 현재 시각으로 폴백하므로 캐시 버전 기준으로 쓰기 어렵다.

### iOS 변경

- `RoomSummaryItem`과 `RoomRecord`에 서버 item/photo count와 scene version 반영
- 방 목록 진입 시 개별 JSON 항목 수 조회 제거
- 하위 호환이 필요한 동안 필드가 없을 때만 기존 폴백 실행
- 폴백 병렬 요청은 최대 4~6개처럼 제한된 동시성 적용
- 로컬 JSON count 경로가 남으면 `export.items()` 대신 count 전용 속성으로 변환 배열 생성을 생략

### 검증

- 0개, 1개, 100개 항목 룸 요약 계약 테스트
- 편집 저장 후 itemCount와 sceneVersion 증가 확인
- 구버전 서버 응답에서 iOS 디코딩 실패 없이 폴백
- 100개 룸 목록에서 동시 요청 상한과 취소 확인
- 다른 기기에서 룸 편집 후 현재 기기의 캐시가 새 버전으로 교체되는지 확인

### 게스트 모드

서버 룸 목록 최적화이므로 게스트 기능에는 영향이 없다. 로컬 프로젝트의 기존 항목 수 표시는 그대로 유지한다.

## 11. P2 — RoomScans·RoomScenes 자동 디스크 캐시 정책

### 현재 근거

- 설정 화면에서 `RoomScans`, `RoomScenes`, `Spatium` 캐시 용량 확인과 전체 삭제를 제공한다.

- 서버 방과 스캔을 장기간 많이 열면 방별 USDZ·JSON이 계속 누적될 수 있다.

### 권장 정책

- 캐시별 용량과 마지막 접근 시각 기록
- 앱 시작을 막지 않는 utility 작업으로 예산 초과분 정리
- 최근 사용 룸, 현재 열린 룸, 다운로드 중인 파일은 퇴거 제외
- RoomScans와 RoomScenes를 합산할지 별도 예산을 둘지 실제 파일 분포 측정 후 결정
- 정확한 예산은 임의로 고정하지 말고 테스트 사용자 데이터에서 p90·p95 캐시 크기를 확인한 뒤 결정
- 캐시 삭제는 원본 RoomPlan 내보내기 파일과 Application Support의 사용자 가구 GLB를 절대 삭제하지 않도록 현재 경계 유지

### 최신성 정책

- `sceneVersion` 또는 ETag가 같을 때만 캐시 재사용
- 버전이 바뀌면 새 파일을 완전히 받은 뒤 atomic 교체
- 새 파일 설치 실패 시 기존 정상 캐시 유지
- 캐시 메타데이터 손상 시 파일을 전부 삭제하기보다 안전하게 재계산

### 검증

- 예산 초과 시 가장 오래 사용하지 않은 파일부터 삭제
- 현재 사용 중 파일과 사용자 가구 GLB 보존
- 디스크 부족·삭제 실패 시 앱 기능 유지와 오류 처리
- 캐시 정리 작업이 메인 스레드에서 실행되지 않는지 회귀 테스트

### 게스트 모드

캐시만 정리하며 게스트가 만든 로컬 프로젝트와 원본 스캔 파일은 삭제하지 않는다. 캐시가 없으면 기존 원본 또는 서버에서 다시 로드한다.

## 12. P3 — 메모리 multipart 레거시 경로 제거

### 현재 근거

- 실제 API 전송은 `MultipartFormData.writeBodyFile`을 사용해 파일 파트를 1MiB 청크로 복사한다.
- 별도 `MultipartFormData.init(parts:)`는 파일 파트에서 `Data(contentsOf:)`로 전체 파일을 메모리에 올린다 (`MultipartFormData.swift:39-58`).
- 현재 검색 결과 이 초기화 경로는 작은 multipart 계약 유닛 테스트에서만 사용한다.
- 현재 런타임 문제는 아니지만 미래 코드가 파일 파트와 함께 이 초기화를 사용하면 대용량 GLB가 메모리에 다시 올라갈 수 있다.

### 권장 변경

- 메모리 body 생성 API는 `.data` 파트만 허용하도록 타입 또는 throwing 검사 추가
- 파일 파트가 하나라도 있으면 `writeBodyFile`만 사용하도록 API를 분리
- 작은 이미지·텍스트 계약 테스트는 data 전용 helper 사용
- 파일 기반 테스트는 기존 임시 body 파일을 읽어 multipart 경계를 검증

### 게스트 모드

서버 업로드 내부 안전장치이므로 게스트의 파일 공유와 로컬 편집에는 영향이 없다.

10. `perf(ios): 스캔 및 씬 디스크 캐시 예산 적용`
11. `refactor(ios): 대형 에디터 뷰와 씬 역할 분리`

각 작업이 끝날 때 다음을 함께 수행한다.

- 해당 기능의 유닛 또는 UI 회귀 테스트 추가
- Debug 시뮬레이터 빌드
- 전체 유닛 테스트
- 테스트 타깃 `build-for-testing`
- Release 시뮬레이터 빌드
- `git diff --check`
- README의 성능 정책과 테스트 수 갱신
- `이번 앱 프로젝트 정리본.md`의 구조·정책·테스트 목록 갱신
- 게스트 기능 제한이 생기는지 명시
- 변경 파일만 포함하는 상세 `git add`와 다중 `-m` 커밋 명령 작성

## 15. 지금 하지 말아야 할 변경

- 에디터 전체 프레임을 일괄 30FPS로 낮추는 변경
- 근거 없이 SceneKit 캐시 개수나 64MiB 예산을 늘리는 변경
- 모든 SwiftUI 뷰에 `.equatable()`을 적용하는 변경
- SceneKit 노드를 thread-safe 검증 없이 백그라운드에서 직접 변경하는 방식
- 모든 GLB를 앱 시작 시 미리 파싱하는 방식
- 프로젝트·룸 수만큼 네트워크 요청을 무제한 동시 실행하는 방식
- 테스트 수만 늘리고 실기기 CPU·프레임·메모리 기준선을 만들지 않는 방식

## 17. 최종 판단

다음 한 건만 고른다면 **P0 실기기 성능 기준선과 signpost 추가**가 우선이다. 이미 코드 수준의 저위험 최적화를 여러 차례 적용했기 때문에 이후부터는 측정 없이 구조를 크게 바꾸면 개선 효과보다 회귀 위험이 커질 수 있다.

측정과 별개로 서버까지 수정할 수 있다면 가장 큰 구조적 효과는 **서버 룸 USDZ의 Base64 응답을 바이너리 스트리밍으로 교체하는 작업**이다. 이 작업은 게스트 기능에 영향을 주지 않으면서 서버 heap, 응답 크기, 앱 메모리 피크를 동시에 줄일 수 있다.
