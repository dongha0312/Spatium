## 1. 개요

**Spatium**은 사용자가 자기 방을 LiDAR로 스캔해 3D 공간으로 만들고, 그 안에 가구를 배치해 인테리어를 시뮬레이션하는 iOS 앱입니다. 가구는 기본 카탈로그뿐 아니라 **가구 사진 한 장으로 직접 생성한 3D 모델**도 사용할 수 있습니다.

전체 사용자 여정:

```
온보딩 → 로그인 / 게스트
        → 프로젝트 생성
        → 방 스캔(RoomPlan/LiDAR) → 스캔 리뷰 → 파일 공유 / 서버 업로드(로그인)
        → 3D 에디터에서 가구 배치·편집 → 저장
   (병행, 로그인) 가구 사진 촬영 → 객체 분리 → 3D 생성 → 보정 → 내 가구로 저장
```

설계의 핵심 방향 세 가지:
- **낙관적 UI + 안전한 롤백** — 서버 응답을 기다리지 않고 화면을 먼저 갱신하고, 실패 시 이전 상태로 되돌립니다.
- **로컬 우선(local-first)** — 게스트/오프라인에서도 프로젝트·룸을 로컬로 만들고, 로그인·업로드 시 서버 자원으로 승격합니다.
- **크래시 방어** — 강제 언래핑·중복 키·경쟁 상태 등 실제로 겪은 크래시 지점마다 방어 로직과 주석이 남아 있습니다. (→ [14절](#14-함정--재발-방지-노트))

### 백엔드 API 스펙과의 버전 싱크

이 앱은 백엔드 응답 스키마와 요청 바디 키에 **강하게 의존**합니다. 그 계약을 문서가 아니라 **테스트로 고정**하고 있습니다.

- `SpatiumTests`의 **"API 명세 계약"** 그룹과 별도 `BackendContractTests` 구조체가 요청/응답 JSON 형태를 스펙에 못 박음:
  - `loginRequestMatchesSpecKeys`, `signUpRequestMatchesSpecKeys`, `socialLoginRequestSendsOnlyProviderAndIdToken`, `socialSignUpRequestMatchesSpecKeys`
  - `envelopeDecodesSpecShape`(공통 엔벨로프 형태), `viewModeRawValuesMatchSpec`(뷰모드 raw 값)
  - `imgTo3DOptionsMatchSpringGatewayAndFastAPI`, `imgTo3DUsesSpringGatewayAndDecodesBinaryMetadataHeader`, `onlySpringFurnitureModelURLsReceiveAPIAuthentication`, `locallySavedFurnitureSynchronizesWithCurrentBackendContract`, `springFurnitureMultipartUsesFileAndJSONParts`, `avatarMultipartUsesBackendImageField`
- 즉, **백엔드 스키마가 바뀌면 이 계약 테스트가 깨져서 감지**되는 구조입니다. 서버 버전을 명시적으로 핀하는 값(버전 헤더 등)은 코드에 없습니다.

> ⚠️ **미확인**: 백엔드와 공유하는 별도 OpenAPI/Swagger 스펙 문서가 있는지, 있다면 위치·갱신 주체는 이 리포에 기록돼 있지 않습니다. → [16절](#16-관련-저장소--미확인-항목)

## 3. 앱 아키텍처

### 3-1. 3계층 구조

```
Core     — 공용 인프라 + 도메인 (네트워킹, 모델, 서비스, 확장, 디자인 시스템)
Features — 화면 단위 모듈 (Auth, Home, Rooms, Scan, Editor, ImgTo3D, Settings, Onboarding)
Shared   — 재사용 UI (UIKit 브리지, 공통 컴포넌트)
```

- **Features**는 화면과 뷰모델을 담고, **Core/Services**의 스토어·서비스를 주입받아 사용합니다.
- **Service**(무상태 API 래퍼) ↔ **Store**(`ObservableObject` 상태 보관 + 낙관적 업데이트) 를 분리합니다.
  - 예) `ProjectService`(순수 API 호출) ↔ `ProjectStore`(목록 상태·캐시·롤백 관리)

### 3-2. 상태 관리 스토어

| 스토어 | 역할 | 특징 |
|--------|------|------|
| `AuthTokenStore` | 토큰/로그인 상태 (`shared`) | 키체인 보관, 실행 중 메모리 재사용, `isLoggedIn` 퍼블리시 |
| `ProjectStore` | 프로젝트·룸 목록 | utility 직렬 캐시 읽기·쓰기, revision 순서 보장, 낙관적 업데이트, 병렬 개수 집계, **캐시 저장 실패 관찰**(`cachePersistenceState`) |
| `UserFurnitureStore` | 사용자 생성 가구 | 앱 루트 `@StateObject`, utility 직렬 초기 로드·GLB·JSON 트랜잭션 |
| `CurrentUserStore` | 로그인 사용자 프로필 (`shared`) | 새로고침 시 함께 갱신 |
| `SpatiumAPIEnvironment` | 서버 주소 (`shared`) | Debug 변경 가능 / Release 고정 (→ [4-5절](#4-5-서버-3종--주소-변경-방법)) |

- 앱 루트(`SpatiumApp`)에서 `UserFurnitureStore`를 `environmentObject`로 주입.
- `UserFurnitureStore`는 `catalog.json` 초기 읽기·디코딩·레거시 단위 변환·최신순 정렬과 GLB 복사·교체·삭제 및 카탈로그 인코딩·원자 쓰기를 전용 utility 직렬 큐에 맡김. 초기 로드 중 들어온 저장·삭제·서버 승격은 로드 완료 후 실행해 기존 카탈로그를 보존하고, 디스크 성공 후에만 공개 목록을 교체하며 카탈로그 쓰기 실패 시 기존 GLB와 메모리 상태를 유지.
- `ProjectStore`는 `projects.json` 초기 읽기·디코딩·인코딩·원자 쓰기·로그아웃 삭제를 전용 utility 직렬 큐에 맡김. 로그인 상태에서는 캐시를 먼저 비동기로 불러온 뒤 서버 새로고침을 이어가며, 증가하는 revision으로 로드 중 변경된 이전 파일 스냅샷과 뒤늦게 도착한 이전 저장을 무시함.
- `ProjectStore`는 `AuthTokenStore.$isLoggedIn`을 구독 → **로그인 시 서버 동기화, 로그아웃 시 캐시 삭제**(이전 계정 데이터 잔존 방지).
- `AuthTokenStore`는 앱 시작 시 접근 토큰을 Keychain에서 한 번 확인하고 이후 API 요청에는 메모리 값을 사용. 갱신 토큰도 첫 사용 시 한 번만 읽고, 저장은 기존 Keychain 항목을 우선 갱신한 뒤 없는 경우에만 추가해 반복 조회와 delete/add 호출을 줄임.

### 3-3. 데이터 흐름 원칙 (낙관적 업데이트 + 롤백)

`ProjectStore`의 이름 변경/삭제/생성이 전형적인 패턴입니다.

```
1) 로컬 상태를 즉시 변경 → 화면 반영 → 전용 utility 큐에서 디스크 캐시 저장
2) 서버 API 호출
3) 실패하면 이전 값으로 되돌리고(rollback) 오류 메시지 노출
4) local- ID(게스트/미동기화)는 서버 호출을 건너뛰고 로컬만 반영
```

## 4. 네트워킹 계층

### 4-1. 공통 응답 엔벨로프

모든 API는 `{ statusCode, message, data }` 형태의 `SpatiumAPIEnvelope<T>`로 감싸져 옵니다. 호출부는 path·method·기대 `data` 타입만 지정하면 됩니다.

```swift
try await SpatiumAPIClient.shared.send(method: "GET", path: "api/projects")
```

`send`(GET/DELETE), `send(body:)`(JSON), `sendMultipart(parts:)`(파일 업로드) 세 가지 오버로드를 제공합니다.

### 4-2. 인증 토큰 자동 재발급

- 요청에 accessToken을 `Bearer`로 첨부.
- **401 발생 시** refreshToken으로 재발급 후 **딱 한 번** 재시도.
- 재발급 자체가 거부되면(세션 만료) 토큰을 비워 로그인 화면으로 복귀.
- `AuthRefreshCoordinator`(actor)가 **재발급을 직렬화** → 401이 동시에 여러 개 터져도 refresh rotation이 꼬이지 않고, 진행 중이면 그 결과를 함께 기다림.

### 4-3. 멀티파트 스트리밍 업로드

- 대용량 USDZ/GLB를 메모리에 통째로 올리지 않기 위해 **원본 파일을 1MiB 청크로 임시 multipart 바디에 복사**.
- JSON 응답 요청은 임시 바디를 `URLSession.upload(fromFile:)`로 보내고, 대용량 GLB 응답 요청은 바디 파일의 `InputStream`을 요청에 연결한 `URLSession` download task로 보내 **업로드 바디와 다운로드 응답을 모두 파일 기반**으로 처리.
- 바디 작성(파일 읽기)은 `Task.detached`로 백그라운드 처리 → 메인 스레드 미차단.
- 바디 작성이 중간에 실패하거나 요청이 끝나면 임시 multipart 파일을 삭제. 파일 응답의 non-2xx 에러 바디는 최대 64KiB만 읽어 오류를 해석.
- 기본 타임아웃: 일반 요청 10초, 멀티파트 120초.
- 메모리 바디 생성기(`MultipartFormData(parts:)`)는 **`.data` 파트 전용**이며 파일 파트가 섞이면 `MultipartFormDataError.filePartRequiresBodyFile`을 throw. 미래 코드가 대용량 GLB/USDZ 파일 파트를 실수로 메모리 경로에 태우는 것을 타입 검사로 차단하고, 파일 파트는 항상 스트리밍 `writeBodyFile`만 사용(테스트 `multipartMemoryBodyRejectsFileParts`).

### 4-4. 에러 처리 순서

- **상태 코드 검사를 빈 바디 처리보다 먼저** 수행 (순서가 반대면 바디 없는 401/5xx가 성공으로 둔갑해 토큰 재발급·롤백이 전부 스킵됨).
- 204/빈 바디는 `data: nil` 엔벨로프로 정상 처리.
- 서버 에러 코드(`SpatiumAPIError.serverCode`)를 노출해 분기 처리에 활용 → [12절](#12-서버-에러-코드--클라이언트-분기).

### 4-5. 앱 직접 연결 서버 · 주소 변경 방법

| 서버 | 기본 주소 (Release 고정) | 용도 |
|------|--------------------------|------|
| Spring API | `http://210.119.12.115:8080` | 인증·프로젝트·룸·가구·AI 게이트웨이 |
| 기본 가구 에셋 | `http://210.119.12.115:3000` | 기본 카탈로그의 공개 `/data` 모델 |

사진 세그멘테이션과 3D 생성은 앱이 FastAPI에 직접 요청하지 않고, 인증된 Spring `/api/ai/*`를 거쳐 PNG/GLB 바이너리로 받습니다. 사용자 가구 모델도 Spring의 보호된 `/api/furniture/{id}/model`에서 내려받습니다. 대용량 GLB 응답은 `Data`로 합치지 않고 `URLSession` 임시 파일로 받은 뒤 앱 캐시·사용자 가구 저장소로 이동하며, 공개 `/data` 기본 가구도 같은 download task 경로를 사용합니다.

주소는 코드 상수가 아니라 `SpatiumAPIEnvironment`(런타임)를 통해 결정됩니다.

- **Release 빌드**: 이전 개발 설치본의 `UserDefaults`를 무시하고 위 배포 주소로 **고정**.
- **Debug 빌드**: `UserDefaults`에 저장된 값 사용, 없으면 위 기본값. **런타임 UI로 변경 가능**:
  - **설정 탭 → 앱 버전 텍스트를 5번 연속 탭** → 숨겨진 **"개발자 설정" 시트**가 열림.
  - 시트의 "연결 (개발자 전용)"에서 Spring API/기본 가구 에셋 2개 주소를 각각 `TextField`로 직접 입력 → `SpatiumAPIEnvironment`의 `@Published` 값에 바인딩되어 즉시 반영·`UserDefaults`에 저장.
  - (관련 파일: `Features/Settings/SettingsView.swift`의 `DeveloperSettingsSheet`, `handleVersionTap`)

## 5. 인증 · 세션

- **이메일 로그인/회원가입** + **Google·Apple 소셜 로그인** + **게스트 모드**.
- **게스트 모드 범위**:
  - 사용 가능: 로컬 프로젝트·룸, RoomPlan 스캔, 스캔 파일 공유, 로컬 3D 편집·임시 저장, 기본 가구 카탈로그.
  - 로그인 필요: Spring이 JWT를 요구하는 AI 배경 제거·3D 가구 생성, 스캔 서버 업로드, 서버 프로젝트 저장·동기화, 사용자 가구 동기화, 프로필·계정 기능.
  - AI 가구 만들기 탭에는 로그인 필요 카드를 표시하고, 스캔 업로드를 누르면 제한 이유를 알림으로 안내한 뒤 로그인 화면으로 연결.
- 소셜 로그인은 `AuthenticationServices` 기반 네이티브 구현:
  - Google: 클라이언트 ID 역스킴 커스텀 리다이렉트(`com.googleusercontent.apps.…`), PKCE.
  - Apple: 리다이렉트 없이 고정 콜백(`spatium://oauth/callback`)으로 서버 소셜 세션 요청.
  - 서버에 소셜 계정이 없으면(에러 코드/404) 소셜 회원가입 화면으로 유도 → [12절](#12-서버-에러-코드--클라이언트-분기).
- **토큰 구조**: accessToken(요청 첨부) + refreshToken.
  - 개편된 백엔드는 refreshToken을 **httpOnly 쿠키**(path=`/api/auth`)로 내려줌.
  - `AuthService`가 쿠키 저장소에서 값을 꺼내 **키체인**에 보관 → 기존 세션 로직 재사용.
  - 재발급은 "바디 → 쿠키 → 기존 저장분" 순으로 폴백. 어디에도 없으면 빈 토큰 저장 대신 디코딩 실패 처리(조용한 로그아웃 방지).
- 최근 커밋에서 **소셜 로그인 인증 과정의 강제 종료(크래시) 방지** 처리 반영 (`f61e706`).

- **가로모드(compact-height)**: 공통 `AppHeader`/`AppFooter`가 `verticalSizeClass == .compact`에서 더 작은 간격과 가로형 탭 구성으로 전환 (`61fe541`).
- **제한적 Liquid Glass 앱 크롬**: iOS 26 이상에서는 공통 `AppHeader`/`AppFooter`를 각각 하나의 둥근 네이티브 `glassEffect` 표면으로 표시한다. 풋터는 `safeAreaBar`와 하단 `soft` 스크롤 엣지 효과로 등록해 스크롤 카드·글자가 글래스 뒤에서 점진적으로 흐려지며 사라지고, 홈 인디케이터 영역까지 별도의 카드가 겹쳐 보이지 않도록 구성한다. 선택된 탭의 밝은 배경은 모든 탭에서 같은 너비를 사용하고, 세로모드 44pt·가로모드 32pt 높이와 대칭 상하 여백으로 글래스 바 안에 정렬한다. 콘텐츠 카드·목록·큰 시트에는 효과를 확장하지 않아 Spatium의 크림·브라운 서페이스와 렌더링 비용을 유지하며, iOS 17~25에서는 같은 둥근 형태의 기존 `ultraThinMaterial`과 `safeAreaInset` 조합으로 폴백한다.
- **전역 오류 알림 우선순위** (`eee9d04`): 흐름 오류(스캔/API) → 로컬 저장(캐시) 오류 순으로 **한 번에 하나씩** 표시. 로컬 저장 오류에는 **"다시 시도" / "나중에"** 버튼을 제공해 기기 저장 실패 원인과 작업 유실 가능성을 안내.

### 7-1. 방 스캔 (Scan)

```
ScanPreparationSheet (한 번에 방 하나 안내·확인)
  → RoomCaptureViewRepresentable (RoomPlan)
  → CapturedRoom

```

- LiDAR 미지원 기기는 스캔 시작 시 안내 후 차단(`RoomCaptureSession.isSupported`).
- **방 단위 스캔 원칙**: 실제 카메라를 열기 전에 `ScanPreparationSheet`에서 "한 방 = 스캔 1회", "문 너머로 이동하지 않기", "다른 방은 새 스캔으로 시작하기"를 확인하게 함. 안내는 원형 스캔 아이콘, 둥근 주의·단계 카드와 캡슐형 CTA로 시각 톤을 통일하고, 본문이 스크롤되는 큰 글씨에서도 시작 버튼을 시트 하단에 고정. 스캔 중에도 `ScanGuidanceBanner`가 같은 원칙을 상단에 계속 표시해 집 전체를 한 세션으로 담을 때 발생하는 벽 경계·가구 위치·카테고리 인식 저하를 예방.
- 게스트는 스캔 리뷰·파일 export·로컬 편집은 계속 사용할 수 있지만, 서버 업로드는 사용할 수 없음. `ScanReviewView`가 항상 제한 문구를 보여주고 업로드 탭 시 로그인 알림을 표시.
- `exportPackage()`는 메시 export가 수 초짜리 동기 작업이라 **`Task.detached`로 백그라운드 실행**(UI 멈춤 방지). 메타데이터(변환행렬·치수)만 메인에서 인코딩.
- 스캔 직후 **로컬 placeholder 룸**을 즉시 목록에 추가 → 업로드 성공 시 서버 룸으로 교체.
- 스캔 세션 자체에 **생명주기 가드**가 있음(`RoomCaptureSessionLifecycle`): 첫 스캔 업데이트 전에는 stop을 허용하지 않음(유닛 테스트로 검증).
- **"다시 스캔"** 크래시 방어 → [14절](#14-함정--재발-방지-노트).

### 7-2. 3D 룸 에디터 (Editor) — 가장 복잡한 화면

한 덩어리였던 뷰모델·씬이 최근 리팩터링(`29f3bbc`, `ab82cbc`, `e63512d`)으로 **기능별 파일로 분리**됐습니다.

**뷰모델** — `RoomEditorViewModel`(공통 상태·초기화·모드 전환·서버 저장 조정) + 확장 4개:
- `+History` — undo/redo, 슬라이더 히스토리 트랜잭션, 스냅샷 복원
- `+DraftPersistence` — 임시 저장 debounce, utility 직렬 읽기·쓰기·삭제, revision 순서 보장, 실패 재시도, 복구본 확인·복원·폐기
- `+FurnitureEditing` — 가구 추가·이동·회전·높이·크기·교체·삭제, 문/창 벽 메우기, 1인칭 배치 제한
- `+DecorEditing` — 책장 꾸미기(소품 배치)

**씬(SceneKit)** — `RoomEditorSceneView`(UIViewRepresentable 연결·Coordinator 상태) + 역할별 확장 4개(`+Shell` 방 셸 구성·벽/바닥 색, `+Selection` 선택 하이라이트·동기화, `+Measurements` 치수선·라벨, `+WallFacing` 벽 투명화·메시 분류)와 지오메트리 타입 파일(`RoomEditorSceneGeometry` — footprint·벽 콜라이더·수평 경계) + 컨트롤러 5개:
- `RoomEditorCameraController` — Skyview/3D/1인칭 카메라 전환, **1인칭 시선·보행·충돌**
- `RoomEditorInteractionController` — 탭 선택, 가구/피규어 드래그, 제스처 우선순위·hit-test
- `RoomEditorFurnitureNodeController` — 가구 노드 생성·재사용·렌더 시그니처·벽 메우기 노드
- `RoomEditorFurniturePlacementController` — 방 경계 계산, 크기 맞춤, **벽 침범 해소·벽 콜라이더 생성**, 이동 범위 제한
- `RoomEditorDecorNodeController` / `RoomEditorShelfDetector` — 꾸미기 노드·선반 지지면 감지

**가구 모델 로더** — `FurnitureModelLoader`가 GLTFKit2로 파싱한 템플릿과 계층 bounds를 재사용합니다. 정적 템플릿 캐시는 최근 사용 순서(LRU)로 **최대 8개**, 원본 GLB 파일 크기 합계 **64MiB**를 기준으로 제한하고 오래된 항목부터 제거합니다. 단일 GLB가 예산보다 크면 반복 파싱을 피하려고 최근 모델 하나만 유지하며, 시스템 메모리 경고 때는 전체 캐시를 비웁니다. 퇴거 대상은 다시 만들 수 있는 템플릿뿐이므로 현재 SceneKit 씬에 복제되어 표시 중인 가구 노드는 유지됩니다.

**편집 UI** — `RoomEditorView`(본체·내비·툴바)에서 `RoomEditorCatalogPanel`(가구 카탈로그 시트·검색·카테고리 칩·상품 행), `RoomEditorCanvasChrome`(3D 캔버스·뷰바·색상 팔레트·배치 안내), `EditorFooterView`, `EditorFurnitureSelectionControlsView`, `EditorDecorControlsView`, `EditorViewHelpSheet`로 분리.

#### 뷰 모드 (`RoomViewMode`)
- `3D`(기본, 궤도 회전) / `스카이뷰`(위에서) / **`1인칭`**(눈높이 둘러보기).
- **1인칭이 "앱 전용, 서버 미동기화"인 이유**: 백엔드 스펙의 `RoomViewMode`에는 `3D`/`SKYVIEW`만 존재하고 **`PERSON`은 앱에서 추가한 확장**입니다(`viewModeRawValuesMatchSpec` 테스트가 서버 값 두 개만 고정). 게다가 1인칭은 **레이아웃 데이터가 아니라 카메라 시점**일 뿐이라 저장할 상태가 없습니다. `PERSON`을 서버에 올리면 계약 테스트가 지키는 스키마가 깨지므로 의도적으로 앱 로컬에만 둡니다. (웹 프론트에도 별도로 1인칭이 추가됨 — 커밋 `b90ce52`)
- 1인칭에서는 가구 추가가 막힘("1인칭 시점에서는 가구를 추가할 수 없어요").

#### 가구–벽 충돌 처리
씬의 벽 시스템이 렌더·드래그·1인칭 이동에서 **일관되게** 충돌을 처리합니다.

- **`WallCollider`** (`RoomEditorFurniturePlacementController`): 방 셸(USDZ)의 각 벽면에서 충돌 면을 생성. 기본은 geometry element에서 실제 벽 평면을 읽고, **읽을 수 없는 특수 USD 메시만 OBB(방향성 바운딩 박스) 추정으로 보완**.
- **`infillColliders`(벽 메우기 콜라이더)** (`RoomEditorFurnitureNodeController`): 셸 벽 콜라이더는 문/창 개구부 자리가 비어 있으므로, 그 자리를 메운 패널을 별도 박스 콜라이더로 만들어 **실제 벽처럼 가구 이동을 막음**.
- **가구 배치/이동 시** (`RoomEditorFurniturePlacementController`): `resolveWallPenetration…`이 벽 관통량(`penetration`)을 계산해 벽 법선 방향으로 밀어냄(`wallContactMargin` 여유 포함). `pendingWallResolveItemID`로 방금 배치/크기변경한 항목을 대상으로 지정.
- **1인칭 이동 충돌** (`RoomEditorCameraController`): SceneKit 기본 카메라 컨트롤로는 충돌을 걸 수 없어 **직접 처리** — 몸통을 반경 `personBodyRadius = 0.25m`의 원기둥으로 보고 X/Z 축을 분리 이동(`personIntersectsFurniture` / `personIntersectsWall`)해 벽·가구를 통과하지 못하게 함. 보행/시선 파라미터(최대 탭 이동 2.8m, 걷기 속도 1.9m/s 등)는 멀미 저감을 위해 제한.
- 위치/크기/높이 조정은 모두 **범위 clamp**(바닥~천장, 가구 크기 범위 등)로 안전하게 제한.

#### undo / redo (`RoomEditorViewModel+History`)
- **인메모리 스냅샷**(`EditorSnapshot`) 히스토리. **`historyLimit = 30`** (undo·redo 각각 최대 30개, 초과 시 오래된 것부터 제거). 디스크가 아닌 **메모리** 보관.
- `beginHistoryTransaction`/`endHistoryTransaction`으로 슬라이더 드래그 같은 연속 조작을 **한 단위로 병합**(테스트: `editorHistoryCoalescesSliderGestureAndSupportsRedo`).
- 새 조작이 들어오면 redo 히스토리는 비움.

#### 임시 저장(draft) · 복구 (`RoomEditorViewModel+DraftPersistence`)
- 편집 중 650ms debounce 후 로컬 파일에 draft 자동 저장(`draftSaveState`: idle/saving/saved/failed). JSON 인코딩·초기 읽기·디코딩·atomic 쓰기·삭제는 앱 전체 공용 utility 직렬 큐에서 실행.
- 모든 저장·삭제에 전역 증가 revision을 부여해 취소된 이전 쓰기가 최신 복구본이나 사용자의 폐기 결과를 되돌리지 못하게 함(테스트: `newerDraftRemovalRejectsAnOlderDelayedWrite`).
- 앱을 다시 열면 메인 스레드 밖에서 복구 가능한 draft를 읽어 감지(`hasRecoverableDraft`)한 뒤 이어서 편집(테스트: `editorDraftPersistsAndRestoresLatestLayout`, `editorDraftReadEncodingWritingAndRemovalRunOutsideMainThread`).
- 저장 실패 시 안내 + 재시도(테스트: `editorDraftSaveFailureIsVisibleAndCanBeRetried`) — 최근 커밋 `d27a244`의 **임시 저장 실패 복구·안전한 종료**.
- 저장된 `RoomLayout` 스냅샷과 현재 레이아웃의 값 비교로 실제 변경 여부(`hasUnsavedChanges`) 판별. 비교할 때마다 JSON을 다시 인코딩하던 할당 제거.

#### 저장
- `POST /api/rooms/save`. 스캔에서 바로 연 경우 소속 프로젝트 ID로 서버 룸을 새로 생성 → `MainTabView`가 로컬 placeholder를 서버 룸으로 교체(중복 방지).
- 서버 룸을 다시 열 때 `GET /api/rooms/{roomId}/scene`의 편집 확정본과 USDZ Base64를 사용. 모델 디코딩·`RoomScenes/scene-<roomId>.usdz` atomic 쓰기는 utility 직렬 큐에서 수행해 화면 전환을 막지 않고, 손상 Base64는 기존처럼 USDZ URL 없이 메타데이터 편집 상태로 처리(테스트: `roomSceneBase64DecodingAndWritingRunOutsideMainThread`, `invalidRoomSceneBase64DoesNotCreateACacheFile`).

### 7-3. 사진 → 3D 가구 생성 (ImgTo3D)

`ImgTo3DView`는 **6단계 스텝 머신**(`ImgTo3DStep`)으로 진행됩니다.

> Spring `/api/ai/*`가 JWT를 요구하므로 **가구 만들기는 로그인 사용자 전용**입니다. 게스트에게는 위저드 대신 이유·이용 불가 범위·로그인 버튼이 포함된 제한 카드를 표시합니다.

```
① upload       사진 선택 (HEIC 등 호환 안 되는 포맷은 PNG로 자동 변환)
② name         가구 이름 입력
③ segmentation 객체 분리 (모델 선택: Grounded-SAM2 등)
④ generation   3D 모델 생성 (Spring AI 게이트웨이)
⑤ correction   결과 확인/보정
⑥ save         내 가구로 저장 → UserFurnitureStore
```

- 상단에 STEP n/6 진행 인디케이터, 완료 단계 체크 표시.
- 단계 전환은 스프링 애니메이션 + `task(id: step)`로 각 단계 진입 시 준비 작업 수행.
- 사진 선택 직후 ImageIO 전처리를 **취소 가능한 백그라운드 작업**으로 실행. 업로드용 긴 변은 최대 **2048px**, 파일은 Spring/FastAPI 계약과 같은 **10MiB 이하**로 제한하고, 화면에는 최대 **1024px 미리보기**만 유지해 고해상도 원본의 전체 픽셀 버퍼가 SwiftUI 상태에 남지 않게 함. 제한 안의 PNG/JPEG 바이트는 그대로 보존하며 HEIC 등만 PNG로 변환하고, 연속 선택 시 요청 ID가 지난 결과는 화면에 적용하지 않음.
- 3D 생성 응답은 임시 GLB 파일로 내려받고 `glTF` 헤더 4바이트만 확인한 뒤 생성 모델 캐시로 이동. 응답 전체와 저장용 복사본을 각각 `Data`로 유지하지 않으며, 실패·취소 시 임시 파일을 정리함.
- 메인 탭 전환 애니메이션과 위저드 진행 상태는 유지하면서, 가구 만들기 탭이 숨겨지거나 앱이 백그라운드로 가면 **화면용 디코딩 `UIImage`와 보정 단계의 `SCNView`·메시·텍스처·정점 샘플 캐시를 해제**. 압축 이미지 Data, 세그멘테이션·생성 상태, GLB URL은 남겨두고 다시 활성화할 때 필요한 미리보기와 3D 씬만 재구성함. `dismantleUIView`에서 사용자 조작 제스처·카메라·scene 참조까지 명시적으로 분리함.
- 세그멘테이션 결과(`ImgTo3DSegmentationResult`)는 상태로 보관하고, 생성 결과(`ImgTo3DGenerationResult`)의 임시 GLB는 즉시 캐시 URL로 이전. 모델 변경 시 기존 생성 결과를 리셋.
- 저장 시 보정된 변환을 유효한 GLB 래퍼로 구워 넣음. GLB 전체를 `Data`로 읽어 청크·본문·결과를 여러 번 복제하지 않고, 백그라운드에서 JSON 청크만 메모리 수정한 뒤 BIN 등 나머지 청크는 **1MiB 버퍼로 파일 간 스트리밍 복사**. 취소·실패 시 불완전한 임시 파일을 제거하고 성공한 목적 파일만 원자적으로 교체하며, 사용자 가구 저장이 끝나면 중간 보정 파일도 삭제함(테스트: `correctedTransformIsBakedIntoValidGLBWrapper`, `correctedTransformFileStreamingPreservesBinaryPayload`, `invalidGLBStreamingBakeLeavesNoDestinationOrTemporaryFile`).
- 보정 파일을 사용자 가구 저장소로 옮길 때의 모델 복사와 카탈로그 JSON 저장도 메인 액터 밖의 직렬 큐에서 실행. 새 모델은 staging 파일로 복사한 뒤 교체하고 기존 파일은 임시 backup으로 보존해, JSON 원자 쓰기가 실패하면 새 파일과 공개 목록을 적용하지 않고 기존 GLB를 복구함.
- **파일 구조** (`3260038`에서 1,600줄+ 단일 뷰를 분리): `ImgTo3DView`(단계 상태·비동기 처리·저장 흐름만) + 단계별 전용 뷰 — `ImgTo3DInputSteps`(업로드/이름), `ImgTo3DProcessingSteps`(객체 분리/3D 생성), `ImgTo3DCorrectionStep`+`ImgTo3DCorrectionControls`(보정 뷰어·이동/회전/크기/축 컨트롤), `ImgTo3DSaveStep`, `ImgTo3DWizardChrome`(진행 인디케이터 등). 하위 뷰에는 필요한 값·Binding·콜백만 전달.
- **가로모드**: 전체 화면 너비 사용 + 긴 단계 콘텐츠 스크롤. 보정 단계는 **모델 뷰어와 조작 패널을 좌우 배치**하고 패널만 독립 스크롤 (`61fe541`).

### 7-4. 프로젝트 · 룸 관리 (Rooms)

`ProjectStore`가 담당. 핵심 규칙:

- **local- ID**: 게스트/미동기화 프로젝트·룸은 `local-xxxx` ID. 서버 호출을 건너뛰고 로컬만 조작. 로그인 시 서버 목록이 캐시를 덮어써 사라질 수 있어, 로그인 화면이 게스트 로컬 프로젝트 수를 백그라운드에서 확인하고 사전 경고. 확인 중에는 이메일·Apple·Google 로그인만 잠시 보류하며 게스트 둘러보기와 회원가입은 유지.
- **목록 병합**: 서버 목록으로 통째 교체하면 이미 불러온 방 목록·업로드 전 로컬 방이 사라짐 → 같은 프로젝트는 기존 `rooms` 유지하며 병합. 서버가 같은 ID를 중복으로 내려줘도(백엔드 버그) 크래시하지 않도록 방어 → [14절](#14-함정--재발-방지-노트).
- **개수 집계 최적화**: 프로젝트별 방 개수를 순차 N+1이 아니라 **`TaskGroup` 병렬 요청 + 마지막 1회 저장**으로 처리. 방별 항목 수도 스캔 JSON을 병렬로 받아 도착하는 대로 화면에 갱신하되, 전체 캐시 기록은 모든 결과를 합친 뒤 1회만 실행.
- **낙관적 삭제/이름변경**: 즉시 반영 후 실패 시 원위치 복원.

## 9. 오프라인 · 캐시 · 동기화 전략

### 9-1. 캐시 · 세션 격리
- **디스크 캐시**: `ProjectStore`가 프로젝트 목록을 `Documents/projects.json`에 atomic 저장. 앱 시작 시 로그인 상태면 캐시 파일 읽기와 JSON 디코딩을 utility 직렬 큐에서 실행해 먼저 보여준 뒤 서버 동기화를 이어감. 인코딩·파일 쓰기·로그아웃 삭제도 같은 큐에서 실행하며, 로드 중 로그아웃·로컬 편집·새 저장으로 revision이 바뀌면 이전 파일 스냅샷을 적용하지 않음.
- **최신 저장 순서**: 모든 캐시 변경에 단조 증가 revision을 부여. 더 최신 revision이 큐에 들어온 뒤 도착한 오래된 저장·삭제는 건너뛰어 연속 편집이나 로그아웃 중 이전 스냅샷이 최신 상태를 되돌리지 못하게 함.
- **캐시 저장 실패 복구** (`eee9d04`): 예전에는 `saveCache`가 인코딩·쓰기 오류를 `try?`로 조용히 무시했지만, 지금은 **모든 오류를 명시 처리**하고 상태를 `ProjectCachePersistenceState`(idle/saved/failed)로 공개.
  - 저장 실패해도 현재 목록은 **메모리에 유지**, 사용자가 계속 작업하면 **재시도 시점의 최신 전체 목록**을 다시 기록(`retryLocalPersistence`).
  - 로그아웃 시 캐시 파일 삭제 실패도 무시하지 않고 실패 상태로 전환.
  - UI(`ContentView`)는 "다시 시도 / 나중에" 버튼이 있는 알림으로 작업 유실 가능성을 안내.
- **세션 격리**: 캐시는 로그인 세션 전용. 로그아웃/계정 전환 시 메모리·디스크 캐시를 모두 비워 이전 계정 데이터가 다음 사용자에게 남지 않게 함.
- **비차단 새로고침**: 당겨서 새로고침·자동 동기화는 실패해도 기존 화면 데이터를 유지(일시적 네트워크 실패를 전역 오류 팝업으로 승격하지 않음, `silently` 플래그).
- **방 스캔 자산 캐시**: `RoomScanAssetService`가 `RoomScans/<roomId>` 디렉터리 생성, 캐시 확인, 다운로드 임시 파일 설치, JSON 읽기·디코딩·편집 아이템 변환, 저장 후 무효화를 앱 전체 공용 utility 직렬 큐에서 실행. 방별 항목 수 병렬 집계와 편집기 진입이 같은 파일을 동시에 받으면 먼저 설치된 캐시를 재사용.
- **설정 캐시 관리**: `RoomScans`·`RoomScenes`·`Spatium` 캐시의 디렉터리 순회, 파일 크기 합산, 삭제를 utility 우선순위의 백그라운드 작업으로 실행. Application Support의 저장 완료된 사용자 GLB와 관리 대상 밖 파일은 삭제하지 않음.

### 9-2. 로컬 → 서버 승격
- 스캔 placeholder 룸, 게스트 로컬 프로젝트가 업로드/로그인 시 서버 자원으로 교체.

### 9-3. 역방향 · 충돌 케이스 (서버 권위 모델)
현재 앱에는 **오프라인 작업 큐(outbox)가 없습니다.** 로컬 낙관적 변경은 "즉시 반영 → 서버 호출 → 실패 시 롤백"만 하며, **다음 성공한 `refresh()`가 서버 목록으로 통째 재구성**합니다(서버 권위). 이 모델에서 갈리는 케이스:

- **오프라인에서 프로젝트 삭제 → 서버 동기화 전 앱 강제 종료**:
  삭제는 즉시 로컬 반영 + 캐시 저장되지만 서버 호출이 완료되지 못함. 재실행 후 로그인/새로고침이 성공하면 **서버 목록이 캐시를 덮어써 삭제했던 프로젝트가 다시 나타납니다**(서버에는 여전히 존재하므로). 즉, 서버에 도달하지 못한 삭제는 유지되지 않음.
- **오프라인에서 이름 변경 → 강종**: 위와 동일하게 다음 서버 동기화에서 서버 값으로 되돌아감.
- **오프라인에서 생성한 `local-` 프로젝트 → 강종**: 캐시에 남아 재실행 시 계속 보이지만, **로그인 시 서버 목록이 캐시를 덮으면 사라질 수 있음** → 이 때문에 로그인 화면이 "게스트 로컬 프로젝트 n개가 사라질 수 있음"을 사전 경고(`guestLocalProjectCount`).

> **정리**: 서버에 반영되지 못한 로컬 변경은 "서버 권위 새로고침" 시점에 서버 상태로 수렴합니다(로컬 변경 손실 가능). 오프라인 편집의 지연 재전송(outbox) 및 삭제 tombstone은 **현재 미구현**이며, 향후 개선 후보입니다.

## 10. 접근성 · UX 디테일

- **VoiceOver 대응**: 3D 캔버스를 직접 탭하기 어려운 사용자를 위해 책장 지지면(선반 레벨)을 목록으로 선택(`DecorShelfLevel`). 소품 배치·이동·설명을 접근성 컨트롤로 수행(테스트: `decorAccessibilityControlsPlaceMoveAndDescribeFigure`).
- **큰 글씨(Dynamic Type)**: 책장 꾸미기 컨트롤이 접근성 글자 크기에서 스크롤 가능하도록 대응(커밋 `e34d2cb`, UI 테스트 `testDecorControlsBecomeScrollableAtAccessibilityTextSize`). 가로모드에서도 동일 보장(`testLandscapeDecorControlsScrollAtAccessibilityTextSize`).
- **가로모드(compact-height) 전면 대응** (`61fe541`): 헤더/탭 바 압축형 전환, 온보딩은 스크린샷·설명 좌우 배치 + 페이지 표시·다음 버튼 한 줄 배치, 새 프로젝트 화면은 스크롤 구조 + `safeAreaInset` 하단 버튼으로 키보드가 떠도 생성 버튼 사용 가능, 이름 수정·확인 시트는 큰 시트 높이 + 내용 잘림 시 스크롤.
- **측정 모드(웹 동일 방식)**: 바닥 mesh 삼각형을 Y높이별로 그룹핑해 가장 넓은 그룹을 메인 바닥으로 선택(`calculateRoomMeasurements` 대응), 외곽 edge(사용 횟수 1)를 실제 방 모양 그대로 치수선(선+눈금+cm 라벨, 오프셋 0.14)으로 표시. 방 높이선(테두리 모서리 +0.18)과 실제 바닥 폴리곤 면적 배지("면적 m²·평", `room-area` 배지 대응)를 함께 표시하고, 측정 중 선택 가구의 가로·세로·높이 라벨(`dimension-label` 대응)도 보여줌. 바닥 mesh가 없으면 bounding box 직사각형 폴백. 뷰 모드와 무관하게 같은 구성.
- **SceneKit 증분 갱신**: 같은 가구 선택에서는 하이라이트 geometry를 재사용하고, 측정 치수선은 표시 상태나 실제 transform이 바뀔 때만 다시 생성. 측정 중 가구 드래그의 치수선 갱신은 최대 60Hz, 카메라 방향 기반 벽 투명도 계산은 최대 30Hz로 제한하고 제스처 종료 시 최종 위치를 강제 반영해 120Hz 기기의 불필요한 노드 생성·렌더 스레드 계산을 줄임.
- **숨은 ImgTo3D 리소스 수명**: 탭 상태를 보존하려고 `ImgTo3DView`를 상시 유지하더라도, 비활성 탭·백그라운드에서는 디코딩 이미지와 SceneKit 보정 하위 트리를 유지하지 않음. 재진입 시 압축 Data·GLB URL에서 화면 리소스만 다시 만들어 탭 전환 부드러움과 메모리 회수를 함께 유지.
- **대용량 GLB 파일 처리**: AI 생성 모델·사용자 가구 다운로드는 download task의 임시 파일을 최종 위치로 이동하고, 사용자 GLB 업로드도 파일 URL과 임시 multipart 파일을 사용. 보정 변환 저장은 JSON 청크만 메모리에 올리고 BIN 청크를 1MiB 단위로 스트리밍해 전송·보정 크기만큼의 `Data` 복제가 동시에 남지 않게 함. 파일 작업 전체를 메인 액터 밖에서 실행해 저장 로딩 UI의 응답성을 유지.
- **에디터 복구본 비차단·최신 순서 저장**: draft JSON 읽기·디코딩·인코딩·atomic 쓰기·삭제를 utility 직렬 큐에서 실행하고 전역 revision으로 늦게 도착한 이전 저장을 거부. 저장 상태 비교는 JSON fingerprint 생성 대신 `RoomLayout` 값 비교를 사용해 편집 종료·undo/redo 경로의 메인 스레드 할당을 줄임. 게스트의 로컬 편집·임시 저장·복구 동작은 유지.
- **서버 룸 모델 비차단 복원**: scene 응답의 대용량 USDZ Base64 디코딩과 캐시 디렉터리 생성·atomic 쓰기를 utility 직렬 큐에서 실행. 동일 방의 동시 복원 파일 교체를 직렬화하고 손상 데이터의 기존 폴백을 유지. 인증 서버 룸에만 적용되므로 게스트의 로컬 스캔·공유·편집 범위에는 변화 없음.
- **로컬 방 스캔 비차단 로드**: 파일 기반 로컬·폴백 방의 스캔 JSON 읽기·디코딩·아이템 변환과 `RoomScans` 캐시 생성·설치·삭제를 utility 직렬 큐로 이동. 파일 URL, 서버 인증 헤더 제한, 캐시 이름, 실패 시 기존 표시값 유지 동작을 보존하며 게스트 로컬 스캔·편집 범위도 유지.
- **사용자 가구 비차단 로드·영속화 직렬화**: `catalog.json` 초기 읽기·디코딩·레거시 단위 변환·최신순 정렬과 GLB 파일 작업 및 카탈로그 인코딩·쓰기를 utility 직렬 큐에서 순차 실행. 메인 액터의 비동기 변경 잠금이 초기 로드를 먼저 기다리므로 앱 시작 직후 저장·삭제·서버 승격이 겹쳐도 기존 목록이나 최신 `catalog.json`을 덮어쓰지 않음. 디스크 성공 후에만 `@Published` 목록을 갱신하고 실패 시 staging·backup 트랜잭션으로 기존 모델을 복구. 게스트의 기기 저장분 보존 동작은 유지.
- **프로젝트 캐시 비차단·증분 저장**: `projects.json` 초기 읽기·디코딩과 전체 인코딩·atomic 쓰기·삭제를 utility 직렬 큐로 이동하고 revision으로 최신 스냅샷의 최종 반영을 보장. 로그인 화면의 게스트 프로젝트 유실 위험 개수도 백그라운드에서 계산. 프로젝트 방 개수와 방 항목 수의 병렬 집계는 `ProjectStore.withBoundedConcurrency` 슬라이딩 윈도로 순간 동시 요청을 최대 5개로 제한하며, 집계 중에는 도착 건마다 전체 캐시를 다시 쓰지 않고 각 집계 완료 후 한 번만 저장. 실패 상태·최신 목록 재시도와 게스트 로컬 작성 흐름은 유지.
- **벽·바닥 색 원본 복원**: 색상 팝오버에 "기본 색상으로 되돌리기" 스와치를 제공(웹 프런트 동일). 색을 고르지 않은 스캔 방은 벽·바닥을 기본색으로 덮지 않고 스캔 원본 재질을 유지하며, 첫 tint 전에 원본 diffuse를 기록해 해제(nil) 시 복원(`applySurfaceTint`, 프런트 `spatiumDefaultWallColor` 대응). `RoomSpace.wallColor`는 optional이 되었고 nil = 원본 유지(박스 방은 기본색 폴백).
- **가구 모델 템플릿 캐시**: GLTFKit2 파싱 결과를 무제한 딕셔너리에 누적하지 않고 최대 8개·원본 GLB 합계 64MiB 기준의 LRU로 관리. 재사용 이점은 유지하면서 여러 방과 사용자 모델을 탐색한 뒤 사용하지 않는 SceneKit 템플릿이 세션 끝까지 남는 것을 방지.
- **프로필 이미지 비차단 처리**: 백엔드 base64 data URL은 SwiftUI `body`에서 반복 변환하지 않고 source 변경 시 한 번만 백그라운드 디코딩·512px 다운샘플링. 사진 보관함에서 고른 아바타도 원본 전체 픽셀 버퍼를 메인 스레드에 만들지 않고 ImageIO로 512px JPEG와 미리보기를 함께 생성. 설정 화면의 캐시 용량 계산·삭제도 백그라운드에서 처리.
- **홈 반복 애니메이션 게이트**: 홈의 스캐너 회전·pulse `repeatForever` 애니메이션은 "홈 탭 표시 + 앱 포그라운드 + Reduce Motion 꺼짐"일 때만 실행(`HomeDashboardView.shouldRunAmbientAnimations`). 가구 만들기 탭 전환으로 홈이 opacity 0으로 계층에 남는 동안과 백그라운드에서는 정지해 유휴 CPU·GPU·배터리 사용을 제거하고, 복귀 시 항상 초기값으로 리셋 후 시작해 중복 시작·속도 누적이 없음. Reduce Motion이 켜지면 진입 슬라이드 애니메이션도 생략.
- **성능 signpost 계측**: `PerformanceSignposts`(OSSignposter, 카테고리 Editor/RoomScene/ImgTo3D)로 에디터 씬 생성(`editor.load`)·USDZ 셸 콜드 파싱(`editor.shell.parse`)·GLB 템플릿 콜드 파싱(`editor.furniture.template.parse`)·가구 노드 재생성(`editor.furniture.nodes.rebuild`)·첫 렌더 프레임(`editor.firstFrame`)·서버 룸 다운로드/복원(`roomScene.download`/`roomScene.materialize`)·ImgTo3D 파싱/자동 정렬(`imgTo3D.model.parse`/`imgTo3D.autoAlign`) 구간을 Instruments에서 구분 측정 가능. 캐시 적중 경로는 계측하지 않아 콜드 비용만 드러남.
- **햅틱**: 업로드 성공/실패 시 `Haptics.success()`/`error()`.
- **키보드 회피**: 탭 전환 시 키보드 자동 내림. 새 프로젝트·프로필 입력은 `fullScreenCover`로 안전영역 자체 관리(UI 테스트로 검증).
- **전역 오류 알림**: 흐름 오류와 스토어 오류를 하나의 `alert`로 통합.

## 11. 백엔드 연동 · 배포 상태

- 서버가 현재 **도메인 없이 IP + 평문 HTTP**로 서비스 → `Info.plist`에서 ATS `NSAllowsArbitraryLoads = true` (전역 허용).
- **권한(Info.plist)**: 카메라(스캔·모델링), 사진 보관함(가구 사진), 로컬 네트워크.
- **개인정보 매니페스트**: `Spatium/PrivacyInfo.xcprivacy` 존재 (Required Reason API 선언, 테스트 `privacyManifestDeclaresRequiredReasonAPIs`).
- Google OAuth용 커스텀 URL 스킴 등록 + 비수출 암호화 답변(`ITSAppUsesNonExemptEncryption=false`) — 릴리스 메타데이터 테스트로 고정.
- ⚠️ **배포 준비 단계**: 백엔드 HTTPS 전환 작업 진행 중. 도메인+HTTPS가 붙으면 ATS 전역 허용을 제거하고 도메인 예외로 좁힐 예정. **그 전까지 ATS·서버 주소는 유지.** (ATS 관련 함정은 [14절](#14-함정--재발-방지-노트) 참고)

## 12. 서버 에러 코드 · 클라이언트 분기

서버 공통 에러 바디는 `{ statusCode, code, message, errors[] }`이며, 앱은 `SpatiumAPIError.serverCode`(=`code`)와 `statusCode`로 분기합니다. **코드에서 실제로 분기에 쓰는 케이스**:

| 상황 | 판별 조건 (코드) | 클라이언트 동작 |
|------|------------------|------------------|
| 소셜 미가입 사용자 | `serverCode`에 `NOT_FOUND` / `SOCIAL_USER` / `SIGNUP_REQUIRED` 포함 **또는** HTTP 404 | 소셜 회원가입 화면으로 유도 (`LoginView.indicatesUnregistered`) |
| 가구 생성 엔드포인트 미배포 | HTTP **404** + (`code == nil` 또는 `code == "NOT_FOUND"`) | 서버 저장 대신 **기기 로컬 저장으로 폴백** (`UserFurnitureStore.shouldSaveLocally`) |
| 액세스 토큰 만료 | HTTP **401** | refreshToken으로 1회 재발급 후 재시도, 실패 시 로그아웃 |
| 기타 서버 오류 | 그 외 non-2xx | `message`를 사용자 알림에 그대로 노출 |

> ⚠️ **부분 확인**: 위는 **클라이언트가 명시적으로 분기하는** 코드만 정리한 것입니다. 백엔드가 내려주는 전체 에러 코드 목록(enum)은 이 리포에 없으며(→ [16절](#16-관련-저장소--미확인-항목)), 분기하지 않는 나머지 코드는 공통 경로에서 `message`만 표시됩니다. 소셜 미가입 판별을 문자열 `contains`로 느슨하게 처리하는 것도 백엔드 코드 표준이 앱에 고정돼 있지 않기 때문입니다.

## 13. 테스트 전략 · 커버리지

테스트는 "회귀가 실제로 났던 지점"과 "백엔드 계약"에 집중돼 있습니다. (유닛 82개 / UI 25개 + 런치 1개, 총 ~2,500줄)

### 13-1. 유닛 테스트 (`SpatiumTests`, Swift Testing) — 영역별
- **백엔드 계약(가장 큰 축)**: 로그인/회원가입/소셜/가구/아바타 요청 바디 키, 엔벨로프 형태, 뷰모드 raw 값, ImgTo3D 옵션이 프론트/FastAPI와 일치하는지 → 스키마 드리프트 감지. 사용자 가구 GLB 파일과 JSON metadata가 실제 파일 기반 multipart 작성 경로에서도 Spring 파트 이름·Content-Type을 유지하는지 검증. ([2절](#백엔드-api-스펙과의-버전-싱크))
- **에디터 로직**: undo/redo 병합·redo, draft 저장/복구/실패-재시도·백그라운드 파일 처리·revision 순서, 꾸미기(책장) 지지면 법선 정규화·모델 정면 방향·소품 교체 정체성 유지, 1인칭에서 가구 배치 거부, 접근성 꾸미기 컨트롤, 선택·측정 SceneKit 노드 재사용.
- **스캔 생명주기**: 첫 업데이트 전 stop 불가, 세션 종료 후 리셋, export 성공 후에만 에디터 준비 완료(실패 시 재시도 상태), 취소는 오류로 표시하지 않음.
- **가구/치수**: 사용자 GLB를 카탈로그 치수에 맞춤, 레거시 cm→m 마이그레이션, 90° 회전 시 월드 치수, 사용자 가구가 렌더 카탈로그에 합류, 서버 미동기화 GLB가 메모리 `Data` 변환 없이 저장 파일 URL로 재업로드됨. 초기 카탈로그 읽기·디코딩·단위 변환·정렬이 메인 스레드 밖에서 실행되는지와 초기 로드 중 시작된 새 저장이 기존 항목을 보존하는지 검증. 동시에 시작한 사용자 가구 저장이 모두 메모리와 재실행 카탈로그에 남고, 카탈로그 쓰기 실패 시 기존 GLB·공개 목록·트랜잭션 임시 파일이 올바르게 롤백되는지도 검증.
- **이미지·3D 리소스 처리**: HEIC→PNG 변환, 제한 안의 JPEG 원본 통과, RAW 없는 카메라 캡처 PNG 인코딩, 업로드·미리보기 픽셀 상한에 따른 독립 다운샘플링, 프로필 data URL의 백그라운드 표시용 다운샘플링, 아바타의 512px JPEG 변환, 서버 룸 USDZ의 메인 스레드 밖 Base64 복원과 손상 데이터 무파일 처리, GLB 보정 시 1MiB 경계를 넘는 BIN 청크의 바이트 보존과 손상 입력의 임시 파일 정리, 가구 모델 템플릿 LRU의 개수·비용·최근 사용 퇴거 정책, ImgTo3D 뷰어 해제 시 scene·모델 노드·정점 샘플 캐시 정리.
- **인증/모델**: JWT 클레임 추출, 프로젝트 JSON 라운드트립, 로그아웃 후 캐시 유지, 릴리스 메타데이터·개인정보 매니페스트.
- **로컬 영속화**: 캐시 디렉터리 생성이 막히는 실제 쓰기 실패를 재현하고, 실패 후 추가한 프로젝트까지 재시도 성공 시 JSON에 보존되는지 회귀 검증(`projectCacheWriteFailureIsVisibleAndCanBeRetried`). 프로젝트 캐시의 초기 읽기·디코딩과 인코딩·쓰기가 메인 스레드 밖에서 실행되는지, 게스트 프로젝트 유실 위험 개수가 비동기로 계산되는지, 겹친 저장 뒤 최신 전체 스냅샷이 디스크에 남는지도 검증(`initialProjectCacheReadAndDecodingRunOutsideMainThread`, `guestProjectRiskCountReadsCacheOutsideMainThread`, `projectCacheEncodingAndWritingRunOutsideMainThread`, `overlappingProjectCacheWritesKeepNewestCompleteSnapshot`). 로컬 방 스캔은 `roomScanPackageReadDecodingAndItemMappingRunOutsideMainThread`와 `roomScanItemCountReadAndCacheInvalidationRunOutsideMainThread`로 패키지 로드·항목 집계·캐시 삭제의 비차단 실행을 검증. 에디터 복구본은 `editorDraftReadEncodingWritingAndRemovalRunOutsideMainThread`와 `newerDraftRemovalRejectsAnOlderDelayedWrite`로 비차단 파일 처리와 최신 삭제 우선 순서를 검증. 사용자 가구는 `initialCatalogReadDecodingMigrationAndSortingRunOutsideMainThread`, `mutationStartedDuringInitialLoadPreservesLoadedCatalog`, `concurrentUserFurnitureAddsPreserveEveryCatalogEntry`, `failedCatalogWriteRestoresExistingModelAndPublishedItems`로 비동기 초기 로드·경합 보존·직렬화·파일 롤백을 검증. 설정의 캐시 정리가 관리 대상 디렉터리만 합산·삭제하고 다른 파일은 보존하는지도 검증. 선반 지지면 감지(`shelfDetectorFindsSeparateHorizontalSupportLevels`)도 추가.

### 13-2. UI 테스트 (`SpatiumUITests`, XCUITest)
- 게스트가 AI 가구 만들기에 진입했을 때 제한 이유와 로그인 버튼이 노출되는지 회귀 검증(`testGuestModeExplainsImgTo3DRestrictionBeforeStarting`).
- 스캔 시작 전 방 하나 단위의 안내, 현재 방 유지 및 다른 방 새 스캔 원칙, 시작 버튼이 모두 노출되는지 회귀 검증(`testScanPreparationExplainsOneRoomPerScan`).
- 스캔 안내를 접근성 최대 글자 크기로 표시해도 본문과 분리된 하단 시작 버튼이 화면 안에서 계속 노출되고 탭 가능한지 검증(`testScanPreparationKeepsStartActionVisibleAtAccessibilityTextSize`).

- 키보드 fullScreenCover 사용(새 프로젝트/프로필), **반복 당겨서 새로고침이 차단성 오류를 띄우지 않음**.

## 14. 함정 · 재발 방지 노트

> 실제 크래시/버그에서 나온 "이건 이렇게 하면 다시 터진다" 기록. 리팩터링 시 특히 주의.

1. **ATS 세부 키 병기 금지** — `NSAllowsArbitraryLoads`에 `NSAllowsLocalNetworking` 등 세부 키를 함께 넣으면 전역 허용이 **무시**되어 HTTP가 전부 차단됨(iOS 10+ 문서화된 동작). IP 주소는 `NSExceptionDomains`로 예외 처리 불가(도메인 전용). (`Info.plist`)
2. **"다시 스캔" 강제 언래핑 크래시** — `ScanReviewView`가 `Binding($scanProject)` 강제 언래핑을 들고 있어, 다시 스캔 시 `scanProject`를 nil로 만들면 `EXC_BREAKPOINT` 크래시. 기존 스캔은 새 스캔이 "완료"될 때만 교체할 것. (`ContentView`/`MainTabView`)
3. **프로젝트 목록 병합 시 중복 키 크래시** — 서버가 같은 프로젝트 ID를 두 번 내려주면 `Dictionary(uniqueKeysWithValues:)`가 크래시. `uniquingKeysWith`로 첫 항목 유지 병합 사용. (`ProjectStore.refresh`)
4. **API 응답 처리 순서** — 상태 코드 검사를 빈 바디 처리보다 **먼저**. 순서가 반대면 바디 없는 401/5xx가 성공 엔벨로프로 둔갑해 토큰 재발급·롤백이 전부 스킵됨. (`SpatiumAPIClient.perform`)
5. **refreshToken 빈 문자열 저장 금지** — 빈 토큰을 키체인에 저장하면 다음 재발급이 빈 토큰으로 나가 **조용히 로그아웃**됨. 못 찾으면 디코딩 실패로 처리. (`AuthService`)
6. **401 동시 재발급 금지** — 여러 요청이 동시에 401을 맞으면 refresh rotation이 꼬임. `AuthRefreshCoordinator`(actor)로 재발급을 직렬화. (`SpatiumAPIClient`)
7. **요청 타임아웃 2초 금지** — 셀룰러/혼잡망에서 정상 요청도 끊김. 일반 요청 10초로 설정. (`SpatiumAPIClient.makeRequest`)
8. **탭 if/else 하드 컷** — 스크롤 탭과 가구 만들기 탭을 if/else로 교체하면 전환이 하드 컷. 두 컨테이너를 상시 겹쳐 opacity/scale로 전환. (`MainTabView`)
9. **로컬 저장 오류 `try?` 무시 금지** — 캐시 인코딩·파일 쓰기 오류를 `try?`로 삼키면 사용자는 작업이 저장된 줄 알고 잃는다. 실패를 상태(`ProjectCachePersistenceState`)로 공개하고 재시도 경로를 제공할 것. 재시도는 실패 시점 스냅샷이 아니라 **최신 전체 목록**을 다시 기록. (`ProjectStore`, `eee9d04`)

> **아직 기록되지 않은 함정** (있으면 같은 톤으로 여기에 추가):
> - GLTFKit2 특정 버전(현재 0.5.15) 관련 알려진 버그 — 코드에 기록된 바 없음.
> - RoomPlan의 특정 iOS 버전 이슈 — 코드에 기록된 바 없음(단, geometry element를 못 읽는 USD 메시는 OBB 추정으로 폴백하는 방어는 있음).

## 16. 관련 저장소 · 미확인 항목

### 관련 저장소
이 리포는 **iOS 앱(spatium-ios)** 전용입니다. 함께 구성되는 별도 저장소:
- `spatium-backend` — 서버 (HTTPS 작업 중)
- `spatium-frontend` — 웹 프론트엔드 (앱과 동일하게 1인칭/undo/redo 반영됨)
- `spatium-img-to-3d` — 사진→3D 변환 서버 (FastAPI)

> ⚠️ 위 세 저장소는 이 앱 작업 범위 밖입니다.

### 이 리포만으로는 확인 불가한 항목 (교차 확인 필요)
- **백엔드 API 스펙 문서(OpenAPI/Swagger 등)의 존재·위치·갱신 주체** — 앱은 [2절](#백엔드-api-스펙과의-버전-싱크)의 계약 테스트로 형태만 고정. 별도 스펙 문서와의 연결은 미기록.
- **서버 에러 코드 전체 목록** — 앱은 [12절](#12-서버-에러-코드--클라이언트-분기)의 몇 개만 명시 분기.
- **서버 버전/스키마 버전 핀** — 요청에 버전 헤더 등 없음. 계약 테스트가 유일한 방어선.
- **DB(예: Oracle)·서버 스펙** — 백엔드 저장소 소관이며 앱에는 의존 정보 없음.
