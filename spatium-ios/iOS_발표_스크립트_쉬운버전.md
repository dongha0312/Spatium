# Spatium iOS 발표 스크립트 — 비전공자용 쉬운 버전

대상: Swift와 앱 개발을 모르는 청중  
권장 발표 시간: 약 7~8분  
사용법: 각 페이지의 “발표 원고” 부분만 읽으면 됩니다. 아래 설명은 질문을 받았을 때 참고하세요.

# 1페이지 — iOS 앱 소개

## 발표 원고

“지금부터 Spatium의 iPhone 앱 부분을 설명드리겠습니다.

저희 앱을 한 문장으로 설명하면, 실제 방을 휴대폰으로 스캔한 뒤 가상공간에서 다시 꾸며볼 수 있게 해주는 앱입니다.

보통 방을 사진으로 찍으면 평면 이미지 한 장만 남습니다. 하지만 저희 앱은 방의 벽이 어디에 있는지, 문과 창문의 크기는 얼마인지, 가구가 어느 위치에 놓여 있는지까지 입체적인 정보로 저장합니다.

그래서 사용자는 스캔이 끝난 뒤에도 가구를 옮기거나 돌리고, 크기를 바꾸거나 다른 가구로 교체할 수 있습니다.

지금부터 방을 어떻게 스캔하는지, 작업 내용은 어떻게 보관하는지, 로그인하지 않은 사용자는 어디까지 쓸 수 있는지, 그리고 사진 한 장을 어떻게 3D 가구로 만드는지 차례대로 설명드리겠습니다.”

## 핵심 한 줄

“방을 사진으로 남기는 것이 아니라, 다시 꾸밀 수 있는 입체적인 설계도로 저장합니다.”

## 다음 페이지 연결

“먼저 iPhone이 실제 방을 어떻게 입체적으로 읽는지 설명드리겠습니다.”

# 2페이지 — 방을 입체적으로 읽는 과정

## 발표 원고

“방을 스캔할 때는 iPhone의 카메라와 LiDAR 센서를 함께 사용합니다.

카메라는 방의 모습을 보고, LiDAR는 물체까지의 거리를 측정합니다. 쉽게 말하면 카메라가 ‘무엇이 보이는지’를 확인하고, LiDAR가 ‘얼마나 멀리 있는지’를 재는 것입니다.

이 두 정보를 Apple의 RoomPlan이라는 기능이 합쳐서 벽, 문, 창문, 가구를 구분합니다. 그리고 각 물체가 어디에 있는지, 어느 방향을 보고 있는지, 실제 크기가 어느 정도인지 계산합니다.

저희 앱 화면과 Apple의 스캔 기능은 원래 서로 다른 방식으로 만들어져 있습니다. 그래서 두 화면 사이에 통역기 같은 연결 부분을 만들었습니다. 덕분에 사용자는 일반 앱 화면에서 바로 Apple의 방 스캔 기능을 사용할 수 있습니다.

스캔을 시작하자마자 완료 버튼을 누를 수는 없습니다. 방의 정보가 처음으로 들어온 뒤에만 완료 버튼이 활성화됩니다. 너무 일찍 끝내서 빈 결과가 만들어지는 문제를 줄이기 위해서입니다.

또 스캔 화면을 닫으면 카메라와 거리 센서도 함께 종료합니다. 화면은 사라졌는데 센서가 뒤에서 계속 작동하면 배터리와 메모리를 낭비할 수 있기 때문입니다.”

## 더 쉽게 설명하면

- 카메라: 방의 모양과 물체를 봄
- LiDAR: 물체까지의 거리를 잼
- RoomPlan: 두 정보를 합쳐 방의 입체적인 설계도를 만듦
- 앱의 연결 부분: 일반 앱 화면과 스캔 기능 사이의 통역기

## 화면을 가리키며 말할 부분

“오른쪽 화면처럼 사용자가 방을 천천히 비추면, 인식된 벽과 가구가 화면 안에서 점점 입체적인 형태로 나타납니다.”

## 질문을 받으면

### LiDAR가 없는 휴대폰에서도 되나요?

“방 스캔은 LiDAR가 있는 iPhone이나 iPad에서만 가능합니다. 지원하지 않는 기기에서는 안내 문구를 보여주고, 프로젝트나 다른 기능은 계속 사용할 수 있습니다.”

### 크기는 직접 재는 건가요?

“사용자가 줄자로 입력하는 방식이 아니라, 휴대폰이 카메라와 거리 센서 정보를 이용해 자동으로 계산합니다.”

## 다음 페이지 연결

“그럼 스캔을 시작한 뒤 최종 3D 방이 만들어질 때까지 어떤 과정을 거치는지 보겠습니다.”

---

# 3페이지 — 스캔 결과가 만들어지는 과정

## 발표 원고

“방 스캔은 네 단계로 진행됩니다.

첫 번째는 스캔 시작입니다. 사용자가 시작 버튼을 누르면 카메라와 거리 센서가 작동합니다. 정확도를 높이기 위해 한 번에 여러 방을 돌아다니지 않고, 방 하나를 한 번에 스캔하도록 안내합니다.

두 번째는 실시간 인식입니다. 사용자가 방을 천천히 비추면 벽, 문, 창문, 가구가 차례로 인식됩니다. 앱은 방 정보가 처음 들어온 뒤에만 완료 버튼을 사용할 수 있게 합니다.

세 번째는 결과 정리입니다. 사용자가 완료를 누르면 그동안 모은 정보를 한 번 더 정리합니다. 촬영 도중 흔들리거나 여러 번 감지된 정보를 합쳐 최종 방 데이터를 만듭니다.

네 번째는 편집할 수 있는 형태로 바꾸는 과정입니다. 인식된 가구마다 이름, 크기, 위치, 방향을 따로 기록합니다. 그래야 나중에 가구 하나만 선택해서 옮기거나 삭제할 수 있습니다.

완성된 결과는 두 종류의 파일로 나눠 저장합니다.

첫 번째 3D 파일은 방의 실제 모양을 보여주는 입체 모형입니다. 두 번째 정보 파일은 벽과 가구의 이름, 크기, 위치를 적어둔 설명서입니다.

비유하면 3D 파일은 완성된 모형이고, 정보 파일은 그 모형을 다시 수정하기 위한 조립 설명서입니다. 둘을 함께 사용해야 방을 보여주는 것뿐 아니라 다시 편집하는 것도 가능합니다.

스캔이 끝나면 리뷰 화면에서 방 이름과 인식된 물체, 각 물체의 크기를 확인합니다. 실제 이동, 회전, 크기 조절, 교체와 삭제는 그다음 3D 편집 화면에서 진행합니다.”

## 네 단계 요약

1. 카메라와 센서 시작
2. 벽·문·창문·가구 인식
3. 수집한 정보를 최종 방 데이터로 정리
4. 가구를 각각 선택하고 편집할 수 있는 정보로 변환

## 두 파일을 쉽게 설명하는 표현

- 3D 파일: 눈으로 보는 방 모형
- 정보 파일: 모형 속 물체들의 이름·크기·위치를 적은 설명서

## 화면을 가리키며 말할 부분

“아래쪽의 왼쪽 파일은 방의 모습을 보여주고, 가운데 파일은 가구 정보를 저장합니다. 오른쪽 리뷰 화면에서는 스캔 결과를 먼저 확인합니다.”

## 주의해서 말할 부분

리뷰 화면에서 가구를 직접 수정하거나 삭제한다고 말하면 안 됩니다.

정확한 표현:

“리뷰 화면에서 결과를 확인한 뒤, 3D 편집 화면에서 가구를 수정하거나 삭제합니다.”

## 질문을 받으면

### 왜 파일을 두 개나 만드나요?

“방의 모양만 저장하면 가구를 각각 선택하기 어렵고, 숫자 정보만 저장하면 실제 방 모습을 그대로 보여주기 어렵습니다. 그래서 모양을 담당하는 파일과 편집 정보를 담당하는 파일을 함께 사용합니다.”

## 다음 페이지 연결

“이렇게 편집을 시작하면 작업 시간이 길어질 수 있기 때문에, 중간 작업을 잃지 않도록 저장하는 기능도 중요합니다.”

---

# 4페이지 — 자동 저장과 작업 복구

## 발표 원고

“3D로 방을 꾸미는 작업은 시간이 오래 걸릴 수 있습니다. 그런데 편집 도중 앱이 종료되거나 인터넷 연결이 끊기면 작업이 사라질 수 있습니다.

이를 막기 위해 앱은 사용자의 편집 내용을 휴대폰 안에 자동으로 임시 저장합니다.

가구를 옮기거나 돌리고, 크기를 바꾸거나 삭제하면 앱은 현재 방 상태가 바뀌었다는 것을 확인합니다. 그리고 사용자의 손동작이 잠깐 멈춘 뒤 가장 최신 상태를 저장합니다.

손가락을 움직일 때마다 계속 저장하지 않고 잠깐 기다리는 이유는, 가구 하나를 옮기는 동안에도 위치 값이 수십 번 바뀔 수 있기 때문입니다. 사용자의 조작이 끝났다고 판단되는 시점에 마지막 결과만 저장하면 휴대폰의 불필요한 작업을 줄일 수 있습니다.

나중에 같은 방의 편집 화면을 다시 열면 이전에 저장된 작업이 있는지 확인합니다. 남아 있는 작업이 있으면 사용자에게 복구할지, 버릴지 선택하게 합니다.

여기서 자동 저장과 서버 저장은 서로 다릅니다. 자동 저장은 작업을 잃지 않기 위해 현재 휴대폰에 남기는 임시 메모이고, 서버 저장은 다른 기기나 웹에서도 볼 수 있도록 계정에 확정해서 저장하는 것입니다.

따라서 인터넷 문제로 서버 저장이 실패하더라도 휴대폰 안의 임시 작업은 그대로 남아 있습니다.

또 실행 취소와 다시 실행 기능을 최대 30단계까지 제공해, 실수로 가구를 움직이거나 삭제했을 때 이전 상태로 되돌릴 수 있습니다.”

## 가장 쉬운 비유

- 자동 임시 저장: 책상 위에 수시로 적는 메모
- 서버 저장: 완성본을 제출하고 보관하는 것
- 실행 취소: 방금 한 행동을 되돌리는 것
- 다시 실행: 되돌린 행동을 다시 적용하는 것

## 실제 흐름

가구 편집  
→ 잠깐 기다림  
→ 가장 최신 방 상태를 휴대폰에 저장  
→ 같은 방을 다시 열면 복구 여부 확인

## 정확하게 말해야 할 부분

“앱을 다시 켜면 자동으로 마지막 편집 화면까지 이동한다”는 뜻은 아닙니다.

정확한 표현:

“같은 방의 편집 화면을 다시 열면 저장된 작업을 복구할 수 있습니다.”

또 현재 버전에서는 로그인하지 않은 게스트 프로젝트를 앱 완전 종료 후 다시 목록에 표시하는 과정이 아직 보완 중입니다. 발표에서는 “현재 작업을 휴대폰 안에 임시 저장한다”까지만 강조하는 것이 안전합니다.

## 질문을 받으면

### 인터넷이 끊기면 작업이 모두 사라지나요?

“아닙니다. 서버 저장은 실패할 수 있지만, 편집 중인 상태는 휴대폰 안에 따로 저장합니다.”

### 캐시를 비우면 편집 중인 작업도 사라지나요?

“아닙니다. 다시 받을 수 있는 임시 파일과 사용자의 편집 복구 파일은 서로 다른 장소에 보관합니다.”

## 다음 페이지 연결

“이 로컬 저장 방식을 활용해 회원가입 전에도 핵심 기능을 체험할 수 있는 게스트 모드를 만들었습니다.”

# 5페이지 — 게스트 모드

## 발표 원고

“사용자가 앱을 설치하자마자 회원가입부터 해야 한다면, 앱의 장점을 확인하기 전에 이탈할 수 있습니다.

그래서 Spatium은 로그인 없이 둘러볼 수 있는 게스트 모드를 제공합니다.

게스트 사용자는 휴대폰 안에서 프로젝트를 만들고, 방을 스캔하고, 결과를 확인하고, 기본 가구로 3D 편집을 체험할 수 있습니다. 스캔한 3D 파일과 정보 파일을 다른 앱으로 공유하는 것도 가능합니다.

하지만 서버가 필요한 기능은 사용할 수 없습니다. 예를 들어 스캔한 방을 계정에 저장하거나, 다른 기기와 동기화하거나, 인공지능으로 사진을 3D 가구로 만드는 기능은 로그인이 필요합니다.

로그인 사용자의 프로젝트는 서버에 저장됩니다. 따라서 같은 계정으로 로그인하면 다른 기기나 웹에서도 저장된 방을 다시 확인할 수 있습니다.

게스트가 서버 기능을 누르면 단순히 오류를 보여주는 대신, 왜 로그인이 필요한지 설명하고 로그인 화면으로 연결합니다.

다만 현재 게스트 프로젝트는 로그인 계정으로 자동 이전되지 않습니다. 그래서 게스트 프로젝트가 남아 있는 상태에서 로그인하려고 하면, 기존 작업이 계정으로 옮겨지지 않는다는 경고를 먼저 보여줍니다.”

## 한눈에 비교

### 게스트가 할 수 있는 것

- 휴대폰 안에서 프로젝트 만들기
- 방 스캔하기
- 스캔 결과 확인하기
- 기본 가구로 3D 편집 체험하기
- 스캔 파일 공유하기

### 로그인이 필요한 것

- 서버에 방 저장하기
- 다른 기기와 동기화하기
- 웹에서 다시 확인하기
- 사진으로 3D 가구 만들기
- 사용자 가구를 서버에 저장하기

## 가장 중요한 한 문장

“게스트 모드는 체험을 위한 로컬 사용이고, 로그인은 저장과 동기화를 위한 기능입니다.”

## 말하면 안 되는 표현

“게스트 프로젝트가 로그인하면 자동으로 계정에 합쳐집니다.”

현재는 자동으로 합쳐지지 않습니다.

## 질문을 받으면

### 게스트도 스캔 결과를 가져갈 수 있나요?

“네. 서버에는 올릴 수 없지만, 3D 파일과 정보 파일을 휴대폰의 공유 기능으로 내보낼 수 있습니다.”

## 다음 페이지 연결

“다음은 크기가 큰 3D 파일이 휴대폰에 계속 쌓이지 않도록 관리하는 방법입니다.”

# 6페이지 — 저장 공간과 캐시 관리

## 발표 원고

“3D 파일은 일반 사진보다 용량이 큰 경우가 많습니다.

여러 방을 열면 방 모형 파일이 쌓이고, 사진으로 3D 가구를 만드는 과정에서도 중간 파일이 생깁니다. 이런 파일을 계속 남겨두면 휴대폰 저장 공간이 불필요하게 줄어듭니다.

그래서 앱은 중요한 파일과 다시 만들 수 있는 임시 파일을 처음부터 나눠서 보관합니다.

서버에서 다시 받을 수 있는 방 파일과, 3D 가구를 만드는 동안 생긴 중간 파일은 캐시라는 임시 보관함에 넣습니다.

반대로 사용자가 저장을 완료한 가구와 편집 복구 파일은 중요한 데이터이므로 다른 안전한 보관 장소에 저장합니다.

설정 화면에서는 현재 캐시가 얼마나 저장되어 있는지 보여줍니다. 사용자가 캐시 비우기를 누르면 다시 받을 수 있는 임시 파일만 한 번에 삭제합니다.

캐시를 비워도 서버에 저장된 방이나 사용자가 완성한 가구가 삭제되는 것은 아닙니다. 나중에 같은 방을 열면 필요한 파일을 서버에서 다시 내려받습니다.

또 화면을 빠르게 보여주기 위해 최근 사용한 3D 가구 일부를 메모리에 보관합니다. 하지만 너무 많이 쌓이지 않도록 개수와 크기에 제한을 두고, 휴대폰의 메모리가 부족하다는 신호가 오면 이 임시 정보도 비웁니다.”

## 가장 쉬운 비유

- 캐시: 다시 받을 수 있는 택배 상자
- 저장 완료된 사용자 가구: 사용자가 직접 만든 원본 작품
- 편집 복구본: 작업 중인 문서

택배 상자는 버려도 다시 받을 수 있지만, 원본 작품과 작업 문서는 버리면 안 됩니다.

## 캐시 비우기로 삭제되는 것

- 다시 다운로드할 수 있는 방의 3D 파일
- 방을 빠르게 열기 위해 받아둔 정보 파일
- 3D 가구 생성 중 만들어진 중간 파일
- 보정 중 만들어진 임시 파일

## 삭제되지 않는 것

- 서버에 저장된 프로젝트와 방
- 사용자가 저장을 완료한 3D 가구
- 편집 복구 파일
- 프로젝트 목록

## 질문을 받으면

### 캐시를 비우면 앱이 고장 나나요?

“아닙니다. 다음에 필요한 화면을 열 때 파일을 다시 받기 때문에 처음 로딩이 조금 길어질 수 있을 뿐입니다.”

### 왜 앱이 자동으로 전부 삭제하지 않나요?

“사용자가 곧 다시 열 방까지 계속 삭제하면 다운로드가 반복될 수 있습니다. 메모리는 자동으로 제한하고, 디스크 파일은 설정에서 사용자가 확인한 뒤 한 번에 지우도록 했습니다.”

## 다음 페이지 연결

“마지막으로 iPhone의 카메라와 사진 보관함을 이용해 사진 한 장을 3D 가구로 만드는 과정을 설명드리겠습니다.”

# 7페이지 — 사진 한 장으로 3D 가구 만들기

## 발표 원고

“Spatium은 방만 스캔하는 것이 아니라, 사용자가 원하는 가구도 사진 한 장으로 3D 모델로 만들 수 있습니다.

사용자는 iPhone 카메라로 가구를 바로 촬영하거나 사진 보관함에서 기존 사진을 선택할 수 있습니다.

iPhone 사진은 HEIC라는 형식으로 저장되는 경우가 많습니다. 그런데 인공지능 서버가 이 형식을 항상 안정적으로 읽는 것은 아닙니다. 그래서 앱이 먼저 서버가 읽기 쉬운 PNG 형식으로 바꿉니다.

또 사진이 너무 크면 긴 쪽을 최대 2,048픽셀로 줄이고, 파일 크기도 최대 10메가바이트 정도로 제한합니다. 사용자는 차이를 거의 느끼지 못하면서 업로드 시간과 메모리 사용을 줄일 수 있습니다.

사진을 선택한 뒤에는 여섯 단계를 거칩니다.

첫째, 가구 사진을 선택합니다.

둘째, 사진 속에서 어떤 물체를 찾을지 이름을 입력합니다. 예를 들어 ‘회색 사무용 의자’라고 입력할 수 있습니다.

셋째, 인공지능이 사진에서 해당 의자만 찾아 배경을 제거합니다.

넷째, 배경이 제거된 사진을 이용해 입체적인 3D 가구를 만듭니다.

다섯째, 사용자가 완성된 가구를 손가락으로 옮기고 돌리고 크기를 조절합니다. 앱이 기본 자세를 자동으로 맞춰주지만, 결과가 완벽하지 않을 수 있어 사용자가 직접 보정할 수도 있습니다.

여섯째, 이름과 종류를 정해 내 가구로 저장합니다. 저장된 가구는 방 편집 화면에서 다른 가구처럼 바로 배치할 수 있습니다.

가구를 선택하거나 이동을 시작하고 끝낼 때, 생성이 완료되거나 저장에 성공했을 때는 짧은 진동을 줍니다. 화면만 보는 것이 아니라 손으로도 조작 결과를 느낄 수 있게 한 것입니다.”

## 여섯 단계 요약

사진 선택  
→ 물체 이름 입력  
→ 배경 제거  
→ 3D 가구 생성  
→ 위치·방향·크기 보정  
→ 내 가구로 저장

## AI 과정을 쉽게 설명하는 비유

### 배경 제거

“사진에서 가구만 가위로 오려내는 과정입니다.”

### 3D 생성

“오려낸 가구 사진을 보고 앞·뒤·옆모습이 있는 입체 모형을 추정하는 과정입니다.”

### 모델 보정

“완성된 모형이 옆으로 누워 있거나 너무 크면 사용자가 바르게 세우고 크기를 맞추는 과정입니다.”

## 사진 처리 설명

- PNG·JPEG가 적당한 크기이면 원본을 그대로 사용
- HEIC처럼 서버가 읽기 어려운 형식은 PNG로 변환
- 너무 큰 사진은 크기와 용량을 줄임
- 화면에는 더 작은 미리보기 이미지를 사용
- 사진 처리는 화면 뒤에서 진행해 앱이 멈춘 것처럼 보이지 않게 함

## 보정 화면에서 가능한 것

- 가구를 여러 방향에서 보기
- 바닥 위에서 위치 옮기기
- 앞뒤·좌우 방향 돌리기
- 전체 크기 키우기·줄이기
- 바닥에 자동으로 붙이기
- 실수한 보정 되돌리기

## 질문을 받으면

### 인공지능이 휴대폰 안에서 실행되나요?

“사진의 크기와 형식을 정리하는 작업은 휴대폰에서 하고, 배경 제거와 3D 생성처럼 계산이 큰 작업은 서버에서 처리합니다.”

### 사진 한 장으로 뒷면까지 정확히 알 수 있나요?

“보이지 않는 부분은 인공지능이 학습한 정보를 이용해 추정하기 때문에 항상 완벽하지는 않습니다. 그래서 사용자가 결과의 방향과 크기를 직접 보정할 수 있게 했습니다.”

### 보정한 결과는 화면에서만 바뀌나요?

“아닙니다. 저장할 때 보정 결과를 실제 3D 파일에 반영하기 때문에 다음에 다시 열어도 같은 상태가 유지됩니다.”

# 전체 마무리

## 발표 원고

“정리하면 Spatium iOS 앱은 휴대폰의 장점을 세 가지 방식으로 활용했습니다.

첫 번째는 카메라와 거리 센서를 이용해 실제 방을 편집 가능한 3D 공간으로 바꾼 것입니다.

두 번째는 자동 임시 저장과 실행 취소 기능으로 긴 편집 작업이 사라지지 않도록 한 것입니다.

세 번째는 카메라와 사진 보관함을 인공지능 서버와 연결해 사용자가 원하는 가구를 직접 3D로 만들 수 있게 한 것입니다.

이를 통해 사용자는 방을 스캔하는 것부터 가구를 배치하고, 자신만의 가구를 만들고, 결과를 저장하는 과정까지 하나의 앱에서 이어서 할 수 있습니다.

이상으로 iOS 앱 부분 설명을 마치겠습니다.”

# 발표에서 기술 용어를 바꿔 말하는 방법

| 어려운 표현 | 발표에서 쓸 쉬운 표현 |
|---|---|
| SwiftUI와 UIKit 브리지 | 앱 화면과 스캔 기능 사이의 통역기 |
| UIViewRepresentable | 서로 다른 두 화면을 연결하는 부분 |
| delegate callback | 스캔 기능이 앱에 결과가 왔다고 알려주는 신호 |
| didUpdate | 방 정보가 처음 들어왔다는 신호 |
| CapturedRoom | 스캔을 마친 최종 방 데이터 |
| USDZ | 눈으로 보는 방의 3D 모형 |
| JSON metadata | 물체의 이름·크기·위치를 적은 설명서 |
| RoomLayout | 현재 편집 중인 방의 전체 상태 |
| debounce | 손동작이 잠깐 멈춘 뒤 마지막 결과만 저장하는 방식 |
| atomic write | 파일이 중간에 깨지지 않게 한 번에 안전하게 교체 |
| Application Support | 지우면 안 되는 사용자 데이터를 보관하는 장소 |
| cache | 다시 받을 수 있는 임시 파일 보관함 |
| GLB | 사용자가 만든 3D 가구 파일 |
| multipart 업로드 | 3D 파일과 설명서를 한 번에 서버로 보내는 방식 |
| GroundingDINO·SAM2 | 사진에서 원하는 가구만 찾아 오려내는 AI |
| TripoSR·Stable Fast 3D | 사진을 입체 가구로 만드는 AI |
| haptic feedback | 버튼이나 조작 결과를 알려주는 짧은 진동 |
