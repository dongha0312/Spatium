# Spatium iOS 발표 스크립트 상세본

기준 자료: 「파이널 프로젝트 ppt_다영본」 7장  
기준 구현: 현재 spatium-ios 작업 폴더의 실제 코드  
권장 발표 시간: 약 8~10분

---

## 먼저 고쳐 말해야 하는 부분

슬라이드 문구를 그대로 읽으면 현재 앱과 다른 부분이 있어, 발표에서는 아래처럼 말하는 것이 정확합니다.

1. 3페이지 리뷰 화면
   - 슬라이드: “항목 수정·삭제 후 편집기 진입”
   - 실제 앱: 리뷰 화면에서는 방 이름, 감지 항목, 치수를 확인합니다. 실제 위치·회전·크기 변경, 교체, 추가, 삭제는 3D 편집기에서 합니다.
   - 권장 문구: “인식 요소와 치수를 확인하고 방 이름을 입력한 뒤, 3D 편집기에서 항목을 수정·삭제합니다.”

2. 4페이지 작업 복구
   - 슬라이드: “앱을 다시 열면 마지막 작업 복구”
   - 실제 앱: 앱을 켜자마자 자동으로 편집 화면까지 이동하는 방식은 아닙니다. 같은 방의 3D 편집기를 다시 열면 저장된 복구본을 감지해 ‘복구’ 또는 ‘폐기’를 선택하게 합니다.
   - 권장 문구: “같은 방 편집기를 다시 열면 마지막 임시 저장본을 복구할 수 있습니다.”

3. 4~5페이지 게스트 프로젝트
   - 현재 게스트 프로젝트와 방 목록은 기기 파일에 기록됩니다. 다만 앱을 완전히 종료한 뒤 게스트로 다시 들어왔을 때 그 목록을 자동으로 다시 불러오는 초기 경로는 현재 구현상 보완이 필요합니다.
   - 따라서 발표에서는 “게스트 작업은 현재 세션에서 로컬로 관리되고 서버에는 올라가지 않는다”까지 말하는 것이 안전합니다.
   - “앱을 재실행해도 게스트 프로젝트가 항상 그대로 복구된다”는 표현은 현재 버전에서는 피하는 것이 좋습니다.

4. 7페이지 HEIC 변환
   - 실제 구현은 HEIC만 처리하는 것이 아닙니다.
   - PNG와 JPEG는 조건을 만족하면 원본 바이트를 유지하고, HEIC 같은 비호환 형식은 PNG로 바꿉니다.
   - 해상도나 용량이 너무 크면 긴 변 최대 2,048픽셀, 파일 최대 10MiB 조건에 맞춰 축소합니다.
   - 카메라로 바로 촬영한 사진은 PNG로 준비합니다.

5. 2~3페이지 스캔 완료 조건
   - 슬라이드의 “최초 유효 인식 결과”보다 실제 구현은 조금 단순합니다.
   - 현재 코드는 벽이나 가구 개수를 별도로 검사하지 않고, RoomPlan의 첫 didUpdate 콜백이 오면 완료 가능 상태로 바꿉니다.
   - 권장 문구: “RoomPlan의 첫 인식 업데이트를 받은 뒤에만 스캔 완료를 허용합니다.”

# 1페이지 — iOS 앱 파트 도입

## 그대로 읽을 수 있는 발표 원고

“이제 제가 담당한 Spatium의 iOS 앱 부분을 설명드리겠습니다.

Spatium iOS 앱의 핵심 역할은 현실의 방을 단순한 사진이 아니라, 실제 크기와 위치 정보를 가진 편집 가능한 3D 공간으로 바꾸는 것입니다.

이를 위해 iPhone과 iPad의 카메라, LiDAR 센서, 그리고 Apple의 RoomPlan 프레임워크를 사용했습니다. 사용자가 방을 한 바퀴 스캔하면 벽과 문, 창문, 가구가 인식되고, 그 결과를 3D 메시와 구조화된 데이터로 저장합니다.

이후 사용자는 인식된 방을 3D 편집기에서 열어 가구를 이동하거나 회전하고, 크기를 조절하거나 다른 가구로 교체할 수 있습니다.

지금부터 공간을 캡처하는 과정, 로컬 저장과 복구, 게스트 모드, 캐시 관리, 그리고 카메라와 사진 보관함을 이용한 3D 가구 생성 순서로 설명드리겠습니다.”

## 이 페이지에서 강조할 한 문장

“핵심은 현실 공간을 보기만 하는 3D 모델이 아니라, 다시 편집할 수 있는 데이터로 바꾸는 것입니다.”

## 다음 페이지 연결 멘트

“먼저 카메라와 LiDAR로 방을 읽어 오는 iOS 공간 캡처 계층부터 보겠습니다.”

# 2페이지 — iOS 공간 캡처 계층

## 그대로 읽을 수 있는 발표 원고

“Spatium의 화면은 SwiftUI로 만들었지만, Apple이 제공하는 RoomCaptureView는 UIKit 기반입니다.

그래서 UIViewRepresentable이라는 연결 계층을 만들어 UIKit의 RoomCaptureView를 SwiftUI 화면 안에 넣었습니다. 쉽게 말하면 SwiftUI 화면과 RoomPlan 스캐너 사이에 어댑터를 하나 둔 것입니다.

스캔이 시작되면 RoomCaptureSession.Configuration을 만들고 세션을 실행합니다. 그다음 카메라 영상과 LiDAR 깊이 정보, 기기의 움직임 정보가 RoomPlan으로 전달됩니다. RoomPlan은 이 정보를 결합해 벽, 문, 창문, 개구부, 가구를 인식하고 화면에 실시간으로 표시합니다.

각 요소에는 단순한 이름만 붙는 것이 아니라, 미터 단위의 가로·세로·높이와 3차원 위치, 회전 방향이 포함됩니다. 앱은 이 정보를 나중에 편집 가능한 객체로 변환합니다.

스캔 완료 버튼은 화면을 열자마자 바로 누를 수 없게 했습니다. RoomPlan에서 첫 번째 didUpdate 콜백을 받은 뒤에만 완료 버튼이 활성화됩니다. RoomPlan이 아직 시작 단계에 있을 때 세션을 종료하면 내부 처리 순서와 충돌할 수 있기 때문입니다.

그리고 사용자가 화면을 닫거나 스캔을 취소하면 RoomCaptureSession을 중지하고 ARSession도 pause 처리합니다. 이렇게 해야 카메라와 LiDAR가 뒤에서 계속 작동하지 않고, AR 프레임이 메모리에 남아 3D 편집기와 겹칠 때 메모리가 급증하는 문제도 막을 수 있습니다.”

## 실제 내부 동작

### 1. SwiftUI와 RoomPlan 연결

- RoomCaptureViewRepresentable이 SwiftUI와 UIKit 사이의 브리지입니다.
- makeUIView에서 RoomCaptureView를 생성합니다.
- Coordinator를 RoomCaptureViewDelegate와 RoomCaptureSessionDelegate로 연결합니다.
- SwiftUI의 isScanning 값이 바뀌면 updateUIView가 세션 시작 또는 중지를 요청합니다.

### 2. 세션 상태 관리

앱은 아래 다섯 상태를 구분합니다.

idle → starting → running → ready → stopping → idle

- idle: 아직 세션이 시작되지 않은 상태
- starting: 실행을 요청한 직후
- running: RoomPlan이 세션 시작을 알린 상태
- ready: didUpdate를 최소 한 번 받아 정상 결과가 존재하는 상태
- stopping: 사용자가 완료를 눌러 최종 처리를 기다리는 상태

완료 버튼은 ready 상태에서만 동작합니다.

### 3. 실시간 업데이트의 정확한 의미

- RoomPlan의 기본 화면은 인식 중인 벽과 가구를 실시간으로 그립니다.
- 앱이 받는 didUpdate 콜백은 현재 구현에서 전체 방 데이터를 별도 상태에 계속 복사하는 용도보다는, “이제 안전하게 완료할 수 있다”는 준비 상태를 갱신하는 데 사용됩니다.
- 최종 데이터는 didPresent processedResult에서 CapturedRoom 형태로 받습니다.

### 4. 안전한 종료

- 사용자가 완료를 누르면 isScanning이 false가 됩니다.
- SwiftUI 업데이트가 Coordinator의 stopSessionIfNeeded를 호출합니다.
- 화면 자체가 해제될 때는 dismantleUIView가 한 번 더 정리를 보장합니다.
- RoomCaptureSession.stop과 ARSession.pause를 함께 사용해 센서와 카메라 리소스를 해제합니다.

## 예상 질문

### 왜 SwiftUI에서 RoomPlan을 바로 쓰지 않았나요?

“RoomCaptureView가 UIKit 뷰로 제공되기 때문에 UIViewRepresentable로 감싸야 SwiftUI 화면 안에서 사용할 수 있습니다.”

### LiDAR가 없는 기기에서는 어떻게 되나요?

“RoomCaptureSession.isSupported를 먼저 확인합니다. 지원하지 않는 기기에서는 스캔 대신 안내 화면을 보여주고, 프로젝트나 가구 만들기 같은 다른 기능은 계속 사용할 수 있습니다.”

### 실제 치수는 어떻게 얻나요?

“앱이 카메라 영상에서 직접 자로 재는 것이 아니라, RoomPlan이 LiDAR 깊이 정보와 AR 추적 정보를 결합해 미터 단위 dimensions와 4×4 transform을 제공합니다.”

“이제 이렇게 시작된 세션이 최종적으로 USDZ와 JSON, 그리고 편집 화면으로 어떻게 이어지는지 순서대로 설명드리겠습니다.”

# 3페이지 — 캡처 처리 흐름

## 그대로 읽을 수 있는 발표 원고

“캡처 처리는 크게 네 단계로 나뉩니다.

첫 번째는 세션 시작입니다. 스캔 준비 화면에서 한 번에 방 하나만 스캔하도록 안내하고, 지원 기기인지 확인한 뒤 RoomCaptureSession.Configuration으로 스캔을 시작합니다.

두 번째는 실시간 공간 인식입니다. RoomPlan이 didUpdate 콜백을 통해 인식 결과를 갱신하고, 기본 캡처 화면에는 벽과 가구가 점차 3D 형태로 나타납니다. 앱은 첫 인식 업데이트 콜백이 도착한 이후에만 스캔 완료를 허용합니다.

세 번째는 최종 결과 정제입니다. 사용자가 완료를 누르면 세션이 멈추고, RoomPlan이 촬영 중 누적한 정보를 처리합니다. 이 과정이 끝나면 최종 CapturedRoom이 전달됩니다.

네 번째는 편집 데이터 변환입니다. CapturedRoom 안의 가구, 문, 창문, 개구부를 앱의 EditableScanItem으로 바꿉니다. 이 객체에는 종류, 표시 이름, 가로·세로·높이, X·Y·Z 위치, Y축 회전값이 들어갑니다.

결과는 두 종류로 나눠 사용합니다. USDZ는 방의 실제 3D 메시이기 때문에 편집기에서 벽과 바닥을 렌더링하는 배경이 됩니다. JSON은 각 요소의 의미와 수치를 담기 때문에 가구를 개별적으로 선택하고 편집하거나 서버와 웹에서 다시 복원하는 데 사용합니다.

스캔이 끝나면 리뷰 화면에서 방 이름과 감지된 항목 수, 각 항목의 치수를 확인합니다. 그다음 3D 편집기로 들어가 실제 위치·회전·크기를 수정하거나 가구를 교체·추가·삭제합니다.

파일 공유를 선택하면 USDZ와 JSON을 함께 내보내고, 로그인 사용자가 서버 업로드를 선택하면 두 파일을 하나의 multipart 요청으로 프로젝트에 저장합니다.”

## 전체 흐름

스캔 준비  
→ 지원 기기 확인  
→ RoomCaptureSession 실행  
→ 첫 didUpdate 콜백 확인  
→ 사용자가 완료 선택  
→ RoomPlan 최종 처리  
→ CapturedRoom 수신  
→ EditableScanItem 변환  
→ 리뷰 화면  
→ USDZ 내보내기  
→ 3D 편집 또는 파일 공유·서버 업로드

## 단계별 세부 동작

### 1단계: 스캔 준비

- “한 방 = 스캔 1회” 안내를 먼저 보여줍니다.
- 여러 방을 한 세션에 담으면 벽 경계와 카테고리 인식이 흔들릴 수 있으므로, 현재 방 안에서만 이동하게 안내합니다.
- 밝은 조명, 천천히 이동, 벽과 가구를 충분히 비추는 방식이 정확도에 유리합니다.

### 2단계: 실시간 인식

- 카메라는 색상과 외형 정보를 제공합니다.
- LiDAR는 표면까지의 깊이와 거리 정보를 제공합니다.
- ARKit 추적은 기기의 위치와 방향 변화를 제공합니다.
- RoomPlan이 이 정보를 묶어 벽, 문, 창문, 개구부와 가구 카테고리를 추정합니다.
- 앱은 첫 didUpdate 이후 완료 버튼을 “스캔 준비 중”에서 “스캔 완료”로 바꿉니다.

### 3단계: 최종 결과

- 완료를 누른 즉시 CapturedRoom이 생기는 것이 아닙니다.
- 먼저 캡처 세션이 끝나고 CapturedRoomData가 만들어집니다.
- RoomCaptureView가 이 데이터를 후처리한 뒤 didPresent processedResult로 최종 CapturedRoom을 전달합니다.
- 처리 중에는 “스캔 데이터 처리 중” 진행 표시를 보여줍니다.

### 4단계: 편집 데이터 변환

앱은 다음 요소를 별도 항목으로 만듭니다.

- room.objects → 가구
- room.doors → 문
- room.windows → 창문
- room.openings → 개구부

각 항목에는 다음 값이 저장됩니다.

- detectedCategory: RoomPlan이 추정한 카테고리
- width, height, depth: 실제 치수
- positionX, positionY, positionZ: 3D 공간 위치
- detectedRotationY: 스캔 당시 Y축 방향
- modelName: 교체된 3D 모델 이름
- decorations: 책장 위에 배치한 소품 정보

### USDZ와 JSON을 둘 다 쓰는 이유

USDZ:

- 스캔한 방의 벽, 바닥, 구조를 눈에 보이게 렌더링합니다.
- RoomPlan의 mesh 옵션으로 내보냅니다.
- 내보내기는 시간이 걸릴 수 있어 백그라운드 작업으로 실행합니다.

JSON:

- 방 이름
- 가구·문·창문·개구부의 분류
- 치수와 4×4 변환 행렬
- 편집 후 확정된 객체 목록
- 벽과 바닥 색상
- 책장 위 소품 같은 앱 확장 정보

즉, USDZ는 “보이는 공간”, JSON은 “이해하고 편집할 수 있는 의미”를 담당합니다.

### 리뷰 화면에서 실제로 가능한 것

- 방 이름 입력
- 감지 항목 수 확인
- 항목별 이름과 치수 확인
- USDZ + JSON 파일 공유
- 서버 업로드
- 3D 편집기 진입

현재 리뷰 화면 자체에서는 항목의 위치·회전·크기 변경이나 삭제를 하지 않습니다. 그 작업은 3D 편집기에서 합니다.

### 스캔 직후 프로젝트에 보이는 방

- 스캔이 끝나면 먼저 local-로 시작하는 로컬 방 기록을 만들어 목록에 즉시 표시합니다.
- 서버 업로드가 끝나면 로컬 기록을 서버가 발급한 실제 방 ID로 교체합니다.
- 같은 스캔을 편집기와 업로드 버튼에서 중복 저장하지 않도록, 이미 서버 ID가 생겼는지도 검사합니다.

## 예상 질문

### JSON만으로는 안 되나요?

### 내보내는 동안 앱이 멈추지 않나요?

“RoomPlan의 mesh export는 몇 초가 걸릴 수 있어 detached task에서 실행합니다. UI는 진행 상태를 보여주고 메인 화면은 멈추지 않게 했습니다.”

## 다음 페이지 연결 멘트

# 4페이지 — 로컬 자동 저장과 작업 복구

## 그대로 읽을 수 있는 발표 원고

“3D 편집은 한 번의 조작으로 끝나는 작업이 아니기 때문에, 앱이 종료되거나 서버 저장이 실패해도 사용자의 편집 내용이 사라지지 않도록 로컬 임시 저장 기능을 구현했습니다.

가구를 추가·삭제하거나 위치, 회전, 크기, 벽과 바닥 색상을 바꾸면 현재 RoomLayout이 변경됩니다. 앱은 변경이 생길 때마다 바로 파일을 계속 쓰는 대신 650밀리초 동안 추가 변경을 기다린 뒤 마지막 상태를 JSON 복구본으로 저장합니다. 사용자가 슬라이더를 움직이는 동안 수십 번 저장하는 일을 줄이기 위한 방식입니다.

복구본은 캐시가 아니라 Application Support 영역에 저장합니다. 캐시 비우기나 운영체제의 캐시 정리로 중요한 편집 복구본이 사라지지 않게 하기 위해서입니다.

같은 방 편집기를 다시 열면 저장된 복구본과 마지막 정상 저장 상태를 비교합니다. 차이가 있으면 사용자에게 임시 저장본을 복구할지, 폐기할지 묻습니다. 복구를 선택하면 가구 배치와 색상, 소품 상태가 다시 적용됩니다.

서버 저장에 성공하면 그 상태를 정상 저장 기준으로 갱신하고 복구본을 삭제합니다. 반대로 서버 저장이 실패하면 로컬 복구본은 남아 있기 때문에 사용자가 다시 시도할 수 있습니다.

또한 편집기에는 최대 30단계의 Undo와 Redo가 있습니다. 위치 이동이나 슬라이더 조작처럼 값이 연속으로 바뀌는 동작은 시작 상태 하나를 기록해, 한 번의 사용자 동작이 한 번의 실행 취소로 돌아가도록 했습니다.”

## 자동 임시 저장의 실제 흐름

사용자 편집  
→ 이전 상태를 Undo 스택에 기록  
→ RoomLayout 변경  
→ 미저장 상태 표시  
→ 기존 예약 저장 취소  
→ 650ms 대기  
→ 최신 RoomLayout을 EditorDraft JSON으로 인코딩  
→ Application Support/RoomEditorDrafts에 atomic write  
→ 하단에 저장 완료 시각 표시

### 왜 650ms를 기다리나요?

- 드래그나 슬라이더는 짧은 시간에 상태가 많이 바뀝니다.
- 매 변화마다 디스크에 쓰면 I/O가 불필요하게 늘어납니다.
- 마지막 조작 후 650ms가 지나면 가장 최신 상태만 기록합니다.

### 오래된 저장이 최신 상태를 덮지 않게 하는 방법

- 각 쓰기와 삭제에 증가하는 revision 번호를 부여합니다.
- 같은 파일에 더 최신 revision이 이미 적용됐다면 늦게 도착한 이전 작업은 무시합니다.
- 파일은 atomic 옵션으로 기록해 중간 상태의 깨진 JSON이 남을 가능성을 줄입니다.

### 화면을 닫을 때

- 예약된 650ms 저장을 기다리지 않고 즉시 저장을 시도합니다.
- 저장 공간이나 권한 문제로 로컬 저장이 실패하면 바로 화면을 닫지 않습니다.
- “다시 시도”, “저장하지 않고 나가기”, “계속 편집” 중 하나를 선택하게 합니다.

### 복구할 때

- 프로젝트 ID, 방 ID, 방 이름을 조합해 같은 방의 복구 파일을 찾습니다.
- 실행할 때마다 달라지는 Swift 기본 hash가 아니라 고정 FNV-1a 해시를 사용해 앱을 재실행해도 동일한 파일 이름을 계산합니다.
- 복구본과 마지막 저장 레이아웃이 같으면 불필요한 복구본으로 판단해 삭제합니다.
- 다르면 복구 다이얼로그를 보여줍니다.

### Undo·Redo

- RoomLayout뿐 아니라 선택한 가구, 꾸미기 중인 책장, 선택한 소품, 이동 모드 상태도 스냅샷에 포함합니다.
- Undo와 Redo는 각각 최대 30단계입니다.
- 새 편집이 발생하면 Redo 스택은 비웁니다.
- Undo·Redo 스택 자체는 앱 재실행 후까지 저장되지는 않습니다. 재실행 복구는 마지막 임시 저장 상태를 복구하는 기능입니다.

### 서버 저장 실패 시

- 서버 저장이 성공해야 savedLayout이 현재 상태로 갱신되고 임시 파일이 삭제됩니다.
- 실패하면 미저장 상태와 로컬 복구본이 유지됩니다.
- 따라서 네트워크 실패와 로컬 저장 실패는 별개로 처리됩니다.

### 프로젝트 목록 로컬 저장

- 프로젝트 목록은 Documents/projects.json에 저장합니다.
- 게스트 프로젝트와 업로드 전 로컬 방에는 local- 접두사가 붙습니다.
- 프로젝트 저장도 revision과 atomic write를 사용합니다.
- 저장 실패 시 최신 메모리 상태로 다시 저장할 수 있는 재시도 버튼을 제공합니다.

### 현재 게스트 재실행 복구의 주의점

- 게스트 프로젝트 데이터는 파일에 기록됩니다.
- 하지만 현재 초기 로딩 로직은 로그인 상태에서만 프로젝트 캐시를 불러오도록 되어 있습니다.
- 그러므로 “게스트 프로젝트가 앱 완전 종료 후에도 자동 복구된다”는 주장은 현재 코드와 정확히 일치하지 않습니다.
- 발표에서는 편집 복구본과 프로젝트 캐시를 분리해 설명하고, 게스트 재실행 복구는 후속 개선 항목으로 보는 것이 안전합니다.

## 예상 질문

### 캐시를 비우면 임시 저장도 삭제되나요?

“아닙니다. 다운로드 캐시는 Caches에 있지만, 에디터 복구본과 저장 완료된 사용자 가구는 Application Support에 따로 보관합니다.”

### 자동 저장과 서버 저장은 같은 건가요?

“아닙니다. 자동 저장은 작업 유실 방지를 위한 기기 내부 복구본이고, 서버 저장은 다른 기기와 웹에서 다시 보기 위한 확정 저장입니다.”

# 5페이지 — 로그인 없이 둘러보는 게스트 모드

## 그대로 읽을 수 있는 발표 원고

“Spatium은 사용자가 회원가입부터 강제당하지 않고 핵심 기능을 먼저 경험할 수 있도록 게스트 모드를 제공합니다.

첫 실행 온보딩이 끝난 뒤 로그인 화면에서 ‘로그인 없이 둘러보기’를 선택하면 메인 화면으로 들어갑니다. 이때 인증 토큰은 발급되지 않고, 게스트 세션 상태만 앱 안에서 유지됩니다.

게스트도 로컬 프로젝트를 만들고, LiDAR로 방을 스캔하고, 감지 결과를 검토하고, 3D 편집기에서 기본 가구를 배치하거나 이동할 수 있습니다. USDZ와 JSON 파일을 기기의 공유 시트로 내보내는 것도 가능합니다.

대신 서버가 필요한 기능은 제한됩니다. 스캔 결과를 서버 프로젝트에 업로드하거나, AI로 사진을 3D 가구로 만들거나, 여러 기기에서 같은 프로젝트를 동기화하는 기능은 로그인이 필요합니다.

로그인 사용자는 프로젝트와 방을 서버 API로 저장합니다. 따라서 같은 계정으로 로그인하면 다른 기기나 웹에서도 서버에 저장된 결과를 다시 확인할 수 있습니다.

게스트가 제한 기능을 누르면 단순히 실패시키지 않고, 왜 로그인이 필요한지 안내한 뒤 로그인 화면으로 연결합니다. 또한 게스트 프로젝트는 로그인 계정으로 자동 이전되지 않기 때문에 로그인 전에 유실 가능성을 경고합니다.”

## 게스트와 로그인 사용자의 실제 차이

### 게스트

- 인증 토큰 없음
- local- ID의 프로젝트 생성
- RoomPlan 공간 스캔 가능
- 리뷰 화면 확인 가능
- USDZ + JSON 파일 공유 가능
- 기본 가구를 이용한 로컬 3D 편집 가능
- 편집 복구본 생성 가능
- 서버 업로드 불가
- AI 배경 제거·3D 생성 화면 진입 제한
- 다른 기기·웹과 동기화 불가

### 로그인 사용자

- 이메일, Apple, Google 로그인으로 인증 토큰 보유
- 프로젝트와 방을 서버 API로 생성
- 스캔 USDZ + JSON 업로드 가능
- 3D 편집 결과를 서버 metadata로 저장
- 사진 배경 제거와 3D 생성 API 사용 가능
- 사용자 가구 서버 저장·동기화 가능
- 같은 계정으로 다른 기기와 웹에서 확인 가능

## 앱 내부 분기

### 앱 시작

첫 실행 온보딩  
→ 로그인 여부 확인  
→ 로그인 상태이면 MainTabView  
→ 아니면 로그인 또는 게스트 선택

### 프로젝트 생성

- 로그인 상태: 서버의 프로젝트 생성 API 호출
- 게스트 상태: 서버 호출 없이 local- ID를 생성하고 기기 파일에 기록

### 스캔 업로드

- 파일 공유는 게스트도 가능
- 서버 업로드는 access token을 확인
- 토큰이 없으면 “게스트 모드에서는 업로드할 수 없다”는 안내와 로그인 버튼 표시

### 가구 만들기

- 배경 제거와 3D 생성은 서버 계산 자원이 필요합니다.
- 메인 탭에서 로그인 여부를 확인해 게스트에게는 기능 제한 카드와 로그인 버튼을 보여줍니다.

### 게스트 프로젝트의 계정 이전

- 현재 자동 마이그레이션은 없습니다.
- 로그인 화면이 Documents/projects.json 안의 local- 프로젝트 수를 확인합니다.
- 남아 있는 로컬 프로젝트가 있으면 “로그인하면 더 이상 볼 수 있다”가 아니라 “로그인하면 더 이상 볼 수 없다”고 경고합니다.
- 발표 중 “게스트 작업이 로그인하면 자동 동기화된다”고 말하면 안 됩니다.

# 6페이지 — 모바일 저장 공간과 캐시 관리

## 그대로 읽을 수 있는 발표 원고

“3D 앱은 일반 이미지 앱보다 로컬 저장 공간 관리가 더 중요합니다. 방 하나를 열 때도 USDZ와 JSON이 내려오고, 여러 방을 열면 파일이 계속 누적됩니다. 사진으로 가구를 만들 때도 생성된 GLB와 보정 중간 파일이 생깁니다.

그래서 Spatium은 다시 받을 수 있는 데이터와 사용자가 만든 중요한 데이터를 저장 위치부터 분리했습니다.

다시 다운로드할 수 있는 방 스캔 파일과 서버 방의 Scene 파일, 가구 생성 중간 모델은 Caches 영역에 둡니다. 설정 화면에서는 이 세 영역의 전체 용량을 계산해 보여주고, 사용자가 ‘캐시 비우기’를 누르면 한 번에 삭제합니다.

반면 사용자가 저장을 완료한 가구 모델과 에디터 복구본은 Application Support에 저장하므로 캐시 비우기의 대상이 아닙니다. 프로젝트 목록 파일도 Documents에 있어 영향을 받지 않습니다.

캐시를 삭제해도 서버 데이터 자체가 지워지는 것은 아닙니다. 다음에 같은 방을 열면 필요한 JSON과 USDZ를 다시 내려받습니다.

또 디스크 캐시와 별도로, 3D 에디터에서 파싱한 가구 모델의 메모리 캐시는 최근 사용 기준으로 최대 8개, 원본 파일 크기 합계 64MiB로 제한합니다. 메모리 경고가 오면 이 캐시를 전부 비워 앱이 강제 종료되는 위험을 낮췄습니다.”

## 실제 관리 대상

### Caches/RoomScans

- 서버나 파일 경로에서 불러온 방의 JSON
- 방의 USDZ
- 방 ID별 디렉터리로 분리
- 이미 내려받은 파일이 있으면 재사용
- 에디터 저장으로 metadata가 바뀌거나 방을 삭제하면 해당 방 캐시 무효화

### Caches/RoomScenes

- 서버의 scene API가 Base64로 내려준 USDZ
- 디코딩과 파일 쓰기를 백그라운드 직렬 큐에서 수행
- 같은 방을 동시에 열어도 파일 교체가 겹치지 않게 처리

### Caches/Spatium

- GeneratedModels: AI가 생성한 GLB 중간 결과
- CorrectedModels: 사용자가 회전·이동·크기를 보정해 저장하기 직전 만든 임시 GLB

## 캐시 비우기에서 보존되는 데이터

- Application Support/Spatium/UserFurniture의 저장 완료된 사용자 가구 GLB
- 사용자 가구 목록 catalog.json
- Application Support/RoomEditorDrafts의 편집 복구본
- Documents/projects.json의 프로젝트 목록 캐시
- 서버에 저장된 프로젝트·방·가구 원본

## 설정 화면 동작

1. 설정 화면 진입
2. Caches 아래 RoomScans, RoomScenes, Spatium을 순회
3. 각 파일 크기를 합산
4. ByteCountFormatter로 KB·MB 단위 표시
5. 캐시 비우기 선택
6. 세 디렉터리를 백그라운드에서 삭제
7. 남은 크기를 다시 계산
8. 성공 햅틱 전달

용량 계산과 삭제를 메인 화면 스레드 밖에서 실행하므로 파일이 많아도 버튼과 스크롤이 멈추지 않습니다.

## 메모리 캐시

- GLB를 매번 파싱하면 느리기 때문에 파싱된 SceneKit 노드 템플릿을 재사용합니다.
- 최근 사용된 모델을 남기는 LRU 방식입니다.
- 최대 8개 또는 원본 파일 크기 합계 64MiB를 넘으면 오래 사용하지 않은 모델부터 제거합니다.
- 단일 모델 하나가 64MiB보다 큰 경우 반복 파싱을 막기 위해 그 모델 하나만 유지할 수 있습니다.
- 시스템 메모리 경고를 받으면 전부 비웁니다.
- 이미 방에 배치된 가구 노드는 캐시 퇴거와 관계없이 화면에 남습니다.

## 예상 질문

### 캐시를 비우면 사용자가 만든 가구도 없어지나요?

“아닙니다. 저장 완료된 사용자 가구는 Application Support에 있고, 설정에서 지우는 대상은 Caches 아래의 재생성·재다운로드 가능한 파일뿐입니다.”

### 캐시는 자동으로도 정리되나요?

“SceneKit의 메모리 캐시는 개수와 크기 기준으로 자동 제한되고 메모리 경고 때 비워집니다. 디스크 캐시는 현재 설정 화면에서 사용자가 직접 전체 삭제하는 방식입니다.”

## 다음 페이지 연결 멘트

“마지막으로 iOS 기기의 카메라와 사진 보관함을 이용해 사진 한 장을 3D 가구로 만드는 과정을 설명드리겠습니다.”

# 7페이지 — 네이티브 카메라와 사진 보관함 연동

## 그대로 읽을 수 있는 발표 원고

“웹에서도 사진 업로드는 가능하지만, iOS 앱에서는 모바일 기기의 카메라와 사진 보관함을 네이티브로 연결했습니다.

카메라는 UIImagePickerController를 SwiftUI에 연결해 즉시 촬영할 수 있게 했고, 사진 보관함은 PhotosPicker를 사용합니다. 사진 권한이 아직 결정되지 않았으면 시스템 권한을 요청하고, 이미 거부된 경우에는 설정 화면으로 이동할 수 있게 안내합니다.

사진을 선택하면 바로 서버로 보내지 않고 먼저 기기에서 업로드용으로 정규화합니다. PNG와 JPEG가 크기 조건을 만족하면 원본 바이트를 유지하고, HEIC처럼 서버의 딥러닝 파이프라인에서 바로 처리하기 어려운 형식은 PNG로 변환합니다. 고해상도 사진은 긴 변을 최대 2,048픽셀로 줄이고, 파일 크기는 최대 10MiB로 제한합니다. 화면 미리보기는 최대 1,024픽셀로 따로 만들어 메모리 사용도 줄였습니다.

가구 생성은 여섯 단계입니다. 사진 선택, 객체 이름 입력, 배경 제거, 3D 생성, 모델 보정, 저장 순서입니다.

배경 제거 단계에서는 사용자가 입력한 ‘회색 사무용 의자’ 같은 설명을 서버로 보내고, GroundingDINO와 SAM2 또는 YOLO가 대상 객체를 찾아 투명 배경 PNG로 반환합니다.

그다음 투명 배경 이미지를 TripoSR 또는 Stable Fast 3D 모델에 전달해 GLB 파일을 생성합니다. 앱은 결과 파일의 앞부분이 실제 glTF 형식인지 확인한 뒤 기기 캐시에 저장합니다.

생성된 모델은 SceneKit 화면에서 직접 보정할 수 있습니다. 사용자는 보기, 이동, 회전, 크기 모드를 선택하고 손가락 드래그로 위치와 각도를 바꿀 수 있습니다. 이동은 0.1미터, 회전은 15도, 크기는 0.05 단위로 스냅되어 조작 결과가 안정적으로 맞춰집니다.

보정이 끝나면 변환값을 GLB 파일에 실제로 반영하고, 사용자 가구 목록에 저장합니다. 서버 저장이 가능하면 서버에도 업로드하고, 서버 경로가 아직 준비되지 않은 특정 경우에는 보정 결과를 잃지 않도록 기기 내부에 저장합니다.

사진 선택, 모드 변경, 조작 시작과 종료, 생성 성공과 실패, 최종 저장 결과에는 햅틱 피드백을 적용해 화면을 보지 않아도 조작 결과를 느낄 수 있게 했습니다.”

## 6단계 전체 흐름

### 1. 사진 업로드

- 사진 보관함 또는 카메라 선택
- 사진 보관함은 PhotosPickerItem에서 원본 Data를 읽음
- 카메라는 UIImagePickerController에서 originalImage를 받음
- 새 선택이 들어오면 이전 이미지 준비 작업을 취소하고 새 request ID를 부여
- 늦게 끝난 이전 작업이 최신 선택을 덮어쓰지 못하게 ID 확인

### 2. 객체명 입력

- 사용자가 사진 속 대상 설명 입력
- 예: “회색 사무용 의자”
- GroundingDINO + SAM2 또는 YOLO 선택
- TripoSR 또는 Stable Fast 3D 선택
- 모델별 메시·텍스처·리메시 옵션 설정

### 3. 배경 제거

- /api/ai/remove-background로 multipart 전송
- GroundingDINO + SAM2 사용 시 object_query 전달
- YOLO 사용 시 자동 대상 클래스 사용
- 서버가 투명 배경 PNG 반환
- 응답 헤더에서 실제 분리 객체명, 영어 변환 검색어, 신뢰도, 사용 모델 확인
- PNG 파일 시그니처를 검사해 잘못된 응답 차단

### 4. 3D 생성

- 분리된 PNG를 /api/ai/image-to-3d로 전송
- TripoSR은 mc_resolution 사용
- Stable Fast 3D는 texture_resolution과 remesh 옵션 사용
- 긴 생성 시간을 고려해 요청 제한 시간을 길게 설정
- 서버의 GLB 응답은 메모리에 통째로 올리지 않고 임시 파일로 스트리밍
- 파일 첫 4바이트가 glTF인지 확인
- 검증된 파일을 Caches/Spatium/GeneratedModels로 이동

진행률은 서버의 실제 세부 퍼센트를 받는 방식이 아니라, 작업 중 92%까지 단계적으로 증가시키고 파일 수신이 끝나면 100%로 완료하는 사용자 안내용 진행률입니다.

### 5. 모델 보정

자동 보정:

- 모델 정점을 샘플링
- 현재 방향을 기준으로 X축과 Z축 회전 후보를 비교
- 먼저 5도 간격으로 넓게 탐색하고, 가장 좋은 후보 주변을 1도 간격으로 다시 탐색
- 축 정렬된 bounding box 부피가 가장 작은 자세를 선택
- 모델의 바닥 발자국 중심을 원점에 맞추고 바닥에 붙임

수동 보정:

- 보기: 빈 공간 드래그로 카메라 회전, 핀치로 확대
- 이동: 화면 방향에 맞춰 X·Z 이동, 축 선택 시 X·Y·Z 개별 이동
- 회전: 자유 회전 또는 X·Y·Z 축 선택
- 크기: 0.5배에서 2배까지 조절
- 바닥 고정, 가운데 정렬, 시점 초기화 제공
- 정면·측면·상단·원근 시점 제공
- 최대 30단계 보정 Undo·Redo 제공

손가락을 뗄 때:

- 위치는 0.1m 단위
- 회전은 15도 단위
- 크기는 0.05 단위로 스냅
- 스냅 완료 햅틱 전달

### 6. 저장

- 현재 이동·회전·크기 보정값을 GLB에 베이크한 임시 파일 생성
- 사용자 입력 이름, 서버가 정규화한 영어 이름, 카테고리, 실제 모델 치수와 함께 저장
- 로그인과 서버 경로가 정상인 경우 사용자 가구 API로 업로드
- 최종 GLB와 catalog.json은 Application Support/Spatium/UserFurniture에 저장
- 저장이 끝나면 사용자 가구 목록과 3D 에디터 카탈로그에서 바로 사용 가능
- 저장에 사용한 CorrectedModels 임시 파일은 완료 후 삭제

## 이미지 정규화 상세

### 사진 보관함

- PNG/JPEG
  - 긴 변 2,048픽셀 이하
  - 10MiB 이하
  - 두 조건을 만족하면 원본 업로드 바이트 유지
- HEIC 등 비호환 형식
  - 방향 정보를 적용해 올바른 방향으로 다운샘플
  - PNG로 인코딩
- 큰 JPEG
  - JPEG 형식을 유지하면서 축소
  - JPEG 품질 0.88 사용

### 카메라

- 촬영 결과를 최대 2,048픽셀 크기의 bitmap context에 직접 그림
- PNG로 인코딩
- 최대 10MiB를 넘으면 픽셀 제한을 반복해서 줄임

### 미리보기

- 업로드용 파일과 별도로 최대 1,024픽셀 이미지를 만듭니다.
- 탭을 이동하거나 앱이 백그라운드로 가면 UIImage와 SceneKit 화면 리소스를 해제합니다.
- 압축 이미지 Data와 생성된 GLB URL은 남겨 두므로 다시 돌아오면 진행 단계가 유지됩니다.

## 예상 질문

### AI 모델이 iPhone 안에서 실행되나요?

“아닙니다. 앱은 사진 전처리와 화면 표시를 담당하고, 배경 제거와 3D 생성은 인증된 Spring API를 통해 서버에서 처리합니다.”

### 왜 HEIC를 PNG로 바꾸나요?

“iPhone 사진의 대표 형식인 HEIC를 딥러닝 파이프라인의 모든 구성 요소가 동일하게 지원하지 않기 때문입니다. 서버 입력을 PNG 또는 JPEG로 정규화해 호환성을 확보합니다.”

### 생성된 모델의 보정값은 화면에서만 적용되나요?

“아닙니다. 저장할 때 이동·회전·크기 변환을 GLB 파일에 베이크하므로 다른 화면이나 에디터에서 다시 열어도 보정 결과가 유지됩니다.”

# 짧은 예상 질문 모음

## RoomPlan과 ARKit의 차이는 무엇인가요?

“ARKit은 기기 위치 추적과 카메라·센서 기반 AR 기능을 제공하고, RoomPlan은 그 위에서 방의 벽·문·창문·가구를 구조화된 CapturedRoom으로 만들어 주는 상위 프레임워크입니다.”

## 왜 방 하나씩 스캔하나요?

“여러 방을 한 번에 연결하면 벽 경계와 가구 위치·분류가 불안정해질 수 있어 방 하나를 하나의 스캔과 룸 데이터로 관리합니다.”

## 서버가 없어도 가능한 기능은 무엇인가요?

“RoomPlan 스캔, 결과 검토, 파일 공유, 기본 가구를 이용한 로컬 편집과 임시 저장은 가능합니다. 서버 업로드, 다른 기기 동기화, AI 가구 생성은 서버와 로그인이 필요합니다.”

## 서버 저장과 자동 임시 저장의 차이는 무엇인가요?

“서버 저장은 계정 기반 확정 저장이고, 임시 저장은 현재 기기에서 작업 유실을 막는 복구 장치입니다.”

## 캐시 삭제 후 방이 사라지나요?

“서버에 저장된 방은 사라지지 않습니다. 로컬 복사본만 삭제되며 다음에 방을 열 때 다시 다운로드합니다.”

## 왜 USDZ와 GLB를 둘 다 쓰나요?

“RoomPlan이 스캔한 방 구조는 USDZ로 다루고, 기본·사용자 가구 카탈로그는 GLB로 다룹니다. SceneKit에서 두 형식을 같은 편집 공간에 렌더링합니다.”

## 현재 구현에서 가장 솔직하게 말해야 할 한계는 무엇인가요?

“게스트 프로젝트는 기기에 기록되지만 완전 종료 후 게스트 초기 로딩 경로가 아직 완성되지 않았고, 게스트 데이터를 로그인 계정으로 자동 이전하는 기능도 없습니다.”
