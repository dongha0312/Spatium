# 1페이지 — iOS 구현 소개

## 발표 대본

“지금부터는 Spatium의 iOS 앱이 실제 공간을 어떻게 3D 데이터로 바꾸는지 설명드리겠습니다.

이 기능을 아주 간단히 말하면, 아이폰으로 방을 천천히 비추면 앱이 그 방의 벽, 문, 창문, 가구를 찾아서 3D 도면으로 만들어 주는 기능입니다.

일반 사진은 한 방향에서 본 평면 이미지이기 때문에 물건의 정확한 거리나 크기를 알기 어렵습니다. 반면 저희 앱은 카메라와 거리 측정 센서를 함께 사용해서, ‘어디에 무엇이 있고 얼마나 큰지’까지 기록합니다.

그리고 스캔이 끝나면 단순히 구경만 할 수 있는 3D 이미지가 아니라, 가구를 옮기거나 돌리고 삭제할 수 있는 편집 가능한 공간 데이터로 변환합니다.

즉, 현실의 방을 휴대폰으로 스캔하고, 그 결과를 앱 안에서 다시 꾸밀 수 있는 디지털 공간으로 만드는 것이 이번 iOS 구현의 핵심입니다.”

## 이 페이지가 의미하는 것

전체 기능은 다음 순서로 움직입니다.

1. 사용자가 아이폰으로 방을 비춥니다.
2. 카메라는 물체의 모습을 보고, LiDAR 센서는 물체까지의 거리를 측정합니다.
3. 애플의 RoomPlan 기능이 두 정보를 합쳐 벽, 문, 창문, 가구를 구분합니다.
4. 앱이 결과를 3D 모형과 편집용 정보로 저장합니다.
5. 사용자는 결과를 확인한 다음 3D 편집기에서 가구를 수정합니다.

## 쉬운 비유

- 카메라: 방을 보는 ‘눈’
- LiDAR: 물체까지의 거리를 재는 ‘줄자’
- RoomPlan: 눈과 줄자의 정보를 이용해 도면을 그리는 ‘측량사’
- Spatium 편집기: 완성된 도면 위에서 가구를 다시 배치하는 ‘인테리어 작업대’

## 다음 페이지로 넘어가는 말

“그러면 아이폰이 방을 스캔하는 동안 앱 내부에서는 어떤 일들이 일어나는지 조금 더 자세히 보겠습니다.”

# 2페이지 — iOS 공간 캡처 계층

## 발표 대본

“이 페이지는 실제 스캔 화면이 어떻게 작동하는지를 보여 줍니다.

먼저 사용자가 스캔 시작 버튼을 누르면 아이폰의 카메라와 거리 측정 센서가 함께 켜집니다. 카메라는 벽과 가구의 생김새를 보고, LiDAR는 각 물체가 휴대폰에서 얼마나 떨어져 있는지 측정합니다.

그다음 애플의 RoomPlan이라는 기능이 이 정보들을 실시간으로 조합합니다. 예를 들어 평평하고 넓게 이어진 면은 벽으로, 벽 중간의 열린 부분은 문이나 창문으로, 방 안의 독립된 물체는 침대나 책상 같은 가구로 판단합니다.

물체를 찾는 것에서 끝나는 것이 아니라, 각 물체가 방의 어느 위치에 있는지, 어느 방향을 향하고 있는지, 가로·세로·높이가 어느 정도인지도 함께 기록합니다. 이 정보가 있어야 나중에 3D 편집기에서 가구를 자연스럽게 옮기거나 회전시킬 수 있습니다.

여기 있는 ‘UIViewRepresentable’이라는 용어는 두 종류의 화면을 이어 주는 번역기라고 생각하시면 됩니다. 저희 앱의 일반 화면과 애플이 제공하는 스캔 화면은 만들어진 방식이 조금 다릅니다. 그래서 이 연결 장치가 사용자 버튼 입력을 스캔 기능에 전달하고, 반대로 스캔 결과는 저희 앱 화면으로 가져옵니다.

스캔 중 방 정보가 처음 정상적으로 들어오면 앱은 완료 버튼을 누를 수 있게 바꿉니다. 사용자가 완료를 누르면 스캔을 종료하고 최종 결과를 만드는 단계로 넘어갑니다.

또 화면을 닫거나 스캔을 취소했을 때는 카메라와 센서도 함께 종료합니다. 이 정리 과정이 빠지면 사용하지 않는 카메라 작업이 뒤에서 계속 남아 배터리와 메모리를 낭비할 수 있기 때문입니다.”

## 버튼을 누른 뒤 실제 작동 순서

1. **스캔 화면 진입**  
   앱이 RoomPlan용 카메라 화면을 준비합니다.

2. **스캔 시작**  
   카메라 영상과 LiDAR 거리 측정이 시작됩니다.

3. **실시간 인식**  
   사용자가 움직일 때마다 RoomPlan이 공간 정보를 계속 갱신합니다. 화면 위의 벽이나 가구 표시도 함께 바뀝니다.

4. **첫 번째 공간 정보 수신**  
   앱은 “스캔 결과가 한 번 이상 들어왔다”는 신호를 받고 완료 버튼을 활성화합니다.

5. **완료 또는 취소**  
   완료하면 최종 보정 단계로 가고, 취소하면 현재 결과를 버립니다.

6. **센서 정리**  
   스캔 세션을 멈추고 카메라 기반 공간 인식 작업도 종료합니다.

## 화면 용어를 쉽게 풀면

| 화면의 용어 | 쉬운 뜻 |
|---|---|
| SwiftUI | 앱의 버튼과 일반 화면을 만드는 방식 |
| RoomCaptureView | 카메라로 방을 비추며 스캔 결과를 보여 주는 화면 |
| UIViewRepresentable | 일반 앱 화면과 스캔 화면 사이의 번역기 |
| RoomCaptureSession | 스캔 시작부터 종료까지 관리하는 작업 단위 |
| ARSession | 카메라와 센서로 휴대폰의 위치 및 주변 공간을 파악하는 작업 |
| didUpdate | 새로운 공간 정보가 들어왔다는 알림 |

## 정확하게 말해야 하는 부분

완료 버튼은 “방 전체를 완벽하게 인식했다”는 뜻으로 켜지는 것은 아닙니다. 현재 앱에서는 **공간 정보가 처음 들어온 뒤** 완료할 수 있게 됩니다. 따라서 실제 사용자는 화면 안내를 보면서 방 전체를 충분히 비춘 뒤 완료해야 더 좋은 결과를 얻을 수 있습니다.

## 다음 페이지로 넘어가는 말

“이렇게 센서가 모은 정보는 바로 저장되는 것이 아니라, 사용하기 좋은 최종 데이터로 한 번 더 정리됩니다.”

# 3페이지 — 캡처 처리 흐름

## 발표 대본

“공간 캡처는 크게 네 단계로 진행됩니다.

첫 번째는 세션 시작입니다. 사용자가 스캔을 시작하면 앱이 카메라와 센서를 준비하고, 지금부터 들어오는 정보를 하나의 방 스캔 작업으로 묶습니다.

두 번째는 실시간 인식입니다. 사용자가 방 안을 천천히 움직이며 비추는 동안 벽, 문, 창문, 가구 정보가 계속 추가되고 수정됩니다. 처음에는 벽의 일부만 보일 수 있지만, 다른 각도에서 다시 비추면 형태와 크기가 점점 더 정확해집니다.

세 번째는 최종 보정입니다. 사용자가 완료 버튼을 누르면 그 순간 보이는 내용만 그대로 저장하는 것이 아닙니다. RoomPlan이 스캔 과정에서 모은 여러 정보를 다시 정리해서 서로 이어지는 벽을 맞추고, 물체의 위치와 크기를 최종 결과로 확정합니다.

네 번째는 편집 가능한 데이터로 바꾸는 단계입니다. 최종 결과는 두 종류로 나누어 다룹니다.

USDZ는 화면에 실제로 보여 줄 수 있는 3D 모형입니다. 쉽게 말해 완성된 미니어처 방과 같습니다.

JSON은 각 물체의 이름, 크기, 위치, 방향을 글과 숫자로 정리한 설계 정보입니다. 예를 들어 ‘책상은 방의 이 위치에 있고, 가로와 세로는 이 정도이며, 이 방향을 보고 있다’는 내용을 기록합니다.

왜 두 가지가 모두 필요하냐면, 3D 모형만 있으면 방을 눈으로 볼 수는 있지만 책상 하나만 정확히 선택해서 옮기기 어렵기 때문입니다. 반대로 설계 정보만 있으면 편집은 할 수 있지만 사용자에게 사실적인 3D 공간을 보여 주기 어렵습니다.

그래서 앱은 USDZ로 공간을 보여 주고, JSON의 정보를 이용해 물체를 선택하고 이동하거나 회전하고 삭제합니다.

처리가 끝나면 먼저 리뷰 화면에서 방 이름, 인식된 물체의 수, 각 물체의 종류와 크기를 확인합니다. 잘못된 부분이 있으면 다음 3D 편집기에서 가구를 이동·회전·교체하거나 삭제할 수 있습니다.”

## 네 단계의 자세한 흐름

### 1. 세션 시작

- 카메라 사용 권한을 확인합니다.
- RoomPlan 스캔 작업을 시작합니다.
- 앱은 아직 결과가 없으므로 완료 버튼을 비활성화합니다.

### 2. 실시간 인식

- 휴대폰의 움직임과 주변 공간의 깊이를 계속 측정합니다.
- RoomPlan이 벽과 물체의 후보를 실시간으로 화면에 표시합니다.
- 새 공간 정보가 들어오면 앱은 정상적으로 스캔 중임을 확인합니다.

### 3. 최종 보정

- 사용자가 완료를 누르면 추가 촬영을 멈춥니다.
- 애플의 처리 기능이 그동안 모은 측정값을 하나의 방 구조로 정리합니다.
- 정리가 끝난 최종 `CapturedRoom`, 즉 ‘스캔된 방 데이터’를 앱에 전달합니다.

### 4. 리뷰 및 편집 데이터 준비

- 3D로 보여 줄 수 있는 USDZ 파일을 만듭니다.
- 물체별 이름·크기·위치·방향이 담긴 JSON 데이터를 만듭니다.
- 리뷰 화면에서는 스캔 결과가 맞는지 확인합니다.
- 실제 이동·회전·교체·삭제는 3D 편집기에서 수행합니다.

## 두 파일의 차이

| 구분 | 쉬운 비유 | 앱에서 하는 역할 |
|---|---|---|
| USDZ | 완성된 3D 미니어처 | 방의 모습과 입체 구조를 화면에 보여 줌 |
| JSON | 가구 배치가 적힌 설계도 | 각 물체를 따로 선택하고 편집할 수 있게 함 |

## 발표 중 주의할 표현

슬라이드에는 리뷰 화면에서 “수정·삭제”한다고 적혀 있지만, 현재 구현에서는 리뷰 화면이 **확인 단계**입니다. 실제 수정과 삭제는 리뷰 다음에 열리는 **3D 편집기**에서 이루어진다고 설명하는 것이 정확합니다.

## 다음 페이지로 넘어가는 말

“이제 방을 스캔하고 편집하는 중에 앱이 종료되거나 인터넷 저장에 실패해도 작업을 잃지 않도록 만든 저장 구조를 설명드리겠습니다.”

# 4페이지 — 로컬 자동 저장과 작업 복구

## 발표 대본

“3D 공간 편집은 일반 텍스트보다 데이터가 크고, 가구를 하나 옮길 때마다 위치와 방향 같은 여러 값이 바뀝니다. 이때 앱이 갑자기 종료되거나 서버 저장에 실패하면 사용자의 작업이 사라질 수 있습니다.

이를 막기 위해 저희는 편집 내용을 먼저 휴대폰 안에 자동 저장합니다.

작동 방식은 메모 앱의 임시 저장과 비슷합니다. 사용자가 가구를 옮기거나 회전시키거나 삭제하면, 앱은 변경 내용을 감지합니다. 그리고 조작이 잠깐 멈춘 뒤 현재 상태를 휴대폰 안의 임시 작업 파일에 기록합니다.

매번 손가락을 조금 움직일 때마다 즉시 저장하면 저장 작업이 너무 자주 발생할 수 있기 때문에, 마지막 변경 후 약 0.65초 동안 추가 조작이 없을 때 저장합니다. 사용자는 거의 즉시 저장되는 것처럼 느끼지만 앱은 불필요한 반복 작업을 줄일 수 있습니다.

편집 도중 앱을 나갔다가 같은 방의 편집기를 다시 열면, 저장된 임시 작업이 있을 경우 이어서 작업할지 아니면 버리고 원래 상태로 시작할지를 선택할 수 있습니다.

서버 저장에 실패해도 휴대폰 안의 임시 작업은 바로 삭제하지 않습니다. 따라서 인터넷 연결이 좋지 않거나 서버에 문제가 생겼을 때도 사용자가 만든 결과를 최대한 지킬 수 있습니다.

또 실행 취소와 다시 실행 기능도 넣었습니다. 가구를 잘못 옮기거나 삭제했을 때 이전 상태로 돌아갈 수 있고, 다시 실행을 누르면 취소했던 작업을 복원할 수 있습니다.

정리하면, 휴대폰 안의 자동 저장은 작업 중인 원고를 보호하는 임시 보관함이고, 서버 저장은 여러 기기에서 사용할 수 있도록 올리는 최종 보관함입니다.”

## 변경 후 자동 저장되는 과정

1. 사용자가 가구를 이동·회전·교체·삭제합니다.
2. 앱이 “방의 상태가 바뀌었다”는 것을 감지합니다.
3. 약 0.65초 동안 추가 변경이 없으면 현재 편집 상태를 저장합니다.
4. 저장 도중 문제가 생겨도 이전 파일이 쉽게 망가지지 않도록 안전한 방식으로 파일을 교체합니다.
5. 같은 방 편집기를 다시 열 때 임시 저장본이 있으면 복구 여부를 묻습니다.
6. 서버 저장까지 성공하면 해당 시점의 정상 저장 상태를 확정합니다.

## 실행 취소와 다시 실행의 원리

- 가구를 편집하기 전 상태를 순서대로 기억합니다.
- **실행 취소**를 누르면 가장 최근 변경 전 상태로 돌아갑니다.
- **다시 실행**을 누르면 방금 취소했던 상태를 다시 적용합니다.
- 현재 앱은 최근 30단계까지 기억해서 메모리가 끝없이 늘어나는 것을 막습니다.
- 앱을 완전히 종료한 뒤에도 임시 편집본은 복구할 수 있지만, 실행 취소의 30단계 기록 전체가 그대로 복원되는 것은 아닙니다.

## 게스트와 로그인 사용자의 저장 차이

- 게스트는 현재 사용 중인 기기 안에서 임시 프로젝트와 편집 작업을 다룹니다.
- 로그인 사용자는 휴대폰 안의 임시 저장으로 작업을 보호하면서, 최종 결과를 서버에도 저장할 수 있습니다.
- 서버 저장이 실패하면 로컬 임시 저장본을 남겨 두어 작업 손실을 줄입니다.

## 정확하게 말해야 하는 부분

“앱을 다시 켜면 자동으로 편집 화면부터 열린다”는 의미는 아닙니다. 사용자가 **같은 방의 편집기를 다시 열었을 때** 임시 작업을 발견하고 복구할지 묻는 방식입니다.

# 5페이지 — 로그인 없이 둘러보는 게스트 모드

## 발표 대본

“저희는 사용자가 앱의 가치를 확인하기도 전에 회원가입부터 요구하지 않도록 게스트 모드를 만들었습니다.

게스트로 시작하면 현재 기기 안에 임시 프로젝트를 만들 수 있고, 방을 스캔하고, 결과를 리뷰하고, 3D 편집기에서 가구를 움직이거나 삭제하는 핵심 기능을 체험할 수 있습니다.

이때 작업 내용은 서버 계정이 아니라 현재 기기의 로컬 공간에서 관리합니다. 그래서 처음 앱을 둘러보는 데에는 로그인 정보가 필요하지 않습니다.

다만 게스트 작업은 서버와 연결되어 있지 않기 때문에 다른 휴대폰이나 태블릿에서 같은 프로젝트를 이어서 사용할 수 없습니다. 서버에 결과를 올리거나 여러 기기에서 동기화하려면 로그인이 필요합니다.

로그인한 사용자는 프로젝트와 편집 결과를 서버에 저장할 수 있습니다. 그러면 같은 계정으로 로그인한 다른 기기에서도 프로젝트를 불러와 이어서 작업할 수 있습니다.

또 사진을 기반으로 새로운 3D 가구를 생성하는 서버 기능도 로그인이 필요합니다. 이 기능은 휴대폰 안에서만 처리되는 것이 아니라 서버에서 생성 작업이 진행되기 때문입니다.

즉, 게스트 모드는 회원가입 없이 스캔과 편집의 핵심 경험을 먼저 보여 주는 체험 공간이고, 로그인은 결과를 안전하게 서버에 보관하고 여러 기기로 확장하는 단계입니다.”

## 게스트가 앱을 사용하는 순서

1. 로그인 화면에서 게스트로 둘러보기를 선택합니다.
2. 앱이 현재 기기에서 사용할 임시 프로젝트를 만듭니다.
3. 사용자가 방을 스캔합니다.
4. 리뷰 화면에서 인식된 방과 가구 정보를 확인합니다.
5. 3D 편집기에서 가구를 이동·회전·교체·삭제합니다.
6. 편집 내용은 로컬 임시 저장 기능으로 보호합니다.
7. 서버 저장이나 다른 기기 동기화가 필요한 시점에는 로그인을 안내합니다.

|---|---:|---:|
| 방 스캔 | 가능 | 가능 |
| 스캔 결과 확인 | 가능 | 가능 |

## 정확하게 말해야 하는 부분

- 게스트 프로젝트가 로그인하는 순간 자동으로 계정에 옮겨지는 구조는 아닙니다. 로그인 전에는 이 점을 안내합니다.
- 현재 구현을 설명할 때는 “게스트 작업은 **현재 사용 세션과 기기 안에서 임시로 관리된다**”라고 말하는 것이 가장 안전합니다.
- 따라서 게스트 모드는 장기 보관보다는 핵심 기능을 부담 없이 체험하게 하는 데 초점이 있습니다.

# 6페이지 — 모바일 저장 공간과 캐시 관리

## 발표 대본

“3D 파일은 사진이나 일반 문서보다 용량이 큰 편입니다. 방 스캔 결과와 가구 모델을 계속 내려받으면 사용자가 인식하지 못하는 사이에 휴대폰 저장 공간을 많이 사용할 수 있습니다.

그래서 저희는 꼭 보관해야 할 사용자 작업과, 필요할 때 다시 만들거나 내려받을 수 있는 임시 파일을 구분했습니다.

여기서 캐시는 한 번 사용한 자료를 다음에 더 빨리 열기 위해 잠시 보관하는 임시 상자라고 생각하시면 됩니다.

예를 들어 서버에서 내려받은 방의 스캔 정보와 3D 장면, 그리고 사진으로 3D 가구를 만드는 과정에서 생긴 중간 파일은 캐시에 저장됩니다. 같은 화면을 다시 열 때 매번 처음부터 내려받지 않아도 되기 때문에 더 빠르게 보여 줄 수 있습니다.

하지만 이런 파일이 계속 쌓이면 저장 공간을 차지합니다. 사용자는 설정 화면에서 현재 캐시가 얼마나 큰지 확인하고, 필요하면 한 번에 삭제할 수 있습니다.

캐시 삭제를 눌러도 사용자가 저장한 가구나 편집 중인 복구 파일처럼 중요한 데이터는 지우지 않습니다. 다시 받을 수 있는 방 스캔 파일, 3D 장면, 생성 과정의 중간 파일만 정리합니다.

삭제한 파일이 나중에 다시 필요해지면 서버에서 다시 내려받거나 생성 과정을 통해 다시 준비할 수 있습니다. 따라서 앱의 속도를 위해 임시 보관하는 장점은 유지하면서도, 사용자가 원할 때 저장 공간을 직접 확보할 수 있습니다.

결과적으로 저희 앱은 현실 공간을 3D로 바꾸는 기능뿐 아니라, 작업 손실 방지, 게스트 체험, 저장 공간 관리까지 실제 모바일 환경에서 안정적으로 사용할 수 있도록 구현했습니다.”

## 캐시에 들어가는 파일

- 서버에서 내려받은 방 구조 정보
- 화면에 보여 줄 방의 3D 장면 파일
- 사진 기반 3D 가구 생성 과정에서 생긴 중간 파일
- 다시 다운로드하거나 다시 생성할 수 있는 임시 자료

## 캐시 삭제 후에도 남는 중요한 데이터

- 사용자가 저장한 가구 데이터
- 편집 도중 자동 저장된 복구용 초안
- 프로젝트 자체의 핵심 정보

## 캐시 삭제가 작동하는 순서

1. 설정 화면을 열면 앱이 캐시 폴더들의 크기를 계산합니다.
2. 계산된 용량을 사용자가 이해하기 쉬운 단위로 표시합니다.
3. 사용자가 삭제 버튼을 누르면 캐시 폴더 안의 임시 파일을 정리합니다.
4. 삭제 후 용량을 다시 계산해 줄어든 결과를 보여 줍니다.
5. 나중에 해당 자료가 다시 필요하면 서버에서 다시 내려받습니다.

## 쉬운 비유

- 사용자 프로젝트와 저장한 가구: 다시 만들기 어려운 ‘원본 서류’
- 자동 저장 초안: 작성 중인 내용을 보호하는 ‘임시 원고’
- 캐시: 필요하면 다시 구할 수 있는 ‘복사본 상자’
- 캐시 삭제: 원본은 남기고 복사본 상자만 비우는 작업

# 질문을 받았을 때 사용할 짧은 답변

## “LiDAR가 정확히 무엇인가요?”

“빛을 쏘고 돌아오는 시간을 이용해 물체까지의 거리를 재는 센서입니다. 카메라가 물체의 모습을 본다면 LiDAR는 그 물체가 얼마나 멀리 있는지를 측정합니다.”

## “사진 한 장으로도 방을 만들 수 있지 않나요?”

“사진 한 장만으로는 보이지 않는 면과 정확한 거리를 알기 어렵습니다. 저희 방식은 사용자가 방 안을 움직이며 여러 방향을 비추고 거리까지 함께 측정하기 때문에 공간 구조를 만드는 데 더 적합합니다.”

## “RoomPlan은 직접 만든 기술인가요?”

“공간을 인식하는 기본 기능은 애플의 RoomPlan을 사용합니다. 저희는 이를 앱 화면과 연결하고, 스캔 시작과 종료, 결과 변환, 리뷰, 3D 편집, 자동 저장, 게스트 모드와 캐시 관리가 하나의 서비스로 작동하도록 구현했습니다.”

## “USDZ와 JSON을 왜 둘 다 저장하나요?”

“USDZ는 방을 3D로 보여 주기 좋고, JSON은 가구 하나하나의 위치와 크기를 읽고 수정하기 좋습니다. 하나는 보이는 모형이고 다른 하나는 편집에 사용하는 설계 정보입니다.”

## “스캔 정확도는 어떻게 높이나요?”

“방 전체를 천천히 비추고, 가구가 가려진 부분은 다른 각도에서도 보여 주는 것이 중요합니다. 앱은 여러 방향에서 들어온 정보를 합쳐 최종 결과를 정리합니다.”

## “인터넷이 끊기면 편집 내용도 사라지나요?”

“편집 내용은 먼저 휴대폰 안에 자동 저장합니다. 그래서 서버 저장에 실패하더라도 로컬 임시 작업을 남겨 두고, 같은 방 편집기를 다시 열었을 때 복구할 수 있습니다.”

## “게스트도 모든 기능을 사용할 수 있나요?”

“방 스캔과 로컬 3D 편집 같은 핵심 기능은 사용할 수 있습니다. 하지만 서버 저장, 다른 기기 동기화, 서버에서 처리하는 사진 기반 3D 가구 생성은 로그인이 필요합니다.”

## “캐시를 지우면 제가 만든 가구도 없어지나요?”

“아닙니다. 캐시 삭제는 다시 내려받거나 다시 생성할 수 있는 임시 파일만 정리합니다. 저장한 가구와 편집 복구용 초안 같은 중요한 데이터는 별도로 보관합니다.”

## “앱이 꺼진 뒤에도 실행 취소 기록이 전부 남나요?”

“편집된 최종 상태는 임시 저장본으로 복구할 수 있지만, 앱을 끄기 전의 실행 취소 단계 전체가 그대로 저장되는 구조는 아닙니다.”

# 발표 직전 30초 요약 카드

1. 카메라는 모습을 보고 LiDAR는 거리를 잰다.
2. RoomPlan이 벽·문·창문·가구와 크기·위치·방향을 정리한다.
3. USDZ는 보이는 3D 모형, JSON은 편집용 설계 정보다.
4. 편집 내용은 약 0.65초 뒤 휴대폰 안에 자동 저장된다.
5. 게스트는 핵심 기능을 체험하고, 로그인 사용자는 서버 저장과 동기화를 쓴다.
6. 캐시는 다시 구할 수 있는 임시 파일만 지우며 중요한 작업은 남긴다.
