## 1. 이 챕터의 핵심 메시지

전체 프로젝트 소개는 기존 발표 자료에서 다룬다고 가정한다. 이 챕터에서는 Spatium 전체 서비스가 아니라 **3D 에디터가 어떤 기능을 제공하며, 구현 과정에서 어떤 기술 문제를 해결했는지**에 집중한다.

청중이 기억해야 할 문장은 다음과 같다.

> **Spatium 3D 에디터는 스캔한 방을 실측 기반의 편집 공간으로 바꾸고, 공간 제약을 적용한 편집 결과를 방 구조와 함께 저장·복원한다.**

발표 흐름은 크게 두 부분으로 나눈다.

```text
3D 에디터 개요
  ↓
PART 1. 구현한 기능
  ├─ 공간·가구 복원과 transform 편집
  ├─ 문·창문과 방 구조 편집
  ├─ 카메라·측정·꾸미기
  └─ Undo / Redo와 저장·복원
  ↓
PART 2. 트러블슈팅
  ├─ 회전 가구 충돌 오차
  ├─ 빠른 드래그 벽 통과
  ├─ 불규칙한 벽과 개구부 충돌
  ├─ 문·창문 돌출
  └─ 벽 infill 상태 불일치
  ↓
기술적 성과
```

## 3. 슬라이드별 상세 내용

## 슬라이드 1. 스캔한 방을 편집 가능한 3D 공간으로 바꿨다

### 화면에 넣을 문구

**스캔한 방을 편집 가능한 3D 공간으로 바꿨다**

```text
RoomPlan 스캔
→ 방·가구 복원
→ 공간 제약 기반 편집
→ 편집 상태 저장·복원
```

보조 문구:

> 단순 3D 뷰어가 아니라 실제 치수와 방 구조를 반영하는 인테리어 편집기

### PPT 본문 상세안 — 복사 후 축약

**3D 에디터의 역할**

- iPhone RoomPlan으로 생성한 USDZ 공간 모델과 metadata JSON을 불러온다.
- 스캔된 방의 벽·바닥·문·창문과 기존 가구를 실제 치수와 위치에 맞춰 3D 씬으로 복원한다.
- 사용자가 가구를 추가하고 이동·회전·높이·크기·교체·삭제할 수 있는 편집 환경을 제공한다.
- 모든 transform은 벽과 문·창문의 충돌 제약을 통과한 경우에만 최종 적용한다.
- 문·창문 삭제, 벽 메우기, 벽 색상 변경처럼 방 구조 자체가 달라지는 편집도 지원한다.
- 최종 결과는 객체 목록뿐 아니라 방 Mesh까지 포함한 metadata로 저장해 다음 접속에서 동일하게 복원한다.

**입력과 출력**

```text
입력
  RoomPlan USDZ + 공간 metadata + 가구 GLB

처리
  공간 복원 + 모델 fitting + transform 편집 + 충돌 제약

출력
  방 구조와 모든 편집 결과를 포함한 replayable metadata
```

**이 기능의 의미**

> 실제 방을 시각화하는 단계에서 끝나지 않고, 실측 공간 안에서 인테리어를 시험하고 그 상태를 다시 이어서 편집할 수 있도록 확장했다.

### 화면 구성

- 완성된 3D 에디터 화면을 크게 사용한다.
- 화면 오른쪽 또는 하단에 4단계 흐름만 배치한다.
- 전체 프로젝트 이름과 서비스 설명은 반복하지 않는다.
- 기존 발표에서 3D 에디터 파트로 넘어오는 챕터 시작 화면 역할을 맡긴다.

### 발표 멘트

“다음은 프로젝트의 핵심 기능 중 하나인 3D 에디터입니다. RoomPlan으로 스캔한 방을 단순히 보여주는 것이 아니라, 실제 치수와 공간 제약을 적용해 가구와 방 구조를 편집하고 그 결과를 다시 복원할 수 있도록 구현했습니다.”

# PART 1. 구현한 기능

## 슬라이드 2. 실측 공간에서 가구를 자유롭게 편집한다

### 화면에 넣을 문구

**실측 공간에서 가구를 자유롭게 편집한다**

- RoomPlan USDZ와 metadata로 방·기존 가구 복원
- 카탈로그 가구 추가 및 실측 치수 입력
- 드래그 이동, Y축 회전, 높이 조정
- 비율을 유지하는 균일 크기 조정
- 가구 교체·삭제와 사용자 생성 모델 배치

하단 설명:

```text
GLB bounds 계산 → 실측 dimensions fitting → 바닥 pivot 보정 → 편집용 OBB 생성
```

### PPT 본문 상세안 — 복사 후 축약

**1. 스캔 공간과 기존 객체 복원**

- RoomPlan USDZ에서 벽과 바닥 Mesh를 불러오고 metadata에서 문·창문·가구 목록을 읽는다.
- 객체의 position, rotation, dimensions를 적용해 스캔 당시의 배치를 재구성한다.
- 저장된 `_spatiumRoom`이 있으면 원본 USDZ보다 편집된 방 구조를 우선 복원한다.

**2. 새 가구 배치**

- 가구 카탈로그에서 원하는 모델을 선택한다.
- 사용자가 가로·세로·높이를 입력하면 내부 단위인 m로 변환해 초기 크기로 사용한다.
- 새 객체는 방 바닥의 유효 위치에 배치되고, 초기 상태부터 벽 침범 여부를 검사한다.
- 실제 GLB를 불러오지 못한 경우에도 dimensions 기반 fallback box를 생성해 편집을 계속할 수 있다.

**3. GLB 모델 정규화**

- GLB마다 원본 단위, scale, hierarchy, pivot이 다르므로 원본 transform을 그대로 신뢰하지 않는다.
- 모든 하위 Mesh의 geometry bounds를 계산해 모델 전체의 실제 크기를 구한다.
- metadata 또는 사용자 입력 dimensions에 맞춰 scale을 적용한다.
- 모델 하단이 바닥에 정확히 닿도록 pivot과 Y 위치를 보정한다.
- 보이지 않는 pick box와 local OBB를 생성해 선택과 충돌의 기준으로 사용한다.

**4. 지원하는 transform 편집**

| 편집 | 처리 방식 |
| --- | --- |
| 수평 이동 | 포인터 Ray와 바닥 Plane 교차점으로 X-Z 목표 위치 계산 |
| 회전 | Y축 quaternion 또는 각도 슬라이더 적용 |
| 높이 | 바닥부터 가구 하단까지의 간격을 cm 단위로 조정 |
| 크기 | 가장 긴 변 기준으로 세 축을 같은 비율로 조정 |
| 교체 | 기존 위치·방향을 유지하고 새 모델 bounds에 맞춰 다시 fitting |
| 삭제 | 씬·편집 상태·저장 대상에서 객체 제거 |

**5. 편집 안정성**

- 이동 중에는 처음 잡은 지점과 객체 중심의 offset을 유지해 객체가 포인터 중앙으로 튀지 않게 한다.
- 회전·높이·크기 변경 후 충돌하면 마지막으로 유효했던 transform으로 되돌린다.
- 크기 변경 시 가구 비율과 기존 하단 높이를 유지한다.
- 사용자 생성 GLB도 일반 카탈로그 가구와 같은 편집 파이프라인을 사용한다.

### 화면 구성

- 왼쪽: 방 복원 또는 가구 추가 전 화면
- 오른쪽: 가구 배치·회전·크기 조정 후 화면
- 하단: 모델 정규화 흐름을 한 줄로 표시
- 기능마다 UI 캡처를 여러 개 나열하기보다 대표 편집 화면 하나에 콜아웃을 넣는다.

### 발표 멘트

“사용자는 스캔된 방 안에 카탈로그 가구를 추가하고 이동, 회전, 높이, 크기를 조절할 수 있습니다. GLB마다 단위와 pivot이 다르기 때문에 실제 geometry bounds를 계산하고 입력 치수에 맞춰 정규화한 뒤 바닥 중심으로 보정했습니다.”

## 슬라이드 3. 문·창문과 방 구조까지 편집한다

### 화면에 넣을 문구

**가구뿐 아니라 문·창문과 방 구조도 편집한다**

| 편집 대상 | 지원 동작 |
| --- | --- |
| 일반 가구 | 추가·이동·회전·높이·크기·교체·삭제 |
| 문·창문 | 같은 reference 계열 모델로 교체 |
| 개구부 | 그대로 유지하거나 벽으로 메우기 |
| 벽 | 색상 변경과 카메라 기준 투명화 |

강조 문구:

> 문을 벽으로 메우면 새 Mesh와 Collider를 생성해 실제 벽처럼 동작한다.

### PPT 본문 상세안 — 복사 후 축약

**일반 가구와 reference 객체의 구분**

- 일반 가구는 자유롭게 이동·회전·높이·크기를 바꿀 수 있는 `sourceType: object` 객체다.
- 문과 창문은 벽의 개구부에 연결된 `sourceType: door/window` reference 객체다.
- 문·창문은 방 구조의 일부이므로 위치를 자유롭게 옮기지 않고 같은 계열 모델로 교체한다.
- 일반 가구를 이동하거나 회전할 때는 벽뿐 아니라 문·창문의 world OBB도 장애물로 검사한다.

**문·창문 교체**

- 사용자가 기존 문 또는 창문을 선택한 뒤 같은 reference 카테고리 모델을 선택한다.
- 기존 position과 방향은 유지하고 새 모델의 bounds를 기준으로 크기를 다시 맞춘다.
- 교체 모델에도 실제 벽의 두께와 중심을 측정한 fitting 로직을 적용한다.

**문·창문 삭제 방식**

| 방식 | 처리 결과 |
| --- | --- |
| 개구부 유지 | reference 모델만 제거하고 기존 벽의 구멍은 그대로 유지 |
| 벽으로 메우기 | 개구부 크기와 벽 두께에 맞는 infill Mesh를 생성 |

**벽으로 메우기 처리 범위**

- reference의 가로·높이 bounds와 소속 벽의 실측 두께를 이용해 채움 Mesh를 만든다.
- 주변 벽의 material을 clone해 색상·roughness·metalness·texture를 동일하게 유지한다.
- 새 Mesh에 벽 metadata를 설정하고 wall Collider를 다시 생성한다.
- 새 벽이 기존 가구와 겹치면 가구를 방 안쪽으로 밀어내 유효 위치로 복구한다.
- infill Mesh를 `_spatiumRoom`에 포함해 저장 후에도 실제 벽으로 복원한다.

**벽 표시 기능**

- 사용자가 선택한 벽 색상을 전체 wall material에 적용한다.
- 카메라 방향과 벽 face normal을 비교해 시야를 가리는 벽 면만 투명하게 만든다.
- 벽 전체를 숨기는 대신 필요한 face만 투명화해 방 내부와 개구부가 자연스럽게 보이게 한다.

### 화면 구성

- 문·창문 교체 전후를 같은 카메라 각도로 비교한다.
- 하단에는 `개구부 유지 / 벽으로 메우기` 두 결과를 나란히 둔다.
- 벽 메우기의 세부 알고리즘은 트러블슈팅 파트에서 설명한다.

### 발표 멘트

“일반 가구뿐 아니라 방 구조의 일부인 문과 창문도 교체할 수 있습니다. 삭제할 때는 개구부를 남기거나 벽으로 메우는 방식을 선택하며, 벽으로 메우면 충돌 영역까지 갱신해 실제 벽처럼 작동합니다.”

## 슬라이드 4. 편집 결과를 다시 사용할 수 있는 상태로 저장한다

### 화면에 넣을 문구

**보기·편집·되돌리기·재접속을 하나의 상태로 연결했다**

주요 편집 지원:

- 일반 3D / Skyview / 눈높이 1인칭 시점
- 카메라 기준 벽 면 투명화
- 방 폭·깊이·높이·면적 측정
- 책장·서랍장 선반 피규어 꾸미기
- 추가·삭제·교체·벽 메우기까지 Undo / Redo
- 방 Mesh를 포함한 replayable metadata 저장

하단 강조 문구:

> 원본 USDZ가 없어도 저장된 편집 상태를 다시 열 수 있다.

### PPT 본문 상세안 — 복사 후 축약

**카메라와 공간 확인 기능**

- 일반 3D 모드에서는 OrbitControls로 방을 회전·확대·축소하며 살펴본다.
- Skyview는 카메라 position과 target을 저장한 뒤 방 전체를 위에서 확인하는 시점으로 전환한다.
- 1인칭 시점은 눈높이 카메라와 WASD·방향키 이동, 드래그 시야 회전을 제공한다.
- 카메라가 방 바깥에 있을 때 시야를 가리는 벽 face를 자동으로 투명화한다.

**측정 기능**

- 바닥 triangle과 외곽선을 분석해 방의 폭·깊이·높이와 바닥 면적을 계산한다.
- 바닥 Mesh 분석이 어려운 경우 전체 bounds 기반 결과로 fallback한다.
- 선택한 가구의 dimensions도 cm 단위 라벨로 표시한다.
- 웹에서는 CSS2DRenderer를 이용해 측정선과 수치 라벨을 3D 씬 위에 표시한다.

**꾸미기 모드**

- 실제 선반 표면이 있는 전용 책장·서랍장 모델에서만 꾸미기 모드를 활성화한다.
- 카메라를 가구 정면의 근접 시점으로 전환하고 실제 triangle Mesh에 Raycast한다.
- 위쪽을 향한 선반 면에만 피규어를 배치하며 표면을 벗어나지 않도록 이동을 제한한다.
- 피규어는 부모 가구의 로컬 좌표로 저장되어 가구가 이동·회전하면 함께 움직인다.

**Undo / Redo**

- 이동·회전·높이·크기 같은 transform 변경을 되돌릴 수 있다.
- 가구 추가·삭제·교체, 피규어 배치, 문·창문 삭제, 벽 메우기까지 같은 snapshot 구조로 복원한다.
- transform만 바뀐 경우 기존 Three.js Node를 재사용한다.
- 객체 수나 방 구조가 바뀐 경우에만 replayable metadata로 씬을 다시 구성한다.

**저장 데이터**

```text
objects
  가구 ID, 모델 URL, dimensions, position, rotation, scale, decorations

doors / windows
  현재 남아 있는 reference와 교체 결과

_spatiumRoom
  벽·바닥 geometry, material, transform, infill Mesh

_spatiumExport
  저장 버전, 시간, scene config, 편집 항목
```

**저장·복원 흐름**

```text
현재 씬 순회
→ 객체·reference·decor 직렬화
→ 방 Mesh 직렬화
→ metadata.json 생성
→ POST /api/rooms/save
→ 다음 진입에서 저장 metadata 우선 복원
```

### 화면 구성

- 가장 큰 영역에는 저장 전과 재접속 후 같은 장면을 비교한다.
- 카메라·측정·꾸미기는 작은 보조 이미지 2~3개만 사용한다.
- 기능을 모두 같은 크기의 카드로 나열하지 않는다.

### 발표 멘트

“카메라와 측정 같은 편집 보조 기능을 제공하고, 변경 사항은 Undo와 Redo로 되돌릴 수 있습니다. 저장할 때는 가구 위치뿐 아니라 방 geometry와 material을 함께 저장하기 때문에 벽을 메운 상태까지 다시 복원됩니다.”

## 슬라이드 5. 회전한 가구의 충돌 오차를 OBB로 해결했다

### 화면에 넣을 문구

**회전한 가구의 충돌 오차를 OBB로 해결했다**

| 단계 | 내용 |
| --- | --- |
| 문제 | 회전 후 벽에 닿기 전에 막히거나 모서리가 벽을 통과함 |
| 원인 | 월드 축에 고정된 AABB가 실제 가구 방향을 반영하지 못함 |
| 해결 | local OBB를 `matrixWorld`로 변환해 회전된 footprint 계산 |
| 결과 | 가구 방향과 실제 크기에 맞는 충돌 판정 |

핵심식:

```text
local OBB + matrixWorld = world OBB
```

### PPT 본문 상세안 — 복사 후 축약

**문제 현상**

- 가구를 0도 상태로 배치했을 때는 충돌이 정상적으로 보였다.
- 가구를 30도, 45도, 90도로 회전하면 실제 모서리와 충돌 박스의 위치가 달라졌다.
- 벽에 충분한 공간이 있는데도 가구가 먼저 멈추거나, 반대로 회전한 모서리가 벽 안으로 들어갔다.

**원인 분석**

- 기존 AABB는 월드 X·Y·Z축에 정렬된 박스다.
- 가구가 회전해도 충돌 박스 자체는 회전하지 않고 회전된 모델을 모두 감싸는 더 큰 사각형이 된다.
- 시각적으로 보이는 모델의 footprint와 충돌 계산의 footprint가 일치하지 않았다.

**해결 방식**

- GLB fitting이 끝난 시점의 bounds로 객체 로컬 좌표계의 `localObb`를 만든다.
- 충돌 검사 시 객체의 `matrixWorld`를 적용해 중심·축·halfSize가 회전된 world OBB를 계산한다.
- world OBB를 벽의 `roomFacingNormal` 방향으로 투영해 실제 projection radius를 구한다.
- 벽의 길이·높이 span과 겹치면서 방 안쪽 경계를 넘는 경우에만 충돌로 판정한다.
- 회전과 크기 조정 후 충돌하면 이전의 유효한 quaternion 또는 scale로 rollback한다.

**적용 함수와 데이터**

```text
object.userData.localObb
worldObbForObject(object)
wallBlocksObjectObb(objectObb, wall)
hasWallCollision(object, activeColliders)
```

**개선 결과**

- 가구가 회전한 방향과 충돌 영역이 동일하게 움직인다.
- 벽에 실제로 닿는 지점까지 배치할 수 있어 불필요한 여백이 줄었다.
- 회전된 모서리가 벽을 침범하는 상황을 더 정확하게 차단했다.
- 동일한 OBB를 이동·회전·높이·크기 변경의 공통 검증 기준으로 사용할 수 있게 됐다.

### 화면 구성

- 동일한 회전 가구 위에 AABB와 OBB를 각각 겹쳐 비교한다.
- AABB는 회색 또는 빨간색, OBB는 서비스 포인트 색으로 표시한다.
- 표 전체보다 비교 이미지가 더 크게 보이도록 구성한다.

### 발표 멘트

“AABB는 월드 축에 고정되어 있어 가구가 회전하면 실제 footprint와 차이가 커집니다. 객체의 local OBB를 world transform으로 변환해 가구의 방향과 폭·깊이를 그대로 충돌에 반영했습니다.”

## 슬라이드 6. 빠른 드래그의 벽 통과를 Sweep으로 막았다

### 화면에 넣을 문구

**현재 위치뿐 아니라 이동 경로를 검사했다**

| 단계 | 내용 |
| --- | --- |
| 문제 | 가구를 빠르게 끌면 한 프레임 사이에 벽을 건너뜀 |
| 원인 | 이동 시작점과 끝점만 검사해 중간 경로가 누락됨 |
| 해결 | 이동 벡터를 작은 step으로 분할해 매 단계 충돌 검사 |
| 결과 | 벽 통과 방지 + 벽 표면을 따라 자연스러운 slide |

```text
큰 movement → step 분할 → 충돌 검사 → inward 성분 제거 → 접선 이동 유지
```

### PPT 본문 상세안 — 복사 후 축약

**문제 현상**

- 천천히 드래그할 때는 벽에서 정상적으로 멈췄다.
- 마우스를 빠르게 움직이면 한 프레임의 위치 변화가 벽 두께보다 커졌다.
- 시작 위치와 최종 위치 모두 충돌하지 않으면 중간에 벽이 있어도 객체가 반대편으로 이동했다.

**원인 분석**

- 일반적인 overlap 검사는 현재 프레임의 정적인 위치만 판단한다.
- 이전 프레임과 다음 프레임 사이의 연속된 이동 경로는 검사하지 않는다.
- 이동량이 Collider 두께보다 큰 경우 벽과 겹치는 중간 상태를 한 번도 계산하지 않는 tunneling이 발생한다.

**해결 방식**

- 요청된 movement vector의 길이를 기준으로 필요한 step 수를 계산한다.
- 전체 이동을 scene config의 `wallConstraints.sweepStep` 이하의 작은 이동으로 분할한다.
- 각 step마다 가구의 world OBB와 모든 active Collider의 충돌을 검사한다.
- 벽을 향하는 inward 성분은 제거하고 벽과 평행한 접선 성분은 유지한다.
- 허용된 step만 누적해 실제로 적용 가능한 최종 movement를 만든다.

**처리 흐름**

```text
requestedMovement
→ splitBySweepStep()
→ applyCandidateStep()
→ activeColliders() 검사
→ inward component 제거
→ tangent component 재검사
→ allowedMovement 누적
```

**개선 결과**

- 드래그 속도와 관계없이 가구가 벽을 건너뛰지 않는다.
- 충돌 순간 전체 이동을 취소하지 않고 벽에 닿는 지점까지 이동한다.
- 벽에 닿은 상태에서도 벽과 평행한 방향으로 자연스럽게 미끄러진다.
- 사용자 입력 속도에 따른 충돌 결과의 차이가 줄어 일관된 조작감을 제공한다.

### 화면 구성

- 왼쪽: 한 번에 이동해 벽을 통과하는 잘못된 경로
- 오른쪽: 여러 step으로 나뉘어 벽에서 제한되는 경로
- 가능하면 실제 동작 GIF 또는 짧은 영상을 넣는다.

### 발표 멘트

“OBB는 현재 위치의 교차는 정확히 판단하지만 이동 경로까지 검사하지는 않습니다. 빠른 드래그의 이동량을 작은 step으로 나누어 tunneling을 막고, 벽과 평행한 성분은 유지해 표면을 따라 움직일 수 있게 했습니다.”

## 슬라이드 7. 불규칙한 벽을 face 단위 Collider로 해석했다

### 화면에 넣을 문구

**벽 전체 박스 대신 실제 벽 면을 충돌 영역으로 사용했다**

| 단계 | 내용 |
| --- | --- |
| 문제 | 문·창문 개구부까지 막히고 ㄱ자 벽 주변이 과도하게 차단됨 |
| 원인 | 불규칙한 벽 Mesh 전체를 하나의 bounds로 처리함 |
| 해결 | triangle 분석으로 face Collider와 `roomFacingNormal` 생성 |
| 결과 | 실제 벽 면만 차단하고 개구부는 통과 가능 |

예외 처리:

> face 분석이 불가능한 Mesh는 bounds 기반 OBB로 fallback한다.

### PPT 본문 상세안 — 복사 후 축약

**문제 현상**

- RoomPlan으로 스캔한 벽은 하나의 반듯한 직육면체가 아니라 여러 triangle로 구성된 Mesh다.
- 벽 Mesh 전체 bounds를 Collider로 사용하면 문·창문 개구부도 막힌 영역으로 처리됐다.
- ㄱ자 벽, 짧은 벽, 비스듬한 벽에서는 실제 벽이 없는 공간까지 충돌로 판단됐다.

**원인 분석**

- 전체 bounds는 Mesh의 세부 형태와 구멍을 표현하지 못한다.
- 하나의 벽 Mesh에 서로 다른 방향의 face가 들어 있어도 단일 박스로 합쳐졌다.
- 단순 OBB 교차만으로는 가구가 방 바깥 방향에서 겹치는 경우와 방 안쪽 경계를 침범하는 경우를 구분하기 어려웠다.

**해결 방식**

- 벽 geometry의 triangle과 normal을 분석해 실제 벽 표면에 해당하는 face를 분리한다.
- face별로 충돌 OBB, 길이·높이 방향 `spanAxes`, 2D 경계 `spanPolygon`을 만든다.
- 각 face에서 방 안쪽을 향하는 `roomFacingNormal`과 경계 projection을 계산한다.
- 가구 OBB가 벽 span과 겹치면서 room-facing boundary를 넘는 경우에만 이동을 제한한다.
- triangle 분석이 불가능한 예외 Mesh는 전체 bounds OBB로 fallback해 편집을 중단하지 않는다.

**Collider에 저장하는 정보**

```text
obb
spanAxes / spanPolygon
roomFacingNormal
roomFacingProjection
triangleStart / triangleCount
원본 wall Mesh 참조
```

**개선 결과**

- 문과 창문의 실제 개구부를 가구가 통과할 수 있게 됐다.
- ㄱ자 벽과 불규칙한 벽 주변의 과도한 충돌 영역이 줄었다.
- 카메라 기준 벽 투명화도 같은 face 정보를 활용할 수 있게 됐다.
- 일반 Mesh와 예외 Mesh를 모두 처리할 수 있는 견고한 fallback 경로를 확보했다.

### 화면 구성

- 벽 wireframe 위에 Collider를 표시한 디버그 화면을 사용한다.
- 문·창문 개구부에 Collider가 없는 모습이 명확히 보여야 한다.

### 발표 멘트

“스캔된 벽은 반듯한 박스가 아니기 때문에 전체 bounds를 사용하면 개구부까지 막힙니다. 벽 triangle을 분석해 face 단위 Collider를 만들고, 방 안쪽 normal을 기준으로 경계 침범을 판단했습니다.”

## 슬라이드 8. 문·창문을 실제 벽 두께에 맞췄다

### 화면에 넣을 문구

**평면으로 기록된 reference를 실제 벽 geometry로 보정했다**

| 단계 | 내용 |
| --- | --- |
| 문제 | 문·창문 모델이 벽 앞뒤로 튀어나옴 |
| 원인 | RoomPlan reference가 두께 0에 가까운 평면·벽 앞면 좌표로 저장됨 |
| 해결 | 가까운 벽의 실제 정점을 두께 축에 투영해 두께와 중심 측정 |
| 결과 | 모델 두께 clamp와 벽 중심 자동 정렬 |

```text
가까운 벽 탐색 → 두께 축 결정 → 정점 투영 → 두께 clamp → 중심 보정
```

### PPT 본문 상세안 — 복사 후 축약

**문제 현상**

- RoomPlan metadata의 문·창문 `dimensions.z`가 0 또는 매우 작은 값으로 기록됐다.
- 저장 position은 벽의 두께 중심이 아니라 방 안쪽 표면 좌표에 가까웠다.
- fallback 두께를 position 중심으로 대칭 적용하면 모델의 절반이 벽 앞으로 돌출됐다.

**원인 분석**

- RoomPlan의 문·창문은 부피를 가진 구조물이 아니라 벽 위의 평면 reference로 표현된다.
- 모델 GLB는 실제 프레임과 유리 두께를 갖고 있으므로 metadata를 그대로 적용할 수 없다.
- 가장 가까운 face triangle만 사용하면 문틀 옆면처럼 벽 두께와 무관한 면을 잘못 선택할 수 있었다.

**해결 방식**

1. reference position과 각 벽 Mesh의 world Box3 사이 거리를 계산한다.
2. 일정 거리 안에서 가장 가까운 벽 Mesh 전체를 소속 벽으로 선택한다.
3. 벽의 local bounds에서 가장 짧은 축을 두께 방향으로 판단한다.
4. 벽의 모든 geometry 정점을 해당 축에 투영해 min, max, center를 계산한다.
5. reference 모델 두께를 `실측 벽 두께 - margin` 이하로 clamp한다.
6. reference position을 벽의 두께 중심으로 이동한다.
7. 가까운 벽을 찾지 못하면 기존 position과 fallback 두께를 사용한다.

**핵심 함수**

```text
nearestWallMesh()
localThicknessAxis()
geometryProjectionRange()
fitReferenceToWallThickness()
measureWallThicknessAtPosition()
```

**개선 결과**

- 문과 창문 모델이 벽 앞뒤로 튀어나오는 현상이 줄었다.
- 서로 다른 두께의 벽에서도 고정 값이 아닌 실제 스캔 geometry에 맞춰졌다.
- 모델 교체 시에도 같은 fitting을 다시 적용해 일관성을 유지했다.
- 벽 infill 생성에서도 동일한 두께 측정 결과를 재사용할 수 있게 됐다.

### 화면 구성

- 돌출된 Before와 벽 안에 맞춰진 After를 같은 시점으로 비교한다.
- 벽의 두께 축과 중심선을 이미지 위에 간단히 표시한다.

### 발표 멘트

“RoomPlan의 문과 창문은 실제 두께 정보가 부족합니다. 고정 두께를 추정하는 대신 가장 가까운 벽 Mesh의 정점을 투영해 실제 두께와 중심을 측정하고 reference 모델을 자동으로 맞췄습니다.”

## 슬라이드 9. 벽 메우기의 Mesh·충돌·저장을 동기화했다

### 화면에 넣을 문구

**벽을 메우면 시각 상태와 공간 제약을 함께 바꿔야 한다**

| 단계 | 내용 |
| --- | --- |
| 문제 | 새 벽의 색상이 다르거나 기존 가구가 벽 안에 남음 |
| 원인 | infill Mesh만 만들고 material·Collider·가구를 갱신하지 않음 |
| 해결 | 원본 벽 material clone, Collider 재생성, 전체 penetration resolve |
| 결과 | 새 벽이 렌더링·충돌·저장·복원에서 동일하게 동작 |

```text
infill Mesh 생성
→ 벽 material 복제
→ wall metadata 지정
→ Collider 재생성
→ 기존 가구 재검사
→ _spatiumRoom 저장
```

### PPT 본문 상세안 — 복사 후 축약

**문제 현상**

- 문을 삭제하고 박스 Mesh만 추가했을 때 새 영역의 색과 질감이 주변 벽과 달랐다.
- 새 벽을 만든 위치에 기존 가구가 있으면 가구가 벽 안에 파묻힌 상태로 남았다.
- 화면에서는 벽처럼 보이지만 Collider 목록에는 포함되지 않아 가구가 통과할 수 있었다.
- 객체 목록만 저장하면 재접속 시 새 벽이 사라졌다.

**원인 분석**

- 방 구조 변경은 렌더링 Mesh 하나만의 변경이 아니다.
- material, wall 분류 metadata, Collider, 기존 객체 제약, 직렬화 대상이 서로 다른 모듈에서 관리됐다.
- 한 단계라도 갱신하지 않으면 화면·충돌·저장 상태가 서로 달라졌다.

**해결 방식**

1. reference `localObb`의 가로·높이에 padding을 더해 개구부를 덮을 크기를 계산한다.
2. 소속 벽의 실측 두께와 중심으로 `createWallInfillMesh()`를 생성한다.
3. 매칭 벽의 material을 clone해 색상·roughness·metalness·texture를 유지한다.
4. `userData.isUsdWallMesh = true`를 설정해 일반 벽과 동일하게 분류한다.
5. 기존 reference를 제거하고 `createWallColliders()`를 다시 실행한다.
6. 모든 기존 가구에 `initializeWallConstraints()`를 적용해 새 벽과의 침투를 해소한다.
7. 현재 벽 색상이 지정돼 있으면 새 Mesh에도 다시 적용한다.
8. infill Mesh를 `_spatiumRoom.walls`에 직렬화한다.

**동기화 범위**

| 계층 | 갱신 내용 |
| --- | --- |
| 렌더링 | infill geometry와 주변 벽 material |
| 공간 의미 | wall Mesh 식별용 `userData` |
| 충돌 | wall Collider 전체 재생성 |
| 기존 객체 | penetration resolve 재실행 |
| 저장 | `_spatiumRoom`에 geometry·material·transform 포함 |
| 복원 | JSON 방 복원 후 wall metadata와 Collider 재구성 |

**개선 결과**

- 새 벽이 주변 벽과 같은 색상과 질감으로 표시된다.
- 가구가 새 벽을 통과하거나 벽 안에 남는 문제가 사라졌다.
- Undo / Redo에서 문 삭제와 벽 메우기 상태를 함께 복원할 수 있다.
- 저장 후 재접속해도 infill이 실제 벽으로 남아 동일한 충돌 제약을 제공한다.

### 화면 구성

- 삭제 전 / 개구부 유지 / 벽 메우기의 3단 비교를 사용한다.
- 벽 메우기 화면에는 가구가 새 벽에 막히는 결과도 보여준다.

### 발표 멘트

“처음에는 채움 Mesh만 생성해 색상과 충돌 상태가 일치하지 않았습니다. 기존 벽 material을 복제하고 Collider를 재생성한 뒤 모든 가구의 침투를 다시 해소했습니다. 마지막으로 방 Mesh 자체를 저장해 재접속 후에도 동일한 벽으로 복원했습니다.”

## 8. 예상 질문과 답변

### Q1. AABB 대신 OBB를 사용한 이유는 무엇인가요?

회전된 가구의 실제 footprint를 반영하기 위해서다. AABB는 월드 축에 고정되어 회전 시 불필요한 영역까지 충돌로 보거나 모서리 충돌을 놓칠 수 있다.

### Q2. OBB와 Sweep의 차이는 무엇인가요?

OBB는 특정 위치에서 회전된 객체의 충돌 형태를 정확히 계산한다. Sweep은 프레임 사이의 이동 경로를 작은 단계로 검사해 벽을 건너뛰는 현상을 막는다.

### Q3. 문·창문의 두께는 어떻게 결정하나요?

가장 가까운 벽 Mesh의 가장 짧은 축을 두께 방향으로 판단하고, 실제 정점을 그 축에 투영해 두께와 중심을 계산한다.

### Q4. 벽을 메운 결과는 어떻게 다시 복원하나요?

infill을 일반 벽 Mesh로 표시하고 geometry·material·transform을 `_spatiumRoom`에 저장한다. 재로딩 시 방 Mesh와 Collider를 다시 생성한다.

### Q5. 웹과 iOS의 충돌 로직은 같은가요?

핵심 의미는 같다. 웹은 Three.js OBB를 사용하고 iOS는 `FurnitureFootprint` 구조로 동일한 회전 footprint와 벽 제약을 계산한다.

### Q6. 저장 데이터가 너무 커지지 않나요?

객체 transform뿐 아니라 방 geometry까지 포함하므로 크기는 증가하지만, 원본 없이 구조 변경을 복원할 수 있다는 장점이 있다. 위치·회전만 바뀔 때는 기존 Node를 재사용하고 구조가 바뀔 때만 필요한 범위를 재생성한다.
