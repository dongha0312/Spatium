# Spatium 3D 에디터 PPT 구성 가이드

> 기준 문서: `SPATIUM_3D_EDITOR_INTEGRATED_GUIDE.md`  
> 권장 발표 시간: 8~10분  
> 권장 본편: 13장 + 선택 부록 4장  
> 예상 청중: 프로젝트 심사위원, 교수, 개발자 면접관, 팀원

---

## 1. 발표의 목표

이 발표의 목적은 기능 목록을 전부 설명하는 것이 아니다. 청중이 발표를 마쳤을 때 다음 내용을 기억하게 만드는 것이 핵심이다.

> **Spatium은 실제 스캔 공간을 단순히 보여주는 뷰어가 아니라, 실측 치수와 공간 제약을 반영해 편집하고 방 구조 변경까지 다시 복원할 수 있는 3D 인테리어 에디터다.**

발표의 중심 질문은 다음 세 가지다.

1. 실제 스캔 데이터를 어떻게 편집 가능한 3D 공간으로 바꿨는가?
2. 가구가 벽을 뚫지 않도록 어떤 기술적 문제를 해결했는가?
3. 편집 결과를 어떻게 완전하게 저장하고 다시 복원하는가?

따라서 발표 흐름은 다음 서사를 권장한다.

```text
실제 공간을 편집하고 싶다
  ↓
스캔 데이터에는 구조적 한계가 있다
  ↓
실측 fitting + OBB + 벽 face Collider + Sweep으로 해결한다
  ↓
방 구조 변경까지 저장해 완전한 편집 세션을 복원한다
  ↓
결과적으로 실제 공간 제약이 작동하는 3D 인테리어 편집기가 된다
```

## 3. 슬라이드별 상세 구성

## 슬라이드 1. 실제 방을 편집 가능한 3D 공간으로

### 화면에 넣을 문구

**Spatium 3D 공간 편집기**

RoomPlan 스캔 기반 실측 인테리어 편집 및 재구성

```text
팀명 또는 이름 · 프로젝트명 · 발표일
```

### 화면 구성

- 가장 좋은 에디터 실행 화면을 전체 배경으로 사용한다.
- 배경 위에 어두운 반투명 오버레이를 약하게 적용한다.
- 제목은 왼쪽 아래 또는 중앙에 크게 배치한다.
- 부제와 발표자 정보 외에는 넣지 않는다.

### 사용할 이미지

- 실제 스캔 방이 복원되고 가구가 배치된 에디터 화면
- 가능하면 벽, 문, 가구가 한눈에 보이는 Skyview 또는 사선 시점

### 발표 멘트

“Spatium은 iPhone으로 스캔한 실제 방을 3D로 복원하고, 실측 크기의 가구를 공간 제약 안에서 배치한 뒤 다시 편집할 수 있도록 저장하는 인테리어 에디터입니다.”

---

## 슬라이드 2. 스캔 결과는 곧바로 편집 공간이 되지 않는다

### 화면에 넣을 문구

**스캔 결과는 곧바로 편집 공간이 되지 않는다**

- 벽은 반듯한 박스가 아니라 불규칙한 삼각형 Mesh다.
- 문·창문은 두께가 거의 없는 평면으로 기록된다.
- 회전된 가구와 빠른 드래그는 단순 충돌 검사로 처리하기 어렵다.
- 가구 위치만 저장하면 벽을 메운 결과를 다시 만들 수 없다.

하단 강조 문구:

> **렌더링 문제가 아니라 공간 구조·충돌·상태 복원의 문제였다.**

### 화면 구성

- 왼쪽 60%: RoomPlan 원본 또는 wireframe 화면
- 오른쪽 40%: 위 네 가지 문제를 짧게 배치
- 문제마다 아이콘을 여러 개 붙이기보다 대표 스크린샷 한 장에 콜아웃 3개 정도만 사용한다.

### 발표 멘트

“USDZ를 화면에 띄우는 것만으로는 편집기가 되지 않았습니다. 스캔 벽은 불규칙하고, 문과 창문에는 실제 두께가 없으며, 충돌과 저장도 별도의 공간 로직이 필요했습니다.”

---

## 슬라이드 3. 스캔부터 재편집까지 하나의 흐름으로 연결했다

### 화면에 넣을 문구

**스캔부터 재편집까지 하나의 흐름으로 연결했다**

```text
RoomPlan 스캔
→ USDZ + metadata
→ 방·가구 복원
→ 제약 기반 편집
→ replayable metadata 저장
→ 저장 상태 재복원
```

아래에 짧은 설명 세 개만 둔다.

- 입력: 실제 공간 geometry와 실측 metadata
- 편집: 위치·회전·높이·크기·구조 변경
- 출력: 원본 없이 재생 가능한 편집 상태

### 화면 구성

- 가로형 6단계 흐름도를 사용한다.
- 각 단계에는 단어 1~2개와 대표 이미지 또는 단순 아이콘만 둔다.
- 복잡한 화살표 분기는 피한다.

### 발표 멘트

“입력부터 저장까지의 기준 데이터는 하나의 편집 세션으로 연결됩니다. 특히 저장 결과는 단순 가구 목록이 아니라 방 구조까지 포함한 재생 가능한 metadata입니다.”

---

## 슬라이드 4. 웹과 iOS는 같은 편집 모델을 공유한다

### 화면에 넣을 문구

**렌더러는 달라도 편집 모델은 같다**

중앙 핵심식:

```text
Room + Object + Transform + Constraint
```

| 웹 | 공통 편집 의미 | iOS |
| --- | --- | --- |
| React + Three.js | 공간·객체·변환·제약 | SwiftUI + SceneKit |
| Raycaster | 객체 선택 | SCNView hitTest |
| Three.js OBB | 벽 충돌 | FurnitureFootprint |
| metadata JSON | 저장·복원 | RoomLayout / editedObjects |

### 화면 구성

- 좌측 웹, 중앙 공통 모델, 우측 iOS의 3단 구성을 사용한다.
- 실제 코드 파일명은 본편에서 최소화하고, 부록에 넣는다.
- 기술 로고를 나열하는 슬라이드로 만들지 않는다.

### 발표 멘트

“웹과 iOS는 사용하는 렌더러와 입력 방식은 다르지만, 공간과 객체, transform, constraint라는 같은 편집 모델을 공유합니다. 따라서 플랫폼별 구현은 달라도 사용자에게 제공하는 편집 의미는 같습니다.”

---

## 슬라이드 5. 실측 치수로 서로 다른 3D 모델을 정규화한다

### 화면에 넣을 문구

**모델 원본이 달라도 실측 크기는 일치해야 한다**

```text
GLB 로드
→ 전체 bounds 계산
→ metadata 치수에 맞게 scale
→ 바닥 중심으로 pivot 보정
→ local OBB와 pick box 생성
```

강조 문구:

> 모델의 원본 scale과 pivot을 신뢰하지 않고 실제 geometry를 기준으로 정규화한다.

### 화면 구성

- 같은 가구의 보정 전·후 비교를 가장 크게 보여준다.
- 보정 전에는 떠 있거나 크기가 맞지 않는 모습, 보정 후에는 바닥과 치수가 맞는 모습을 사용한다.
- 파이프라인은 화면 하단에 짧게 배치한다.

### 발표 멘트

“외부 GLB는 모델마다 단위와 pivot이 다릅니다. 그래서 원본 설정을 그대로 사용하지 않고 전체 geometry bounds를 계산해 실측 치수로 맞추고 바닥 중심을 다시 잡았습니다.”

---

## 슬라이드 6. 회전한 가구는 OBB로 충돌을 계산한다

### 화면에 넣을 문구

**AABB 대신 OBB로 실제 회전을 충돌에 반영했다**

왼쪽:

```text
AABB
- 월드 축에 고정
- 회전 시 빈 공간까지 충돌 영역에 포함
- 너무 일찍 막히거나 모서리가 통과
```

오른쪽:

```text
OBB
- 가구 회전과 함께 회전
- 실제 폭·깊이에 가까운 footprint
- 벽 normal 방향 projection 가능
```

하단 핵심식:

```text
local OBB + matrixWorld = world OBB
```

### 화면 구성

- 같은 회전 가구 위에 AABB와 OBB 박스를 각각 겹친 비교 그림을 사용한다.
- 텍스트보다 박스 차이가 눈에 먼저 들어오게 만든다.
- AABB는 회색 또는 빨간색, OBB는 서비스 포인트 색으로 표시한다.

### 발표 멘트

“축에 고정된 AABB는 회전 가구의 실제 모양을 표현하지 못합니다. Spatium은 local OBB를 world transform으로 변환해 회전된 footprint 그대로 벽 충돌을 검사합니다.”

---

## 슬라이드 7. 이동 경로를 검사해 빠른 드래그도 막는다

### 화면에 넣을 문구

**현재 위치만이 아니라 이동 경로를 검사한다**

```text
큰 이동 요청
→ 작은 step으로 분할
→ 각 step에서 충돌 검사
→ 벽 방향 성분 제거
→ 접선 방향 이동 유지
```

결과:

- 빠르게 끌어도 벽을 건너뛰지 않는다.
- 벽에 닿으면 멈추지 않고 표면을 따라 미끄러진다.

### 화면 구성

- 왼쪽은 한 프레임 이동으로 벽을 통과하는 잘못된 예시
- 오른쪽은 여러 점 또는 짧은 화살표로 나눈 Sweep 경로
- 가능하면 GIF나 짧은 화면 녹화를 넣는다.

### 발표 멘트

“빠른 드래그에서는 한 프레임의 이동량이 벽 두께보다 클 수 있습니다. 이동을 작은 step으로 나누고 각 단계에서 충돌을 검사해 tunneling을 막았습니다. 벽과 평행한 성분은 유지해 사용감도 자연스럽게 만들었습니다.”

---

## 슬라이드 8. 불규칙한 스캔 벽은 면 단위로 해석한다

### 화면에 넣을 문구

**벽 전체 박스가 아니라 실제 벽 면을 Collider로 만든다**

문제:

- 전체 bounds는 문·창문 개구부까지 막는다.
- ㄱ자 벽과 불규칙한 Mesh에서 과도한 충돌이 생긴다.

해결:

- 벽 triangle을 분석해 face 단위 Collider 생성
- `roomFacingNormal`로 방 안쪽 경계 판정
- 분석 실패 시 bounds OBB로 fallback

### 화면 구성

- 벽 wireframe 위에 face Collider가 표시된 디버그 화면을 사용한다.
- 개구부가 비어 있는 것이 명확히 보이게 확대한다.
- 설명 문구는 최대 세 줄로 유지한다.

### 발표 멘트

“스캔 벽 전체를 하나의 박스로 보면 문과 창문 구멍까지 막힙니다. 실제 triangle을 분석해 벽 면 단위 Collider를 만들고, 방 안쪽 normal을 기준으로 경계 침범 여부를 판단했습니다.”

---

## 슬라이드 9. 문과 창문은 실제 벽 두께에 맞춰 보정한다

### 화면에 넣을 문구

**평면 reference를 실제 벽 geometry에 맞춘다**

```text
가장 가까운 벽 Mesh 탐색
→ 가장 짧은 축을 두께 방향으로 판정
→ 벽 정점을 투영해 실제 두께와 중심 측정
→ reference 두께 clamp
→ 벽 중심으로 위치 재정렬
```

강조 문구:

> 고정 두께를 추정하지 않고 스캔된 벽에서 직접 측정한다.

### 화면 구성

- 문이 벽 밖으로 튀어나온 Before와 벽 안에 맞춰진 After를 나란히 둔다.
- 가운데에 벽 두께 축과 중심선을 간단히 표시한다.

### 발표 멘트

“RoomPlan의 문과 창문은 두께가 거의 없는 평면이고 위치도 벽 앞면을 기준으로 할 수 있습니다. 가까운 벽의 실제 정점을 두께 축으로 투영해 두께와 중심을 구하고 모델을 자동으로 맞췄습니다.”

---

## 슬라이드 10. 문을 지우면 방 구조와 충돌도 함께 바뀐다

### 화면에 넣을 문구

**벽 메우기는 시각 효과가 아니라 방 구조 편집이다**

```text
문·창문 선택
→ 개구부 유지 또는 벽 메우기
→ 벽 두께·재질 복제
→ infill Mesh 생성
→ Collider 재생성
→ 기존 가구 침투 재검사
```

하단 강조 문구:

> Mesh, material, collision, metadata를 함께 갱신해야 진짜 벽이 된다.

### 화면 구성

- 삭제 전, 개구부 유지, 벽 메우기의 3단 비교가 가장 이해하기 쉽다.
- 세 화면을 같은 카메라 각도로 캡처한다.

### 발표 멘트

“문을 벽으로 메우는 기능은 단순히 문 모델을 숨기는 것이 아닙니다. 실제 벽 Mesh와 Collider를 만들고 기존 가구까지 다시 검사해야 이동, 충돌, 저장이 모두 일관되게 동작합니다.”

---

## 슬라이드 11. 객체뿐 아니라 방 자체를 저장한다

### 화면에 넣을 문구

**가구 위치만 저장해서는 편집 세션을 복원할 수 없다**

```text
replayable metadata
  ├─ objects / doors / windows
  ├─ position / rotation / scale / dimensions
  ├─ decorations
  ├─ wall color / infill
  ├─ _spatiumRoom geometry + material + transform
  └─ export version / config
```

결과:

> 원본 USDZ가 없어도 벽을 메운 상태까지 그대로 다시 연다.

### 화면 구성

- 왼쪽은 저장 데이터 구조, 오른쪽은 저장 전과 재오픈 후 동일한 화면을 둔다.
- 실제 JSON 전체를 캡처하지 않는다. 핵심 필드만 구조도로 보여준다.

### 발표 멘트

“객체 transform만 저장하면 벽을 메우거나 색상을 바꾼 결과는 사라집니다. 그래서 방 geometry와 material까지 `_spatiumRoom`에 직렬화해 저장본 하나로 전체 편집 세션을 복원합니다.”

---

## 슬라이드 12. 하나의 snapshot으로 모든 편집을 되돌린다

### 화면에 넣을 문구

**Undo / Redo의 단위도 재생 가능한 편집 상태다**

- 이동·회전·높이·크기
- 가구 추가·삭제·교체
- 피규어 배치
- 문·창문 삭제와 벽 infill
- 벽 색상과 방 Mesh

복원 전략:

```text
transform만 변경 → 기존 Node 재사용
구조 변경 → 씬 재구성
```

### 화면 구성

- 편집 상태 A → B → C와 Undo 화살표를 보여준다.
- 실제 Undo 전후 화면을 두 장 넣어도 좋다.

### 발표 멘트

“이력 역시 단순 좌표 배열이 아니라 replayable metadata snapshot입니다. transform만 바뀌면 기존 Node를 재사용하고, 객체나 방 구조가 바뀌었을 때만 씬을 다시 구성합니다.”

---

## 슬라이드 13. 실제 공간 제약이 작동하는 편집 경험을 완성했다

### 화면에 넣을 문구

**실제 공간을 보는 단계에서 직접 편집하는 단계로 확장했다**

```text
실측 모델 fitting
+ 회전 대응 OBB
+ 벽 face Collider
+ Sweep 이동
+ 방 구조 저장
= 재편집 가능한 3D 인테리어 공간
```

마무리 문장:

> Spatium은 실제 치수와 공간 제약을 반영해 인테리어를 편집하고, 그 결과를 완전한 공간 상태로 다시 복원한다.

### 화면 구성

- 완성도가 가장 높은 최종 에디터 화면 한 장을 크게 사용한다.
- 핵심 기술은 화면 아래 한 줄로만 배치한다.
- 단순한 “감사합니다” 슬라이드로 끝내지 않는다.

### 발표 멘트

“결과적으로 Spatium은 스캔 공간을 보여주는 뷰어에서 벗어나, 실제 치수와 공간 제약이 작동하고 방 구조 변경까지 보존되는 3D 인테리어 편집 경험을 제공합니다.”

## 6. PPT 디자인 가이드

### 6.1 기본 규격

- 화면비: 16:9
- 덱 제목: 50pt 이상

- 한 슬라이드 본문: 최대 4~5개 bullet

### 6.2 화면 배치 원칙

- 슬라이드 하나에는 주장 하나만 둔다.
- 제목은 “OBB 충돌” 같은 주제명이 아니라 “회전한 가구는 OBB로 충돌을 계산한다”처럼 결론형으로 쓴다.

- 실제 코드는 본편에 길게 넣지 않는다. 꼭 필요하면 5~8줄만 확대한다.
- 애니메이션은 이동 경로, 벽 투명화, 저장 후 복원처럼 변화가 핵심인 장면에만 사용한다.

## 7. 발표할 때 강조할 것과 줄일 것

### 강조할 것

- “왜 어려웠는가”를 기능 소개보다 먼저 설명한다.
- OBB, Sweep, face Collider가 각각 다른 문제를 해결한다는 점을 구분한다.
- 문·창문 fitting은 스캔 데이터의 한계를 geometry 측정으로 보정한 사례라고 설명한다.
- 벽 infill은 Mesh만 추가한 것이 아니라 Collider와 기존 가구까지 갱신한다고 강조한다.
- 저장이 가구 위치 목록이 아니라 방 자체를 포함한다는 점을 결론과 연결한다.

### 줄일 것

- 모든 파일명과 함수명을 본편에서 읽지 않는다.
- 설정 키 전체를 나열하지 않는다.
- 웹과 iOS의 구현 차이를 지나치게 깊게 설명하지 않는다.
- 기능 목록을 한 장에서 10개 이상 보여주지 않는다.
- Three.js, React, SwiftUI 등의 기술 로고만 나열하는 별도 장은 만들지 않는다.

## 8. 데모 운영 가이드

### 권장 데모 순서

```text
방 열기
→ 가구 추가
→ 빠르게 벽으로 이동
→ 회전·크기·높이 변경

```

### 데모 전 준비

- 네트워크 없이도 사용할 수 있는 캡처 영상 또는 GIF를 준비한다.
- 데모용 방과 가구를 미리 정한다.
- 문·창문 fitting이 잘 보이는 카메라 위치를 저장한다.
- 저장 후 재오픈할 room ID를 미리 확인한다.
- 충돌 디버그 옵션을 켠 버전과 끈 버전을 각각 준비한다.
- 실시간 데모가 실패하면 바로 영상으로 전환할 수 있게 한다.

### 데모 중 설명 원칙

- 마우스 동작을 말로 전부 중계하지 않는다.
- 동작 전에 “벽을 빠르게 통과시켜 보겠습니다”처럼 확인할 포인트를 먼저 말한다.
- 결과가 나온 뒤 “이때 OBB와 Sweep이 함께 작동합니다”라고 기술과 연결한다.
- 저장 후 다시 열리는 장면을 마지막 증거로 사용한다.

## 9. 예상 질문과 답변

### Q1. AABB 대신 OBB를 사용한 이유는 무엇인가요?

회전한 가구의 실제 footprint를 반영하기 위해서다. AABB는 월드 축에 고정되어 회전 시 불필요한 빈 공간까지 충돌 영역으로 포함하거나 모서리 충돌을 놓칠 수 있다.

### Q2. OBB만 있으면 빠른 이동도 막을 수 있나요?

아니다. OBB는 특정 시점의 교차를 정확히 판정하지만 프레임 사이의 경로는 보지 않는다. 이동 벡터를 작은 단계로 나누는 Sweep을 함께 적용해야 tunneling을 막을 수 있다.

### Q3. 문과 창문의 두께는 어떻게 결정하나요?

고정 값을 사용하지 않고 가장 가까운 벽 Mesh의 가장 짧은 축을 두께 방향으로 선택한 뒤, 실제 정점을 투영해 두께와 중심을 계산한다.

### Q4. 벽을 메운 결과는 어떻게 저장되나요?

생성된 infill을 일반 벽 Mesh로 표시하고 `_spatiumRoom`의 geometry·material·transform에 포함한다. 재로딩 시 방 Mesh와 Collider를 다시 만들기 때문에 실제 벽처럼 동작한다.

### Q5. 웹과 iOS가 완전히 같은 코드인가요?

코드는 다르지만 편집 데이터의 의미와 핵심 제약 모델은 같다. 웹은 Three.js OBB, iOS는 `FurnitureFootprint`처럼 플랫폼에 맞는 구현을 사용한다.

### Q6. 성능은 어떻게 관리했나요?

GLB template을 캐시해 clone하고, transform만 바뀔 때는 기존 Node를 재사용한다. 방 구조가 바뀔 때만 Collider나 씬을 필요한 범위에서 재생성한다.
