# Spatium 3D 에디터 — 발표/포트폴리오 정리

> 발표 슬라이드/포트폴리오용 요약 문서.
> 구현 상세와 파일 위치는 [3D_MODELING_LOGIC.md](./3D_MODELING_LOGIC.md) 참고.

## 1. 한 줄 소개

**iPhone LiDAR로 스캔한 실제 내 방을 웹에서 3D로 복원하고, 실측 치수 기반으로 가구를 배치해볼 수 있는 웹 3D 인테리어 에디터**

- 프론트: React + Three.js (라이브러리 없이 직접 구현한 에디터 코어)
- 입력: iOS RoomPlan 스캔 데이터 (USDZ 모델 + 치수 metadata JSON)
- 출력: 다시 열어서 이어 편집할 수 있는 저장 포맷 (JSON)

## 2. 주요 기능 (데모 순서 추천)

| # | 기능 | 한 줄 설명 |
| --- | --- | --- |
| 1 | **스캔한 방 복원** | RoomPlan USDZ + metadata를 Three.js 씬으로 복원. 벽/바닥/문/창문/기존 가구가 실측 크기 그대로 배치됨 |
| 2 | **가구 배치** | 카탈로그에서 가구 선택 → 크기(cm) 입력 → 바닥 클릭으로 배치. GLB 모델이 입력 치수에 맞게 자동 스케일 |
| 3 | **이동 / 회전 / 높이 조정** | 드래그로 바닥 위 이동, 슬라이더/드래그로 Y축 회전, 높이 슬라이더로 선반·액자처럼 벽 중간 높이 배치 |
| 4 | **벽 충돌 방지** | 가구가 벽·문·창문을 통과할 수 없음. 빠르게 드래그해도 뚫리지 않음 (아래 기술 하이라이트 ①) |
| 5 | **문/창문 편집** | 다른 모델로 교체하거나, "개구부로 남기기 / 벽으로 메우기" 중 선택해서 삭제. 메운 자리는 원래 벽과 같은 재질로 채워짐 |
| 6 | **카메라 기준 벽 투명화** | 카메라가 방 밖에 있으면 시야를 가리는 벽 면만 자동으로 투명해져서 어느 각도에서도 방 내부가 보임 |
| 7 | **측정 모드** | 방 폭/깊이/높이, 바닥 면적(㎡/평), 바닥 외곽선을 씬 위에 표시 |
| 8 | **Skyview / 벽·바닥 색상** | 위에서 내려다보는 시점 전환, 벽·바닥 색상 변경 |
| 9 | **저장 & 재편집** | 편집 결과를 JSON으로 저장. 원본 USD 파일 없이도 저장본만으로 완전히 복원 (기술 하이라이트 ④) |
| 10 | **사용자 가구 (AI 연동)** | 사진 한 장 → 배경 제거 → 3D 모델 생성 → 내 가구로 등록 → 에디터 카탈로그에서 바로 배치 |

## 3. 아키텍처 (슬라이드 1장)

```text
3dEditor.js (화면)                    ← 카탈로그 / 저장 버튼 / 방 전환 UI
  └─ RoomSceneEditorPage (뷰포트)     ← 선택 패널 / 회전·높이 슬라이더
       └─ useRoomSceneEditor (코어 훅) ← 편집 세션 전체를 관장
            ├─ scene/sceneLoaders     : USD/GLB/JSON 로딩
            ├─ scene/wallColliders    : 벽 판별 + 면 단위 콜라이더 생성
            ├─ scene/objectFactory    : 가구/문/창문 오브젝트 생성
            ├─ scene/collision        : OBB 충돌 판정 + 이동 제한
            ├─ scene/wallVisibility   : 카메라 기준 벽 투명화
            ├─ scene/roomMetadata     : 저장 JSON 직렬화/복원
            └─ scene/roomMeasurements : 면적/치수 계산
```

**설계 포인트**: 씬 그래프를 직접 다루는 로직(scene/*)과 React 상태 관리(hooks/*)를 분리.
Three.js 오브젝트는 React 밖에서 관리하고, UI에 필요한 값만 state로 동기화한다.

## 4. 기술 하이라이트 (발표에서 깊게 다룰 부분)

### ① OBB 충돌 판정 + Sweep 이동 제한 — "가구가 벽을 못 뚫는 이유"

**문제**: 축 정렬 박스(AABB)로는 회전한 가구의 충돌을 정확히 판정할 수 없고,
빠르게 드래그하면 한 프레임에 벽을 건너뛰어 통과해버린다 (터널링).

**해결**:
- 모든 가구는 회전을 따라가는 **OBB**(Oriented Bounding Box)로 충돌 판정
- 이동 벡터를 한 번에 적용하지 않고 **작은 step으로 쪼개서** 각 step마다 벽 침범 성분을 제거
- "충돌하면 되돌리기"가 아니라 **적용 전에 이동량 자체를 제한** → 벽에 밀착해서 미끄러지듯 움직임

```text
요청된 이동 → step 분할 → 각 step에서 벽 안쪽 성분 제거 → 허용된 이동만 누적 적용
```

- 회전/높이 조정은 적용 후 충돌 검사 → 실패 시 마지막 유효 상태로 롤백
- 문/창문도 이동 시 장애물로 취급 (벽 콜라이더 + 문/창문 OBB를 합쳐서 판정)

### ② 벽 면(face) 단위 콜라이더 — "스캔 데이터의 불규칙한 벽 다루기"

**문제**: 스캔된 방은 벽이 반듯한 직육면체가 아니다. 벽 mesh 하나를 통째로 박스 취급하면
개구부(문/창문 구멍)까지 막히고, ㄱ자 벽에서 오판이 생긴다.

**해결**:
- 벽 mesh의 삼각형을 분석해 **벽 면 단위로 콜라이더를 생성**
- 각 콜라이더에 **방 안쪽을 향하는 normal**(roomFacingNormal)을 기록
- 단순 박스 교차가 아니라 "**방 내부 경계선을 침범했는가**"를 기준으로 판정 → 벽 바깥쪽이나 벽 위 개구부에서는 불필요하게 막히지 않음
- face 분석이 실패하는 mesh는 bounds 기반 OBB로 fallback (견고성)

### ③ 문/창문 벽 두께 자동 fitting — "스캔 데이터의 한계 보정"

**문제**: RoomPlan은 문/창문을 **두께 0인 평면**으로 기록하고, 좌표도 벽 앞면 기준이다.
임의 두께를 부여하면 절반이 항상 벽 앞으로 튀어나온다.

**해결** (모델을 배치할 때마다 자동 보정):

```text
문/창문 위치 → 가장 가까운 벽 mesh 탐색
            → 그 벽의 가장 짧은 축 = 두께 방향으로 판정
            → 벽 정점 전체를 그 축에 투영해 실측 두께/중심 측정
            → 문/창문 두께를 실측 두께로 clamp + 위치를 벽 중심으로 재정렬
```

→ 어떤 두께의 벽이든 문/창문이 벽 안에 정확히 맞물린다.
"벽으로 메우기" 기능도 같은 측정 로직을 재사용해 원래 벽과 두께·재질이 일치하는 채움 벽을 만든다.

### ④ 재생 가능한 저장 포맷 — "원본 파일 없이 다시 열리는 방"

**문제**: 편집 결과를 저장할 때 가구 위치만 저장하면, 벽을 메우는 등 **방 자체가 바뀐 편집**은 복원할 수 없다.
매번 원본 USDZ에 의존하는 것도 취약하다.

**해결**:
- 가구/문/창문은 위치·회전·치수·모델 URL을 담은 JSON으로 직렬화
- **방 mesh 자체(geometry + material + transform)도 JSON으로 직렬화**해 metadata에 포함 (`_spatiumRoom`)
- 다시 열 때 저장본이 있으면 원본 USD보다 우선 복원 → 저장 JSON 하나로 편집 세션이 완전히 재현됨

### ⑤ 카메라 기준 벽 투명화 — "3인칭 시점 인테리어 뷰"

- 매 프레임 카메라 방향과 각 벽 면의 normal을 비교해 시야를 가리는 면만 투명화
- 벽 mesh 전체가 아니라 **geometry group 단위로 material을 교체**해 해당 면만 사라짐
- 문/창문도 벽과 함께 자연스럽게 흐려지도록 같은 기준을 적용

## 5. 트러블슈팅 스토리 (포트폴리오 "문제 해결 경험"용)

| 문제 | 원인 | 해결 |
| --- | --- | --- |
| 빠르게 드래그하면 가구가 벽을 통과 | 프레임 사이 이동량이 벽 두께보다 커서 건너뜀 (터널링) | 이동을 step 단위로 쪼개 각 step마다 침범 성분을 제거하는 sweep 방식으로 변경 |
| 문/창문이 벽 앞뒤로 튀어나옴 | 스캔 데이터가 두께 0 평면 + 벽 앞면 좌표라서, 두께 부여 시 절반이 돌출 | 소속 벽의 실측 두께를 측정해 두께 clamp + 벽 중심으로 위치 재정렬 |
| 일부 GLB 가구에서 정체불명의 검은/흰 판이 렌더링 | 에셋 제작 툴의 UI 킷 잔여물 mesh가 GLB에 섞여 export됨 | 이름/재질 패턴 기반 아티팩트 제거를 모든 모델 로딩 경로에 적용 |
| 문/유리창의 유리가 불투명하게 렌더링 | GLB의 유리 재질이 alphaMode OPAQUE로 export됨 | 재질명 기반으로 유리를 찾아 투명도 강제 적용 (인스턴스별 material clone으로 부작용 차단) |
| 벽으로 메운 자리가 티 나게 보임 | 채움 mesh가 기본 색상이라 원래 벽과 이질감 | 매칭된 벽의 material을 clone해 재질(색·roughness·텍스처)까지 동일하게 적용 |
| 벽을 메운 자리에 있던 가구가 벽에 파묻힘 | 새 벽 생성 후 기존 가구 위치를 재검사하지 않음 | 콜라이더 재생성 후 전체 가구에 벽 제약 초기화를 다시 실행해 밀어냄 |

## 6. 사용 기술 요약 (슬라이드 하단/기술 스택용)

- **Three.js** — WebGL 렌더링, OrbitControls, CSS2DRenderer(치수 라벨)
- **three/examples OBB** — 회전 대응 충돌 판정
- **USDLoader / GLTFLoader** — RoomPlan USDZ 방 모델, GLB 가구 모델 로딩
- **React hooks 아키텍처** — Three.js 씬(비 React)과 UI 상태(React)의 경계 분리
- **자체 저장 포맷** — geometry 직렬화 포함, 원본 없이 재편집 가능

## 7. 발표 데모 시나리오 (5분 기준 제안)

1. 마이페이지에서 스캔된 방 열기 → **방이 실측 그대로 복원됨** 강조 (30초)
2. 가구 추가: 크기 입력 → 배치 → 드래그 이동 → **벽에 밀어붙여도 안 뚫림** 시연 (1분)
3. 빠르게 드래그해도 통과하지 않음 + 회전/높이 슬라이더 (1분)
4. 문 선택 → 다른 문으로 교체 → 창문 "벽으로 메우기" → **원래 벽과 똑같은 재질** 강조 (1분)
5. 측정 모드 켜서 면적/치수 확인 → 벽 색상 변경 → Skyview (30초)
6. 저장 → 나갔다가 다시 열기 → **메운 벽까지 그대로 복원** (30초)
7. (옵션) 사진으로 가구 만들기 → 내 가구로 배치 (30초)
